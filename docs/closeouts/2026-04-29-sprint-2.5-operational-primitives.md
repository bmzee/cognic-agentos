## What ships in `feat/sprint-2.5-operational-primitives` after Sprint 2.5

### Three new critical-controls modules

- **`src/cognic_agentos/core/sla.py`** — Pure-functional SLA timer. `SLAStatus` (StrEnum: GREEN / WARNING / BREACHED), `SLAPolicy` (frozen+slotted dataclass with locked validation order: `total_budget > 0` → `warning_threshold ≥ 0` → `warning_threshold < total_budget`), `SLADeadline` (frozen+slotted projection), `SLATimer.compute_deadline` + `SLATimer.classify` (static, side-effect-free). No DB, no emission; caller decides whether breach triggers downstream effects. Naive datetimes rejected at the boundary (Sprint 2 R3 canonical-form mirror). 100% line / 100% branch.
- **`src/cognic_agentos/core/escalation.py`** — Strict escalation state machine. `EscalationState` StrEnum (5 members), `_LEGAL_TRANSITIONS` adjacency map, `IllegalEscalationTransition` + `EscalationNotFound` exceptions, `EscalationStore.open()` + `.transition()` + `.get_by_id()` + `.list_open()` + private `_read_current_state_within_txn`. State machine: `open → acknowledged → assigned → (resolved | re-escalated → acknowledged)`. Transition validation runs INSIDE the chain-head FOR UPDATE lock via `append_with_precondition` (Option A primitive — closes TOCTOU). Read-side projection rejects malformed chains: lifecycle linkage + adjacency + event_type/to_state agreement + genesis-row shape all enforced (5 layers of malformed-chain rejection landed in plan-review fixes). 100% line / 100% branch.
- **`src/cognic_agentos/core/guardrails.py`** — Pluggable filter pipeline. `Guardrail` runtime-checkable Protocol, `GuardrailResult` + `PipelineResult` frozen+slotted dataclasses, `GuardrailDirection` StrEnum (INPUT / OUTPUT), `GuardrailPipeline.check()` (runs every guardrail unconditionally — no short-circuit; emits one `audit_event` per trip with `event_type="guardrail.trip"`, `iso_controls=("ISO42001.A.7.4",)`; fail-loud emission posture mirrors `AuditStore.append`). Bundled filters: `RegexPIIGuardrail` (credit_card / ssn_us / phone / email — phone tightened to documented 10-15 digits per reviewer P1) + `InjectionGuardrail` (instruction_override / system_prefix / token_injection_markers). PII privacy: `GuardrailResult.matches` carries pattern names ONLY (never raw matched text); `GuardrailResult.detail` is in-process-only and NOT persisted to the audit chain (T7-reviewer-P1 fix). 100% line / 100% branch.

### One additive primitive on a Sprint-2 critical-controls module

- **`core/decision_history.append_with_precondition[T](*, record_builder, precondition)`** — async caller-supplied validator runs INSIDE the chain-head FOR UPDATE transaction; on success, the validator's T-typed return flows into the synchronous `record_builder` to construct the `DecisionRecord` that gets hashed + inserted under the same lock. This is the load-bearing primitive that closes the TOCTOU window for state-machine validators (escalation lifecycle, future model-promotion lifecycle). Existing `append()` is now a thin wrapper using an async no-op precondition + identity record_builder; PEP 695 generic inference resolves `T = None` without `# type: ignore`. **Sprint 2 R3 "snapshot before first await" hardening preserved** via `_validate_and_normalize_record(record)` helper called pre-await in the wrapper (caller-mutation regression test pinned the preservation).

## Plan-review findings closed

Sprint 2.5 went through **multiple rounds of pre-implementation + per-task plan-review** that surfaced concrete defects:

- **Plan-PR #7 (4 review rounds):** P1 TOCTOU window in original transition() draft (closed via Option-A `append_with_precondition`); P2 T2 test scope vs placeholder `_current_state` (closed via T3 covering the validator path); P3 SLA validation order made the sample test fail (closed by reordering checks); P1 stale (record, precondition) API in T2 contract (closed by lifting Option A to T2 from the start); P2 sample append() wrapper used sync lambdas behind `# type: ignore` (closed by async no-op + PEP-695 inference); P2 stale rejected API in doctrine summary (closed by full purge); P1 decision_type vs event_type column mismatch in T3+T4 SQL (closed by event_type throughout); P2 SQLite race-outcome assertion was flaky (closed by removing race-outcome assertion from unit suite, deferring to T9); P3 SLA test snippet missing `dataclasses` import (closed).
- **T2 implementation review:** P1 caller-mutation regression — wrapper captured caller-owned `record` by reference in the lambda; closed by `_validate_and_normalize_record` pre-await snapshot helper preserving Sprint 2 R3 hardening.
- **T3 implementation review:** P2 `_read_current_state_within_txn` accepted any `escalation.%` row; closed by `seen_opened` guard requiring an `escalation.opened` row for the id.
- **T4 implementation review:** P1 projection ignored sequence/adjacency violations; closed by 5-layer lifecycle validation rejecting the whole malformed group. P2 projection ignored event_type/to_state mismatch; closed by `event_type == _decision_type_for_state(to_state)` check.
- **T6 implementation review:** P1 phone regex was 7-20 digits not the documented 10-15; closed by tightened pattern + 7 boundary-negative regression tests + 1 documented overlap test.
- **T7 implementation review:** P1 audit payload persisted free-form `GuardrailResult.detail` (privacy hole — pack-supplied filters could put PII in detail); closed by dropping the field from the payload entirely + `TestPipelinePayloadDoesNotPersistDetail` regression with rogue PII detail.
- **T9 implementation review:** P1 `asyncio.gather` race test wasn't deterministic — sequential A-then-B execution could pass without proving FOR UPDATE; closed by `_PausingEscalationStore` test-only subclass forcing actual lock contention + load-bearing `not task_b.done()` assertion.

Each finding produced one or more regression tests pinning the fix. The closeout note count above (and the test-count growth from `~25` projected to `158 unit + 6 integration`) reflects the depth of the review-driven defensive testing.

Pinned here so post-merge consumers don't read "guardrails landed" as "the bank-deployment-ready guardrail program landed".

| Concern | Sprint-2.5 status | Notes |
|---|---|---|
| **Fair-lending / adverse-action controls (Reg B / ECOA)** | **Carryover.** | Adverse-action notice generation, statistical disparate-impact validation, and Reg-B-mandated reasoning surfaces are not in Sprint-2.5 scope. Sprint 2.5 ships the **evidence substrate** (decision_history chain) on top of which adverse-action controls land — but the controls themselves require domain logic + bank-pack mappings deferred to bank-overlay packs (per ADR-001 plugin discipline) + Sprint 5+ governance content. |

| **RAG / prompt-injection hardening** | **Partial — baseline regex MVP delivered.** | `InjectionGuardrail` ships three regex pattern categories: `instruction_override` ("ignore previous instructions"), `system_prefix` (`system:` / `system prompt:`), `token_injection_markers` (`<\|im_start\|>`, `<\|endoftext\|>`, `### system`). High-recall by design. **Carryover:** ML-based intent classifiers, structured-prompt sandboxing (XML/JSON-schema enforcement on tool inputs), context-window separators (per ChatML / Anthropic system-prompt isolation), per-pack injection-pattern bundles, and red-team-corpus regression suites are all Wave 2 / Sprint 11+ per BUILD_PLAN. |

Plugin-trust-gate work (cosign + per-tenant allow-list per ADR-002 / ADR-016) is Sprint 5+. Documented overlap on separator-formatted CCs (high-recall design boundary). **Carryover:** ML-based PII (semantic email-shape, document-context bank-account-numbers, DLP integrations), language-specific fields (e.g. national IDs by jurisdiction), and bank-specific pattern bundles. |

PII raw-text leakage prevention proven end-to-end on real PG + Oracle in T10. |

**The shorthand:** Sprint 2.5 ships the **chain-of-custody for governance decisions + a baseline content-filter at the perimeter**. Real bank-grade guardrails require additional regulatory content, ML detection, monitoring, and pack-trust enforcement — all explicitly named in ADRs, BUILD_PLAN, or this carryover table. Banks deploying AgentOS post-Sprint-2.5 get a substrate that downstream sprints + bank-overlay packs build on; they do NOT yet get a complete adverse-action / model-risk / RAG-output-grounding compliance program.
