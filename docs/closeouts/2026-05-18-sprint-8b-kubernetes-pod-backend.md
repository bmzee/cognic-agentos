## User-found P1 at T8B-c (refines the subagent trust-but-verify doctrine)

**Root cause** (`backend_factory.py:88+106` at the subagent's initial commit):
```
return <Backend>(**kwargs)
```
The factory's docstring at `:46-50` documented injection ("The factory then injects `settings` into the forwarded kwargs") — but the code did NOT deliver. Tests passed only because the `_docker_kwargs()` / `_k8s_kwargs()` helpers PRE-FILLED `"settings": MagicMock()` (test fixture papered over the production gap). Production callers following the documented contract would hit `TypeError: <Backend>.__init__() missing 1 required keyword-only argument: 'settings'`. Worse: callers passing a SEPARATE `settings` via kwargs would silently route on one Settings object while constructing the backend with another (silent-different-Settings-object bug class).

**Fix:** single injection line `kwargs["settings"] = settings` at the top of `get_backend`'s function body BEFORE the if-tree — applies to both arms + inherits to any future Wave-2 backend. Override semantic (overwrites any caller-passed `settings` kwarg) closes the silent-different-Settings-object class.

**Test updates:** removed `"settings"` from helper kwargs (docstring at `tests/unit/sandbox/test_backend_factory.py:53-57` explicitly names the omission as intentional — "would have papered over the original P1"); routing tests now bind `routing_settings = _settings(...)` as a named local + assert `backend._settings is routing_settings` after isinstance narrow; added `test_factory_override_wins_over_kwargs_settings` pinning the override semantic by passing a bogus kwargs settings + asserting `backend._settings is routing_settings, NOT bogus`.

**TM-revert proof per `feedback_security_regression_hardening`:** removing the injection line → 3/5 tests FAILED with 3 distinct failure modes (2 `TypeError` from missing settings on constructor; 1 `AssertionError` naming bogus vs routing settings) → restored → 5/5 PASS. Fix IS load-bearing.

**Doctrine refinement:** subagent's tests passed locally because helpers PRE-FILLED a value that the production code was supposed to inject itself. Trust-but-verify gate ladder did NOT catch this because gate ladder runs tests as-shipped. External verification (running production code WITHOUT the test-fixture safety net OR running through the documented contract directly) caught it. Refines the deferred trust-but-verify subagent-memory note from T8B-b: subagent verification step should include "production caller succeeds with documented contract" — beyond just "do the new tests pass."
