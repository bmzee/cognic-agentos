## Doctrine-review checklist (resolve BEFORE T1 implementation begins)

The author flagged these architectural decisions as needing user review against ADRs / BUILD_PLAN before they're locked in by code:

1. **Should `core/sla` emit `decision_history` records on breach detection?** *Plan default: NO* — SLA is pure timer math. Caller (escalation, LLM gateway, etc.) decides whether breach triggers an emission. Keeps SLA pure-functional + trivially testable. Alternative: SLA emits `decision_history` records with `decision_type="sla.breach"` directly, automating the audit trail at the cost of coupling.
2. **Where do escalation transitions land — `decision_history` or a dedicated `escalation_event` chain?** *Plan default: `decision_history`* — escalation IS a governance-relevant decision; reusing the existing chain avoids a new evidence table + new chain-head row + new alembic migration. Read-side projection is O(n) walk; flagged for Sprint 5+ snapshot-table optimisation when escalation volume is real.
3. **Where do guardrail trips land — `audit_event` or a dedicated `guardrail_event` chain?** *Plan default: `audit_event`* — ISO 42001 evidence chain wants "system blocked PII export" as an audit event. Volume concern: every blocked input/output hits the chain-head FOR UPDATE row. Sprint-2's 50-concurrent canary proves the primitive scales; flag for Sprint 5+ if guardrail traffic becomes a hotspot.
4. **Should guardrail passes also emit?** *Plan default: NO* — passes are noise. Only trips emit. ISO 42001 cares about what was blocked.
5. **Failure model for emission inside a guardrail/escalation operation.** *Plan default: fail-loud, mirroring `AuditStore.append`* — caller decides whether the request path can survive without the emission. No silent buffering, no in-process queue.
6. **Strict state-machine for escalation — under the SAME lock as the append.** *Plan default: open → acknowledged → assigned → (resolved | re-escalated → acknowledged); illegal transitions raise.* The original draft had a TOCTOU window: `transition()` would read `_current_state()` outside the lock, validate, then call `append()` which acquires the lock — two competing legal transitions from the same state could both pass validation against the same prior state. **Fix landed in this revision: extend `core/decision_history` with `append_with_precondition(record, precondition)` that runs the caller-supplied validator INSIDE the chain-head FOR UPDATE transaction, after reading the head and before inserting the row.** Two competing transitions from `ASSIGNED` → exactly one wins; the loser's validator sees the chain has advanced past `ASSIGNED` and raises. Live race test required (T9).

If the user accepts these defaults the plan proceeds as written; otherwise patch the affected tasks before implementation.

```python
# src/cognic_agentos/core/sla.py
from __future__ import annotations

import dataclasses
from datetime import datetime, timedelta
from enum import StrEnum


class SLAStatus(StrEnum):
    GREEN = "green"        # now < warning_at
    WARNING = "warning"    # warning_at <= now < deadline
    BREACHED = "breached"  # now >= deadline


@dataclasses.dataclass(frozen=True, slots=True)
class SLAPolicy:
    """Caller-side policy bundle. Frozen + slotted; safe to share."""
    name: str
    total_budget: timedelta
    warning_threshold: timedelta  # MUST be < total_budget; constructor validates.

    def __post_init__(self) -> None:
        # Validation order matters: each check's error message must be
        # reachable. With the previous order
        # (warning_threshold >= total_budget FIRST), the case
        # SLAPolicy(total_budget=0, warning_threshold=0) raised the
        # warning-threshold message because 0 >= 0 is True — never
        # reaching the "total_budget must be positive" check. Order
        # the checks so each precondition has at least one input that
        # exercises only it.
        if self.total_budget <= timedelta(0):
            raise ValueError(f"total_budget must be positive; got {self.total_budget}")
        if self.warning_threshold < timedelta(0):
            raise ValueError(
                f"warning_threshold must be non-negative; got {self.warning_threshold}"
            )
        if self.warning_threshold >= self.total_budget:
            raise ValueError(
                f"warning_threshold ({self.warning_threshold}) must be strictly "
                f"less than total_budget ({self.total_budget})"
            )


@dataclasses.dataclass(frozen=True, slots=True)
class SLADeadline:
    """Absolute deadline + warning point for a specific SLA-tracked operation."""
    policy_name: str
    started_at: datetime
    warning_at: datetime
    deadline: datetime


class SLATimer:
    """Pure-functional SLA timer. No DB, no emission, no I/O.

    Two operations:
      - compute_deadline(start, policy): build the absolute SLADeadline.
      - classify(now, deadline): GREEN / WARNING / BREACHED.

    Both methods are static + side-effect-free; the class exists only
    for namespacing.
    """

    @staticmethod
    def compute_deadline(start: datetime, policy: SLAPolicy) -> SLADeadline:
        if start.tzinfo is None or start.tzinfo.utcoffset(start) is None:
            # canonical-form rejects naive datetimes per Sprint 2 R3.
            # Reject here at the boundary too; SLA-tracked timestamps
            # downstream may participate in evidence-pack export.
            raise ValueError(f"start must be timezone-aware; got {start!r}")
        return SLADeadline(
            policy_name=policy.name,
            started_at=start,
            warning_at=start + policy.warning_threshold,
            deadline=start + policy.total_budget,
        )

    @staticmethod
    def classify(now: datetime, deadline: SLADeadline) -> SLAStatus:
        if now.tzinfo is None or now.tzinfo.utcoffset(now) is None:
            raise ValueError(f"now must be timezone-aware; got {now!r}")
        if now >= deadline.deadline:
            return SLAStatus.BREACHED
        if now >= deadline.warning_at:
            return SLAStatus.WARNING
        return SLAStatus.GREEN
```

## Task 2 — `core/decision_history`: `append_with_precondition` primitive

**Critical-controls module — same discipline as Task 1.** This task TOUCHES `core/decision_history` (Sprint 2 critical-controls), so it gets the per-edit halt-before-commit treatment + must keep the file at ≥95% line + ≥90% branch.

The reviewer caught a TOCTOU hole in the original draft of T2/T3: `EscalationStore.transition()` would read the current state, validate, then call `DecisionHistoryStore.append()`. Between the read and the lock acquisition, two competing legal transitions from the same state could both pass validation against the same prior state and produce two contradictory transitions in the chain.

The fix is to extend `DecisionHistoryStore` with an `append_with_precondition` method that runs a caller-supplied validator INSIDE the same transaction that holds the chain-head FOR UPDATE lock. Validators can run SELECTs against the live chain state and observe the committed-up-to-the-lock snapshot. If the validator raises, the transaction rolls back without inserting a row; the chain head is unchanged.

This is an additive contract — the existing `append()` shape is preserved; in fact, it becomes a thin wrapper around `append_with_precondition(record, precondition=None)` so the implementations share the lock + insert + compare-and-set sequence verbatim.

**Files:**
- Modify: `src/cognic_agentos/core/decision_history.py` (factor `append` to share with the new method).
- Modify: `tests/unit/core/test_decision_history.py` (new test class for the precondition primitive).

```python
from collections.abc import Awaitable, Callable

from sqlalchemy.ext.asyncio import AsyncConnection


class DecisionHistoryStore:
    # ... existing __init__, _CHAIN_ID ...

    async def append(self, record: DecisionRecord) -> tuple[uuid.UUID, bytes]:
        """Existing public method, unchanged contract.

        Implementation factored to delegate to append_with_precondition
        with precondition=None so both methods share the lock + insert +
        compare-and-set sequence. No behaviour change for existing
        callers; the same canonical envelope, same chain head update,
        same fail-loud posture.
        """
        return await self.append_with_precondition(record, precondition=None)

    async def append_with_precondition(
        self,
        record: DecisionRecord,
        precondition: Callable[
            [AsyncConnection, int, bytes], Awaitable[None]
        ] | None = None,
    ) -> tuple[uuid.UUID, bytes]:
        """Append a DecisionRecord atomically with an optional precondition
        evaluated INSIDE the locked chain-head transaction.

        Concurrency semantics:
        - Acquires SELECT FOR UPDATE on governance_chain_heads exactly as
          ``append`` does.
        - After reading ``(latest_sequence, latest_hash)``, calls
          ``await precondition(conn, latest_sequence, latest_hash)``
          inside the same transaction. The precondition can run SELECT
          queries against decision_history (or any other table) and
          observe the chain state up to the moment the lock was
          acquired.
        - If the precondition raises, the transaction rolls back. No
          row inserted. Chain head unchanged. The exception propagates
          to the caller.
        - If the precondition returns None, the append continues
          exactly as ``append`` does — INSERT row, compare-and-set
          chain head, return ``(record_id, new_hash)``.
        - ``precondition=None`` is equivalent to plain ``append`` and
          is how ``append`` is implemented.

        Contract on the precondition:
        - MUST be SELECT-only. Calling INSERT/UPDATE/DELETE on chain_heads
          or any evidence row from the precondition breaks chain
          canonicalisation invariants and is undefined behaviour.
        - SHOULD be cheap. The chain-head FOR UPDATE lock is held until
          the transaction commits; long-running validators block
          concurrent appenders. Use indexed SELECTs only.
        - MAY raise any exception type; the caller's try/except
          dispatches on the exception class. Sprint 2.5 use-case is
          state-machine validators that raise IllegalEscalationTransition.

        Use case: state-machine validators (escalation lifecycle, model
        promotion lifecycle, etc.) that must atomically read prior
        chain state, validate the requested mutation, and emit. Without
        this primitive, validate-then-append has a TOCTOU window where
        two workers can race past validation against the same prior
        state.
```

## Task 3 — `core/escalation`: state-machine + write side (using `append_with_precondition`)

**Critical-controls module — same discipline as Task 1.**

Escalation lifecycle drives a strict state machine; each transition emits exactly one `DecisionRecord` into the existing `decision_history` chain via the new `DecisionHistoryStore.append_with_precondition` from T2. State validation runs INSIDE the chain-head FOR UPDATE lock, closing the TOCTOU window. No new evidence table; no new chain-head row.

State machine:

```
                  ┌─────────────────────────────┐
                  ▼                             │
   open ──► acknowledged ──► assigned ──► resolved
                                   │
                                   └─► re-escalated ──┘
```

Transitions:
- `open` is created by `EscalationStore.open()`; emits `decision_type="escalation.opened"`.
- `acknowledged` ← `EscalationStore.transition(escalation_id, EscalationState.ACKNOWLEDGED, ...)`. Valid only from `open` or `re-escalated`.
- `assigned` ← valid only from `acknowledged`.
- `resolved` ← valid only from `assigned`. Terminal.
- `re-escalated` ← valid only from `assigned`. Loops back through `acknowledged`.

Illegal transitions raise `IllegalEscalationTransition` from inside the precondition validator — meaning the FOR UPDATE-locked transaction rolls back without emitting a row. This is what closes the concurrency hole the reviewer caught: under contention, the second of two competing legal transitions sees the chain advanced past its expected `from_state` and raises against the new `from_state`.

**Files:**
- Create: `src/cognic_agentos/core/escalation.py` (state-machine + write side).

```python
# src/cognic_agentos/core/escalation.py (write side; public read-side projection in T4)
from __future__ import annotations

import dataclasses
import json
import uuid
from datetime import UTC, datetime
from enum import StrEnum
from typing import Any

from sqlalchemy import select, text
from sqlalchemy.ext.asyncio import AsyncConnection, AsyncEngine

from cognic_agentos.core.decision_history import (
    DecisionHistoryStore,
    DecisionRecord,
    _decision_history,
)


class EscalationState(StrEnum):
    OPEN = "open"
    ACKNOWLEDGED = "acknowledged"
    ASSIGNED = "assigned"
    RESOLVED = "resolved"
    RE_ESCALATED = "re-escalated"


# Adjacency map enforced at runtime. Each key is the FROM state;
# each value is the set of legal TO states. ASSIGNED has TWO legal
# targets (RESOLVED + RE_ESCALATED) — this is exactly the case where
# the original draft's TOCTOU window opened: two workers from the
# same ASSIGNED state both validate, both append, both win. With
# the precondition primitive landed in T2 the loser sees the new
# RESOLVED/RE_ESCALATED state INSIDE the lock and raises.
_LEGAL_TRANSITIONS: dict[EscalationState, frozenset[EscalationState]] = {
    EscalationState.OPEN: frozenset({EscalationState.ACKNOWLEDGED}),
    EscalationState.ACKNOWLEDGED: frozenset({EscalationState.ASSIGNED}),
    EscalationState.ASSIGNED: frozenset({
        EscalationState.RESOLVED,
        EscalationState.RE_ESCALATED,
    }),
    EscalationState.RE_ESCALATED: frozenset({EscalationState.ACKNOWLEDGED}),
    EscalationState.RESOLVED: frozenset(),  # terminal
}


class IllegalEscalationTransition(ValueError):
    """Raised inside the precondition validator when the requested
    transition is not in _LEGAL_TRANSITIONS for the chain's current
    end-state. Raising from inside the validator causes the
    locked transaction to roll back; no row inserted, chain head
    unchanged. Carries from_state + to_state for diagnostics.
    """

    def __init__(
        self,
        escalation_id: uuid.UUID,
        from_state: EscalationState,
        to_state: EscalationState,
    ) -> None:
        super().__init__(
            f"escalation {escalation_id}: illegal transition "
            f"{from_state.value} → {to_state.value}"
        )
        self.escalation_id = escalation_id
        self.from_state = from_state
        self.to_state = to_state


class EscalationNotFound(LookupError):
    """Raised when transition() or get_by_id() is called for an
    escalation_id that has no escalation.opened record in the chain.
    """

    def __init__(self, escalation_id: uuid.UUID) -> None:
        super().__init__(f"no escalation.opened record for {escalation_id}")
        self.escalation_id = escalation_id


class EscalationStore:
    """Write-side of the escalation state machine + atomic state
    validation. Public read-side queries (get_by_id / list_open) live
    in this same module — see Task 4.

    Concurrency: each transition goes through
    DecisionHistoryStore.append_with_precondition (T2 primitive). The
    precondition validator runs INSIDE the chain-head FOR UPDATE lock
    and reads the chain's current end-state for the given
    escalation_id; if the requested transition is illegal from that
    state, the validator raises IllegalEscalationTransition and the
    locked transaction rolls back. Two concurrent legal transitions
    from the same state → exactly one wins; the loser sees the new
    end-state and raises against it.
    """

    def __init__(self, engine: AsyncEngine) -> None:
        self._engine = engine
        self._history = DecisionHistoryStore(engine)

    async def open(
        self,
        *,
        actor_id: str,
        level: str,
        reason: str,
        request_id: str,
        tenant_id: str | None = None,
    ) -> tuple[uuid.UUID, uuid.UUID, bytes]:
        """Open a new escalation. Returns (escalation_id, decision_record_id, hash).

        Caller-supplied request_id ties the escalation to the originating
        request. level is a free-form string (e.g. "p1", "p2"); pack
        contracts may pin it to an enum later. No precondition needed —
        a fresh escalation_id by definition has no prior chain state.
        """

        escalation_id = uuid.uuid4()
        record_id, h = await self._history.append(
            DecisionRecord(
                decision_type="escalation.opened",
                request_id=request_id,
                actor_id=actor_id,
                tenant_id=tenant_id,
                payload={
                    "escalation_id": str(escalation_id),
                    "level": level,
                    "reason": reason,
                    "from_state": None,
                    "to_state": EscalationState.OPEN.value,
                },
            )
        )
        return escalation_id, record_id, h

    async def transition(
        self,
        *,
        escalation_id: uuid.UUID,
        actor_id: str,
        new_state: EscalationState,
        reason: str,
        request_id: str,
        tenant_id: str | None = None,
    ) -> tuple[uuid.UUID, bytes]:
        """Transition an existing escalation to a new state.

        Atomic semantics: state read + validation + append all happen
        inside the chain-head FOR UPDATE locked transaction via
        ``DecisionHistoryStore.append_with_precondition``. Two
        concurrent legal transitions from the same state serialise;
        exactly one succeeds.

        Raises IllegalEscalationTransition if the chain's current
        end-state for this escalation does not allow ``new_state``.
        Raises EscalationNotFound if no escalation.opened record
        exists for ``escalation_id``.
        """

        async def _validator(
            conn: AsyncConnection,
            prev_sequence: int,
            prev_hash: bytes,
        ) -> None:
            current_state = await self._read_current_state_within_txn(
                conn, escalation_id
            )
            legal_targets = _LEGAL_TRANSITIONS[current_state]
            if new_state not in legal_targets:
                raise IllegalEscalationTransition(
                    escalation_id, current_state, new_state
                )

        # The from_state for the payload is captured by the validator
        # AFTER the lock is acquired; we re-read it here for the
        # payload only (the validator's read is the load-bearing
        # one for correctness — this read is informational).
        # Strategy: run the validator first, then capture from_state
        # in a closure to use in the payload below. The cleanest
        # implementation lifts from_state into a nonlocal mutable
        # captured by the validator.
        from_state_holder: list[EscalationState] = []

        async def _validator_capturing(
            conn: AsyncConnection,
            prev_sequence: int,
            prev_hash: bytes,
        ) -> None:
            current_state = await self._read_current_state_within_txn(
                conn, escalation_id
            )
            legal_targets = _LEGAL_TRANSITIONS[current_state]
            if new_state not in legal_targets:
                raise IllegalEscalationTransition(
                    escalation_id, current_state, new_state
                )
            from_state_holder.append(current_state)

        record_id, h = await self._history.append_with_precondition(
            DecisionRecord(
                decision_type=f"escalation.{new_state.value.replace('-', '_')}",
                request_id=request_id,
                actor_id=actor_id,
                tenant_id=tenant_id,
                payload={
                    "escalation_id": str(escalation_id),
                    "reason": reason,
                    # from_state is filled in by post-validation
                    # closure; until validator runs the value is
                    # unknown. Workaround: use a sentinel string
                    # post-validator hook OR capture into the payload
                    # via a deferred-payload pattern. Implementation
                    # Note: the cleanest shape is to capture
                    # from_state into the validator's closure THEN
                    # build the DecisionRecord after the validator
                    # has run. This requires append_with_precondition
                    # to accept a deferred record builder; see
                    # Implementation Note below.
                    "to_state": new_state.value,
                },
            ),
            precondition=_validator_capturing,
        )
        return record_id, h
```

**Implementation Note — `from_state` payload + the validator's role:**

The naive shape above has a chicken-and-egg problem: the `DecisionRecord.payload` is built before `append_with_precondition` is called, but `from_state` is only known after the validator runs (because the validator is what reads the chain's current end-state under the lock).

Two clean fixes — pick ONE in Step 3.3:

- **Option A: Two-arg precondition shape.** `append_with_precondition` takes `precondition: Callable[..., Awaitable[T]]` where the validator's return value `T` is folded into the payload BEFORE the row is hashed/inserted. Generic over T; type-safe via `TypeVar`. Slight extra surface on the T2 primitive — but the most honest factoring (validator both validates AND captures the state needed for the payload).

- **Option B: Mutable closure capture (sketched above).** Validator stores `from_state` in a list/dict captured from the outer scope; transition() reads it back AFTER `append_with_precondition` returns. Doesn't change T2's contract surface. **Problem:** the `DecisionRecord.payload` is hashed BEFORE the validator runs (envelope construction happens after the head SELECT but the validator runs between head SELECT and INSERT). So the captured `from_state` is too late. Rejected.

- **Option C: Validator-first closure capture, deferred payload builder.** The simplest: `append_with_precondition` accepts a `build_record: Callable[[EscalationState], DecisionRecord]` instead of a fixed `record`. Validator runs first, then build_record(captured_state) is called inline to produce the record, then envelope + INSERT proceed. This requires extending T2's contract.

**Decision (locked in this revision): Option A.** T2's signature becomes:

```python
async def append_with_precondition[T](
    self,
    record_builder: Callable[[T], DecisionRecord],
    precondition: Callable[[AsyncConnection, int, bytes], Awaitable[T]],
) -> tuple[uuid.UUID, bytes]:
    """precondition produces a value T which is then passed to
    record_builder to construct the DecisionRecord that will be
    inserted. T is typically the state-machine projection the
    validator captured under the lock.

    A no-arg shape is also exposed for non-projection callers:
    append_with_precondition_simple(record, precondition_returning_None)
    where record is a fixed value and the precondition returns None.
    """
```

Update T2's tests + contract accordingly. The simple existing-`append()` wrapper becomes:

```python
async def append(self, record: DecisionRecord) -> tuple[uuid.UUID, bytes]:
    return await self.append_with_precondition(
        record_builder=lambda _none: record,
        precondition=lambda conn, seq, prev_h: None,
    )  # type: ignore — generic T defaults to None here
```

This decision is the **load-bearing reason** the reviewer's P1 had to be patched at the T2 level not just T3 — the from_state needs to land in the hashed envelope, which requires the validator to run BEFORE the envelope is built.

```python
        # carries diagnostic granularity. (Alternative: one row with
        # an array of trips; rejected because it muddles the
        # audit-row "what was blocked" question.)
        for r in results:
            if r.passed:
                continue
            await self._audit_store.append(
                AuditEvent(
                    event_type="guardrail.trip",
                    request_id=request_id,
                    tenant_id=tenant_id,
                    payload={
                        "guardrail_name": r.guardrail_name,
                        "direction": direction.value,
                        "matches": list(r.matches),
                        "detail": r.detail,
                    },
                    iso_controls=("ISO42001.A.7.4",),
                )
            )

        return PipelineResult(
            direction=direction,
            passed=passed,
            results=results,
        )
```

## Task 9 — Integration test: live race on competing transitions (NEW — reviewer's explicit request)

**Tests-only task; production-grade proof that the T2 + T3 atomic-validator pattern actually serialises competing legal transitions on real PG + Oracle.**

The reviewer's P1 finding was that the original draft had a TOCTOU window where two workers could both validate against the same prior state and both append. The fix landed in T2 (`append_with_precondition`) + T3 (transition uses it). This task is the live PG + Oracle proof that the fix works.

Scenario: drive an escalation to `ASSIGNED`. From `ASSIGNED` there are TWO legal targets (`RESOLVED` + `RE_ESCALATED`). Fire two concurrent `transition()` calls — one to RESOLVED, one to RE_ESCALATED — via `asyncio.gather(...)`. Assert: exactly one succeeds, the other raises `IllegalEscalationTransition` against the new (advanced) end-state. Chain still walks clean.

**Files:**
- Modify: `tests/integration/db/test_sprint_2_5_chain_integration.py`

### Test shape

```python
@pytest.mark.postgres
@_PG_SKIPIF
async def test_competing_transitions_from_assigned_serialise_postgres() -> None:
    """Two legal transitions from ASSIGNED race; exactly one wins.

    Without the T2 atomic-validator primitive, both would observe
    ASSIGNED, both validate as legal, both append — producing a chain
    with ASSIGNED → RESOLVED → RE_ESCALATED, which is an illegal
    sequence (re-escalating a resolved escalation).

    With the primitive: the loser's validator runs INSIDE the
    chain-head FOR UPDATE lock and sees the chain has advanced to
    RESOLVED (or RE_ESCALATED) — its requested transition is no
    longer legal from that state, so it raises and rolls back.
    """

    # Pool sized for genuine concurrency (mirrors Sprint 2's
    # concurrent-append canary).
    engine = create_async_engine(
        _superuser_url("postgres"), pool_size=4, max_overflow=0,
    )
    try:
        await _reset_chain_state(engine, "decision_history")
        store = EscalationStore(engine)

        # Drive escalation to ASSIGNED.
        eid, _, _ = await store.open(
            actor_id="canary", level="p1", reason="race-canary",
            request_id="req-canary-open",
        )
        await store.transition(
            escalation_id=eid, actor_id="canary",
            new_state=EscalationState.ACKNOWLEDGED,
            reason="ack", request_id="req-canary-ack",
        )
        await store.transition(
            escalation_id=eid, actor_id="canary",
            new_state=EscalationState.ASSIGNED,
            reason="assign", request_id="req-canary-assign",
        )

        # Fire two competing transitions from ASSIGNED.
        async def _to_resolved() -> tuple[uuid.UUID, bytes]:
            return await store.transition(
                escalation_id=eid, actor_id="worker-a",
                new_state=EscalationState.RESOLVED,
                reason="worker-a wants resolve",
                request_id="req-canary-resolve",
            )

        async def _to_re_escalated() -> tuple[uuid.UUID, bytes]:
            return await store.transition(
                escalation_id=eid, actor_id="worker-b",
                new_state=EscalationState.RE_ESCALATED,
                reason="worker-b wants re-escalate",
                request_id="req-canary-re-escalate",
            )

        results = await asyncio.gather(
            _to_resolved(), _to_re_escalated(),
            return_exceptions=True,
        )

        # Exactly one succeeded; the other raised IllegalEscalationTransition
        # against the NEW end-state (the winner's to_state).
        successes = [r for r in results if not isinstance(r, BaseException)]
        failures = [r for r in results if isinstance(r, BaseException)]
        assert len(successes) == 1, (
            f"expected exactly 1 success; got {len(successes)} "
            f"(both transitions appended → TOCTOU window not closed): {results!r}"
        )
        assert len(failures) == 1
        assert isinstance(failures[0], IllegalEscalationTransition), (
            f"expected IllegalEscalationTransition; got {failures[0]!r}"
        )
        # The loser's validator saw the winner's advanced state —
        # so failures[0].from_state matches whichever transition won.
        assert failures[0].from_state in {
            EscalationState.RESOLVED,
            EscalationState.RE_ESCALATED,
        }

        # Chain still walks clean — only one transition row landed
        # past ASSIGNED.
        report = await ChainVerifier(engine, "decision_history").walk()
        assert report.is_clean
        # Initial open + 2 setup transitions + 1 winning transition = 4.
        assert report.records_checked == 4
    finally:
        await engine.dispose()
```
