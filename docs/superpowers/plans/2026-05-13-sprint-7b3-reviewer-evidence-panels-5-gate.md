## Plugin Pack Deployment Model (R8 P2 #1)

AgentOS is the **OS / governance kernel**. It does NOT ship agents, tools, skills, or hooks inside the AgentOS repo. Instead, those four kinds of capability deploy as **plugin packs** that install on top of AgentOS:

- `kind = "agent"` — Layer C LLM-driven agents (ship in `cognic-agent-<name>` repos)
- `kind = "tool"` — MCP-server tool packs (ship in `cognic-tool-<name>` repos)
- `kind = "skill"` — MCP-composing skill packs (ship in `cognic-skill-<name>` repos)
- `kind = "hook"` — governance hook packs per ADR-008 + ADR-017 (registered runtime hooks for `dlp_pre` / `dlp_post` etc.)

Sprint 7B.3 is the **admission / reviewer layer** for plugin packs. It does NOT ship runtime install / activate / deactivate orchestration (that's a separate concern landing post-7B). The 5-gate composer + reviewer evidence panels + override path operate on packs of ANY of the 4 kinds; the lifecycle state machine (Sprint 7B.1) already encodes the kind as `PackRecord.kind: PackKind` per `packs/storage.py:371`.

**R9 requirement adjustment — kind is wire-protocol-public.** Every 7B.3 reviewer-facing evidence panel response and the approve-composition response MUST carry `pack_kind: PackKind`, derived from the authoritative `PackRecord.kind` rather than from free-form manifest text. The persisted manifest's `[pack].kind` is still cross-checked against `PackRecord.kind`; mismatch returns 409 with closed-enum `pack_kind_mismatch` before any panel/composer response is produced. This makes the plugin-pack deployment model testable: reviewers and later install/activate orchestration can see whether an approved artefact is an `agent`, `tool`, `skill`, or `hook` without reverse-engineering it from protocol declarations.

**AgentOS compatibility requirement.** MCP/A2A exposure alone is NOT sufficient to be AgentOS-compatible. A pack is AgentOS-compatible when it passes the AgentOS authoring/admission contract: valid manifest, declared `kind`, governance/risk/data fields, supply-chain signatures, OWASP/conformance evidence, lifecycle state, RBAC-reviewed approval, and audit-chain evidence. MCP and A2A are runtime communication surfaces inside that broader plugin-pack contract.

**Kind-awareness in 7B.3 evidence panels:**
- `data_governance` panel — kind-agnostic projection (every pack manifest declares `[data_governance]` per ADR-017)
- `risk_tier` panel — kind-agnostic (risk-tier vocabulary is closed-enum per ADR-014; the same 7-value approval-flow mapping applies regardless of kind)
- `supply_chain` panel — kind-agnostic projection of attestations (per ADR-016 — every kind ships signed)
- `conformance_matrix` panel — **kind-aware**: MCP capability conformance applies to `tool` + `skill` + `agent`; A2A capability conformance applies to `agent` only; AGNTCY-OASF declarations apply to `agent`; hook packs have no MCP/A2A surfaces (per ADR-008 hook quartet). The panel projects ONLY the relevant matrices for the pack's kind; the others surface as N/A — wire-protocol-stable across all 4 kinds.

**Per-kind gate weighting (NOT in 7B.3 scope):** ADR-012 §41 specifies 5 gates that apply to all kinds equally. A future sprint may introduce per-kind gate weighting (e.g. hook packs have stricter adversarial pass-rate thresholds because they execute inside the audit chain). 7B.3 ships the uniform 5-gate composer; kind-specific weighting is a doctrine-track decision deferred until needed.

### Gate-5 doctrine refinement (closeout-authoritative)

ADR-012 §41's enumerated gates 1-5 list "(5) tenant RBAC scope `pack.review.approve` held by reviewer" as gate 5. That check is enforced upstream by FastAPI's `RequireScope("pack.review.approve")` dependency in `review_routes.py` (Sprint 7B.2 T5); by the time the composer runs, gate 5 is already green-by-construction.

The 7B.2 closeout hand-off list (L234) **reinterprets gate 5** as "reviewer-acknowledgement field set on all 4 evidence panels". This is a substantive server-side check that the closeout treats as the authoritative gate-5 definition for 7B.3 — replacing the RBAC-scope check (already enforced) with the reviewer-ack check (newly enforced).

**7B.3 follows the closeout's interpretation.** The composer's gate 5 is reviewer-acknowledgement; the scope check stays where it is in the dependency chain. The plan's R-rounds will tighten this interpretation if the user prefers a different naming or doctrine alignment.

### Gate-2 doctrine note (ADR-012 §84 "Tenant allow-list permits this pack")

ADR-012 §84's 5-gate diagram includes "Tenant allow-list permits this pack" as gate 2. ADR-012 §41 omits this from the enumerated gates 1-5 (its gate 2 is ADR-010 evaluation). The lifecycle state machine places `allow_listed` AFTER `approved` (per state diagram `approved → allow_listed`), so allow-list cannot logically be a pre-approval gate on the same pack lifecycle.

**Interpretation:** §84's "tenant allow-list" gate is forward-compatible language for cross-tenant approval checks (was this pack approved on another tenant where re-approval is required?) — a Wave-2 / Studio concern, not Wave-1. 7B.3 follows ADR-012 §41's gate enumeration. If the R-rounds determine a tenant-cross-approval check is in scope, an additional gate slot is added.
