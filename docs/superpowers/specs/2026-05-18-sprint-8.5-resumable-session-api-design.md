## 1. Scope summary

**What 8.5 ships:** the resumable-session API per ADR-004 §73-93 — `SandboxSession.checkpoint(label) -> CheckpointId` + `SandboxSession.suspend() -> None` + `SandboxBackend.wake(session_id, *, actor, tenant_id) -> SandboxSession` Protocol extensions; a workspace-tar snapshot mechanism that gives DockerSibling + KubernetesPod backend parity; the `CheckpointStore` orchestrator built on the existing Sprint-4 `ObjectStoreAdapter` `local_fs` driver (with one additive Protocol extension — see §3.5); a `CheckpointMetadata` frozen wire-public dataclass with explicit `to_storage_payload()` / `from_storage_payload()` JSON-native serialisation converters per §3.4; **tombstone semantics for destroyed sessions** (per the P1.r4 design call — destroy() writes a `<tenant>/<session>/_tombstoned.json` sentinel via `CheckpointStore.tombstone_session()`; bytes retained until reaper sweep; wake refuses tombstoned sessions fail-closed with the new `sandbox_wake_session_tombstoned` closed-enum reason); 4 new `sandbox.lifecycle.*` audit-event taxonomy values (`checkpointed` / `suspended` / `woken` / `checkpoint_purged` — NO new lifecycle event for tombstoning; destroy() reuses 8A's `sandbox.lifecycle.destroyed` with a new `retained_until` payload key per §5.1 amended); 6 new `SandboxRefusalReason` values (`sandbox_wake_checkpoint_not_found` / `sandbox_wake_checkpoint_corrupt` / `sandbox_wake_checkpoint_retention_expired` / `sandbox_wake_session_tombstoned` / `sandbox_wake_tenant_mismatch` / `sandbox_wake_policy_revalidation_failed`); a background reaper job emitting purge audits + ENFORCING the per-tenant retention floor at the reaper layer (per §4.3 amended — NOT via ObjectStoreAdapter `retention_seconds` WORM lock, which has incompatible semantics — see P1.r3 fix); the chain verifier walking suspend → wake transitions for forgery detection via explicit payload keys; 3 new global Settings caps (`sandbox_checkpoint_retention_s` / `sandbox_max_checkpoints_per_session` / `sandbox_reaper_interval_s`); the existing Sprint-8A `sandbox_credential_adapter_not_configured` admission-time refusal handles vault-path-bearing sessions fail-loud (NO `CredentialAdapter` Protocol extension in 8.5 — see §2.4 amended); critical-controls gate promotions 71 → 73 (NEW `sandbox/checkpoint_store.py` + NEW Sprint-4 `db/adapters/local_object_store_adapter.py` per §9 amended).

**What 8.5 does NOT ship** (each item is named so future implementors and reviewers can verify scope):
- **CRIU / live-process snapshots.** Wave-1 workspace-tar captures the writable workspace + env + lease refs BETWEEN exec() calls — NOT mid-process state. Long-running multi-step workflows are the use case (multi-day agent loops awaiting external input; operator pause/resume for compliance review of paused agent state). Mid-exec process-state preservation is **deferred to Wave-2** if/when banks demand it; would require CRIU on every node + OpenShift restricted-SCC compatibility analysis.
- **Real Vault credential leasing AND any `CredentialAdapter` Protocol extension.** Sprint 10 ships the real `VaultCredentialAdapter` AND the lease-extension methods (`mint_lease` / `revoke_lease`). The existing Sprint-8A `CredentialAdapter` Protocol surface is exactly `async def fetch_secret(path) -> str | None` (verified at `sandbox/admission.py:96-105`); the Sprint-8A negative pins at `tests/unit/sandbox/test_credential_adapter_stub.py:24+99-116` explicitly forbid `mint_lease` / `revoke_lease` until Sprint 10. Sprint 8.5 honors those pins — does NOT extend `CredentialAdapter`. The existing Sprint-8A admission-time refusal `sandbox_credential_adapter_not_configured` (raised at `admission.py` step 3 for any `policy.vault_path:`-bearing session against the fail-loud sentinel) means no vault-bearing session can be CREATED today; no vault-bearing session created → no vault-bearing suspend → no vault-bearing wake reachable. CheckpointMetadata still carries `vault_lease_refs: tuple[VaultLeaseRef, ...] = ()` for forward-compat wire shape (Sprint 10 fills it). NO silent lease re-use. The original Q4 lock's "If the existing CredentialAdapter shape needs extending, treat that as an 8.5 wire-contract change" escape clause is DEFERRED per the §2.4 amended scope-discipline call — Protocol extension lives at Sprint 10 with the real adapter, not 8.5.
- **Scheduler-admitted wake.** Sprint 10.5 wraps `sandbox.create()` AND `sandbox.wake()` per ADR-022. Sprint 8.5 designs `wake()` with the same identity seams as `create()` (`*, actor, tenant_id`) so 10.5 can slot in additively without breaking the wire.
- **S3-backed checkpoint storage.** Sprint 8 (the original numbering — now folded into the post-8A backlog) added `s3` to the `ObjectStoreAdapter` driver set. **No S3 driver lands in 8.5.** Sprint 8.5 consumes whichever driver `Settings.object_store_driver` resolves to (Sprint 4's `local_fs` today). Per the P4.r5 fix: a future S3 driver MUST implement the **extended** `ObjectStoreAdapter` Protocol including the new `list_prefix()` async-generator method per §3.5 (likely via `ListObjectsV2` with continuation-token pagination). No other 8.5-specific changes are required beyond the Protocol extension itself.
- **Per-tenant Rego override of retention.** ADR-004 §89 mentions "long-running workflow tenants extend via Rego policy per ADR-015". Sprint 8.5 ships kernel-level Settings caps (default 24h); per-tenant Rego policy extension is deferred to a later sprint (Sprint 9 evidence export OR Sprint 13.5 approval engine is the candidate home).
- **Distributed reaper coordination.** Sprint 8.5 ships a single-instance background reaper (asyncio task) per AgentOS process. Multi-instance AgentOS deployments running parallel reapers would double-purge — the `ObjectStoreAdapter.delete()` idempotency makes this safe correctness-wise (second delete no-ops or raises `RetentionWindowActiveError`); audit row idempotency is achieved via per-checkpoint-id append-with-precondition. Cross-instance leader election deferred to Sprint 10.5 (scheduler primitive is the natural coordination point).
- **UI surface for paused sessions.** Sprint 7B.4 T10 action handler emits `action_backend_deferred_sandbox_unwired` on `resume`. Sprint 8.5 lifts that deferred-stub — the action handler routes to `sandbox.wake()`. Operator UI surfaces for "list paused sessions" / "approve resume" are deferred to a later sprint (Sprint 13.5 approval engine is the candidate home).

## 2. Doctrinal locks (verbatim from design calls 2026-05-18)

### 2.1 Workspace-tar, not CRIU (Q1)

> "Use /workspace tar snapshots + metadata + lease refs. No CRIU in Wave-1. No docker commit. This gives Docker/K8s parity and works on OpenShift without node-level CRIU privileges. Be explicit: checkpoints are between exec calls, not mid-process snapshots. ADR-004 should get a small amendment clarifying 'filesystem deltas' means writable workspace snapshot, not live process state."

Both Wave-1 backends ship the SAME checkpoint mechanism: `tar` of the writable `/workspace` path (the canonical writable mount declared by every canonical sandbox-runtime image per the ADR-004 amendment landing alongside this spec) + env metadata + Vault lease refs. DockerSibling does NOT use `docker commit` (would tie semantics to a Docker-specific container-layer abstraction); KubernetesPod does NOT use CRIU (alpha + node-level privileges incompatible with OpenShift restricted SCC).

The mechanism IS the cross-backend wire-public contract for checkpoint shape — `tests/conformance/sandbox/` parametrizes both backends through the same checkpoint round-trip assertion.

### 2.2 Dedicated `checkpoint_purged` audit event (Q2)

> "Add `sandbox.lifecycle.checkpoint_purged`. Do not fold into generic lifecycle rows. Reaper actions need examiner-visible evidence."

The reaper emits a dedicated chain row for every purge. Examiners reading the chain can prove "this checkpoint existed at T1, was purged at T2, retention window was N seconds" without inferring from the absence of subsequent chain rows. Folding into a generic `sandbox.lifecycle.destroyed` row would make purge-vs-explicit-destroy indistinguishable in the chain.

### 2.3 Wake-time policy revalidation (Q3)

> "wake() reruns admit_policy against the current tenant policy/catalog/Rego/settings using the stored SandboxPolicy + PackAdmissionContext. Add sandbox_wake_policy_revalidation_failed. A suspended session must not bypass newly tightened policy."

`wake()` calls the same `admit_policy()` Stage-2 pipeline that `create()` runs — against the CURRENT tenant policy, the CURRENT canonical image catalog, the CURRENT `sandbox.rego` bundle, the CURRENT per-tenant max-* Settings. Any refusal from the live pipeline surfaces as `sandbox_wake_policy_revalidation_failed` (NOT the original create-time refusal taxonomy — wake-time refusals are wake-specific).

**Bank trust posture:** a long-suspended session that pre-dates a security tightening MUST NOT bypass the tightening just because it was admitted under an older policy. The defence-in-depth is non-negotiable.

### 2.4 Fail-loud Vault-lease path now (Q4) — AMENDED 2026-05-18 post-source-verification

**Original user lock:** "Store lease refs in checkpoint metadata now. On wake with vault_path, call the consumer-owned credential/lease seam; KernelDefaultCredentialAdapter fails loud until Sprint 10 lands the real Vault adapter. Do not silently re-use old leases. If the existing CredentialAdapter shape needs extending, treat that as an 8.5 wire-contract change and update the T8 negative pins intentionally."

**Amendment after source verification at `sandbox/admission.py:96-105` + `tests/unit/sandbox/test_credential_adapter_stub.py:24+99-116`:** the existing Sprint-8A `CredentialAdapter` Protocol surface is **exactly** `async def fetch_secret(path) -> str | None` — there is no `acquire(policy)` method. The Sprint-8A negative pins explicitly forbid `mint_lease` / `revoke_lease` until Sprint 10. The original lock's escape clause ("If the existing CredentialAdapter shape needs extending, treat that as an 8.5 wire-contract change") applies — BUT extending the Protocol now expands Sprint 8.5 scope to include the Sprint-10 lease-API design + negative-pin flip + Sprint-10 forward-compat surface.

**Honest scope-minimisation:** the existing Sprint-8A admission-time refusal `sandbox_credential_adapter_not_configured` (raised at `admission.py` step 3) means no `policy.vault_path:`-bearing session can be CREATED today against the fail-loud `KernelDefaultCredentialAdapter` sentinel. No vault-bearing session created → no vault-bearing session to suspend → no vault-bearing wake reachable. The vault-lease re-issue path is **naturally deferred to Sprint 10** where:

1. `VaultCredentialAdapter` lands (replaces the sentinel).
2. The `CredentialAdapter` Protocol is extended with `mint_lease` / `revoke_lease` (the Sprint-8A negative pins flip to positive pins).
3. `suspend()` / `wake()` lease-handling logic is added on both backends — they ALREADY exist post-Sprint 8.5 but they fail-loud on `vault_path:` bearing policies until Sprint 10 wires the adapter through.

**What Sprint 8.5 ships (Q4 amended):** CheckpointMetadata carries `vault_lease_refs: tuple[VaultLeaseRef, ...] = ()` for forward-compat wire shape (always empty in 8.5 because no vault-bearing session reaches checkpoint). The `VaultLeaseRef` dataclass is defined now (§3.4) so Sprint 10 can fill it without a wire-shape change. NO `CredentialAdapter` Protocol extension. NO `mint_lease` / `revoke_lease` methods. NO change to Sprint-8A `test_credential_adapter_stub.py` negative pins. NO silent lease re-use (impossible to reach today; forward-compat shape prevents it in Sprint 10 too).

This is a scope-discipline call against the original lock's escape clause — extending `CredentialAdapter` is a Sprint-10 concern, and Sprint 8.5 stays clean by deferring honestly.

### 2.5 Scheduler forward-compat + caller identity (Q5)

> "Design wake() with the same admission identity seams as create(), so Sprint 10.5 can wrap it later. I would make the backend Protocol signature something like: `wake(session_id, *, actor, tenant_id) -> SandboxSession`. ADR's `sandbox.wake(session_id)` can remain shorthand, but implementation needs caller identity for tenant isolation, revalidation, and audit actor_id."

`SandboxBackend.wake()` Protocol signature:

```python
async def wake(
    self,
    session_id: str,
    *,
    actor: Actor,
    tenant_id: str,
) -> SandboxSession: ...
```

Identical identity-seam shape to `SandboxBackend.create()` (which takes `*, actor: Actor, tenant_id: str, pack_context: PackAdmissionContext, use_warm_pool: bool = True`). Sprint 10.5's `SchedulerEngine.submit()` wrapping logic at create-time applies the same wrapping logic at wake-time additively.

ADR-004 §84's `sandbox.wake(session_id)` example wording stays as shorthand for the module-level public API; the Protocol method signature is the implementation contract that requires caller identity.

### 2.6 Extra design lock — session_id alone is NEVER authorization (added 2026-05-18)

> "session_id alone is never authorization. Wake must compare caller tenant_id against checkpoint metadata tenant_id and fail closed with sandbox_wake_tenant_mismatch."

This is a wake-time enforcement contract that pre-empts the entire class of "guess a session_id from another tenant + call wake()" attacks. Implementation:

1. `wake(session_id, *, actor, tenant_id)` loads the `CheckpointMetadata` from `ObjectStoreAdapter` keyed by `<tenant_id>/<session_id>/...` — the per-tenant prefix means a cross-tenant wake attempt CAN'T find the checkpoint (the lookup at the caller's `tenant_id` prefix returns nothing).
2. BUT defence-in-depth: even if a future ObjectStoreAdapter driver flattens the prefix or the wrapper bug bypasses it, the loaded `CheckpointMetadata.tenant_id` field is COMPARED against the caller's `tenant_id` kwarg explicitly. Mismatch → `SandboxLifecycleRefused("sandbox_wake_tenant_mismatch")`.
3. Audit emits the refusal under `sandbox.lifecycle.refused` (existing 8A taxonomy) with the closed-enum `sandbox_wake_tenant_mismatch` reason. Examiners can prove the cross-tenant-wake attempt was caught.

This makes wake symmetric with the broader AgentOS pattern (per `feedback_strict_review_off_gate` + the Sprint 7B.2 cross-tenant 404 doctrine) — never trust an opaque ID for authorization; always cross-check against the caller's identity claim.

## 3. Protocol surface

### 3.1 `SandboxSession` Protocol extensions

`sandbox/protocol.py` Sprint-8A definition at `:226-250` gets +2 methods (additive — existing methods + 6-field shape unchanged):

```python
@runtime_checkable
class SandboxSession(Protocol):
    # Unchanged Sprint-8A fields + methods:
    session_id: str
    policy: SandboxPolicy
    tenant_id: str
    pack_context: PackAdmissionContext
    created_at: datetime
    warm_pool_hit: bool

    async def exec(self, command: list[str], *, timeout_s: float | None = None) -> SandboxExecResult: ...
    async def destroy(self) -> None: ...

    # NEW Sprint 8.5:
    async def checkpoint(self, label: str) -> CheckpointId:
        """Persist a workspace-tar snapshot + env metadata + Vault lease refs
        to the CheckpointStore. Emits sandbox.lifecycle.checkpointed.
        Returns the opaque CheckpointId; same session can checkpoint
        multiple times (subject to Settings.sandbox_max_checkpoints_per_session).
        """

    async def suspend(self) -> None:
        """Take a final checkpoint (label='__suspend__') and release the
        underlying container/Pod. Subsequent exec() calls on this session
        raise. wake() restores the session in a fresh backend resource.
        Emits sandbox.lifecycle.suspended.

        Vault-lease revocation is OUT OF SCOPE for Sprint 8.5 per §2.4
        amended — the existing Sprint-8A `sandbox_credential_adapter_not_configured`
        admission-time refusal prevents any vault-bearing session from
        existing in the first place, so there is no lease to revoke here.
        Lease-revocation logic lands at Sprint 10 alongside the real
        VaultCredentialAdapter + the CredentialAdapter Protocol extension.
        """
```

**`destroy()` behavior extension (Sprint 8.5, per P1.r4 tombstone redesign — Protocol signature unchanged from 8A; behavior changes inside backends):** when `destroy()` is called on a session that has any persisted checkpoints (i.e. any checkpoint() was called AND/OR suspend() was called), the backend additionally calls `CheckpointStore.tombstone_session(session_id=self.session_id, tenant_id=self.tenant_id, tombstoned_by=self._actor_subject)` which writes a `<tenant>/<session>/_tombstoned.json` sentinel blob. The `tombstoned_by` value comes from the session's existing `_actor_subject` field — captured at create-time by both Wave-1 backends and already used as the audit-row `actor_id` source for every other lifecycle event (`docker_sibling.py:677` declaration + `:870` create-time capture + `:1581/1630/1661` audit-row consumers; `kubernetes_pod.py:472/668` symmetric). The Sprint-8A `SandboxSession.destroy(self) -> None` Protocol signature is **unchanged** — destroy() takes no actor parameter (verified at `protocol.py:250`); per the P2.r5 fix, the spec deliberately does NOT extend destroy() to accept an actor kwarg because the session's stored `_actor_subject` is the authoritative source of "who owned this session" for audit-row attribution (changing it post-create would diverge from the create-time admit_policy decision the actor was authorised against). The session's checkpoint bytes are NOT physically deleted by destroy() — they are retained until the reaper sweeps them after the per-tenant retention window expires (per §4.3 amended). The existing 8A `sandbox.lifecycle.destroyed` audit event is emitted with two NEW payload keys: `retained_until: <iso-string>` (= `now + settings.sandbox_checkpoint_retention_s`) AND `tombstone_object_key: <storage-key>` (so examiners can correlate destroy → tombstone → eventual reaper purge). Sessions with NO persisted checkpoints (immediate destroy without ever calling checkpoint() / suspend()) skip the tombstone write — there's nothing to retain — and emit `sandbox.lifecycle.destroyed` WITHOUT the `retained_until` / `tombstone_object_key` payload keys (presence of these keys is the wire-public marker that retention is in effect). Wake() against a tombstoned session refuses with `sandbox_wake_session_tombstoned` per §3.2 step 1(a) — destroyed sessions are NOT wakeable, even though the bytes might still be on disk.

### 3.2 `SandboxBackend` Protocol extensions

`sandbox/protocol.py` Sprint-8A definition at `:254-318` gets +1 method (additive):

```python
@runtime_checkable
class SandboxBackend(Protocol):
    # Unchanged Sprint-8A methods (create / exec / destroy / health) ...

    # NEW Sprint 8.5:
    async def wake(
        self,
        session_id: str,
        *,
        actor: Actor,
        tenant_id: str,
    ) -> SandboxSession:
        """Restore a suspended session from its latest checkpoint.

        Pipeline (security-reviewed ordering):
        1. Load CheckpointMetadata from CheckpointStore by (tenant_id, session_id).
           **Three distinct failure modes** (NOT folded — closed-enum
           distinction load-bearing for examiner incident-response paths):
             (a) `CheckpointStore.load_tombstone(session_id=..., tenant_id=...)`
                 returns a non-None TombstoneRecord (the session was
                 destroy()ed; bytes retained until reaper sweep per
                 P1.r4 tombstone semantics) → SandboxLifecycleRefused(
                 "sandbox_wake_session_tombstoned") with tombstone
                 metadata (tombstoned_at / tombstoned_by / retained_until)
                 in the detail field. **CHECKED FIRST** via the dedicated
                 `load_tombstone()` read helper (P1.r5 fix — there is
                 NO fall-through into `load_latest()`; the tombstone
                 check is its own seam so the wake-pipeline ordering is
                 explicit + the closed-enum mapping stays live at the
                 wake() call site, NOT folded into load_latest()).
                 **P1.r6 fail-closed correction:** if `load_tombstone()`
                 raises `TombstoneCorruptError` (sentinel exists but is
                 malformed / tampered), wake() catches and maps to the
                 SAME `SandboxLifecycleRefused("sandbox_wake_session_tombstoned")`
                 closed-enum value — with the corrupt-tombstone exception
                 message in the `detail` field for incident-response
                 traceability. Operator intent ("session was destroyed;
                 MUST NOT wake") survives a tampered tombstone; closed-enum
                 count stays at 21 (no new value; the tampering signal
                 lives in `detail`, not in the enum vocabulary).
             (b) No tombstone AND metadata missing OR already purged →
                 SandboxLifecycleRefused("sandbox_wake_checkpoint_not_found").
             (c) No tombstone, metadata bytes exist on disk, but
                 CheckpointMetadata.from_storage_payload() raises
                 ValueError (malformed / tampered / schema-drift past
                 the unknown-key tolerance) → SandboxLifecycleRefused(
                 "sandbox_wake_checkpoint_corrupt") with the original
                 ValueError's message in the detail field. Examiners can
                 distinguish destroyed-vs-purged-vs-tampered without
                 correlating bytes.
        2. Cross-check metadata.tenant_id against caller tenant_id kwarg
           (defence-in-depth past the prefix-keyed lookup).
           Raises SandboxLifecycleRefused("sandbox_wake_tenant_mismatch")
           on mismatch.
        3. Check metadata.retention_window_s against (now - metadata.created_at).
           Raises SandboxLifecycleRefused("sandbox_wake_checkpoint_retention_expired")
           if expired (defence-in-depth past the reaper; the reaper MAY
           not have run yet for a just-expired checkpoint).
        4. Re-run admit_policy(policy=metadata.policy, pack_context=metadata.pack_context,
           actor=actor, tenant_id=tenant_id) against the LIVE tenant policy /
           catalog / Rego / settings. Refusal surfaces as
           SandboxLifecycleRefused("sandbox_wake_policy_revalidation_failed").
           This INCLUDES the existing Sprint-8A step 3 credential precondition
           refusal: if metadata.policy carries vault_path and the wired adapter
           is still KernelDefaultCredentialAdapter, admit_policy step 3
           raises sandbox_credential_adapter_not_configured — which surfaces
           HERE as sandbox_wake_policy_revalidation_failed (the wake-time
           refusal taxonomy is wake-specific per §2.3; the original 8A reason
           lives in the refusal's detail field for examiner traceability).
        5. Create fresh backend resource (container or Pod), restore
           workspace-tar snapshot into /workspace. Vault-lease re-issue is
           deferred to Sprint 10 per §2.4 amended (no vault-bearing session
           can reach wake today because step 4 refuses fail-loud).
        6. Return a fresh SandboxSession with the ORIGINAL session_id
           (continuity is the whole point) + new container/Pod IDs.
        7. Emit sandbox.lifecycle.woken chain row carrying explicit payload
           keys ``suspend_event_id`` (the record_id UUID returned from
           the suspend-time emit_sandbox_event call, captured into the
           wake-emit payload — the payload-key NAME ``suspend_event_id``
           is wire-public per ADR-006 for examiner-readability; it points
           at the suspended row's primary-key ``record_id`` column) +
           ``restored_from_checkpoint_id`` (pointing at the loaded
           CheckpointMetadata.checkpoint_id) for the chain-verifier walk.
           NO new DecisionRecord column is added (the existing record_id /
           trace_id / span_id / langfuse_trace_id columns at
           decision_history.py:188-199 are unchanged); linkage lives in
           payload per the standard sandbox.audit emission pattern.
        """
```

`sandbox_wake_checkpoint_corrupt` (P2.r3 fix) — `CheckpointMetadata.from_storage_payload()` raises `ValueError` on malformed metadata (missing required key, tuple-shaped value smuggled in, naive datetime, etc); the wake-pipeline catches this and surfaces as `SandboxLifecycleRefused("sandbox_wake_checkpoint_corrupt")` with the original `ValueError`'s message in the `detail` field. Examiners can distinguish "checkpoint was purged" (`sandbox_wake_checkpoint_not_found`) from "checkpoint exists on disk but bytes are tampered/corrupted" (`sandbox_wake_checkpoint_corrupt`) — different incident response paths.

`sandbox_wake_session_tombstoned` (P1.r4 tombstone redesign) — destroy() writes a `<tenant>/<session>/_tombstoned.json` sentinel via `CheckpointStore.tombstone_session()`. Checkpoint bytes are retained until the reaper sweeps them (after the per-tenant retention window expires; see §4.3 amended). Wake() detects the sentinel BEFORE loading metadata and refuses fail-closed — a destroyed session MUST NOT be wakeable, regardless of whether the underlying checkpoint bytes are still on disk. Examiners can distinguish "session was destroyed by operator" (`sandbox_wake_session_tombstoned`) from "checkpoint reaped" (`sandbox_wake_checkpoint_not_found`) from "checkpoint corrupted" (`sandbox_wake_checkpoint_corrupt`) — three different incident response paths.

**`SandboxPolicyViolationReason` UNCHANGED** at 6 values (8.5 introduces no new runtime exec violations).

### 3.4 New wire-public dataclasses

```python
# sandbox/checkpoint_store.py — NEW module

CheckpointId = NewType("CheckpointId", str)
# Static-only type alias. NewType is a static-analysis hint that does
# NOT enforce runtime opacity (an arbitrary str can still be cast to
# CheckpointId at runtime). Runtime construction discipline is enforced
# by CheckpointStore.mint_checkpoint_id() (uuid4 hex; the ONLY mint
```

### 3.5 `ObjectStoreAdapter` Protocol extension (REQUIRED — additive)

Per the P1.1 source-verification finding, the existing Sprint-4 `ObjectStoreAdapter` Protocol at `db/adapters/protocols.py:141-206` exposes only `put` / `get` / `delete` / `presign` / `health_check` — there is no list/prefix API. Sprint 8.5's `CheckpointStore.load_latest()` (needs to find the newest checkpoint for a session) + `purge_expired()` (needs to walk all tenants/sessions) cannot be implemented as written without a listing mechanism.

**Decision (per user's two options at the P1.1 finding):** extend `ObjectStoreAdapter` with a `list_prefix()` method rather than design a manifest-object-with-rollback-semantics. Rationale:

- **Architecturally honest.** Listing is a normal blob-store operation. The Sprint-4 `local_fs` driver implements it via `pathlib.Path.iterdir()` or `os.scandir()` walking the prefix directory; future `s3` driver (Sprint 8 backlog) maps to `ListObjectsV2`. Both are O(prefix-size) lazy iterators.
- **Eliminates manifest-object corruption design space.** Manifest objects would need their own consistency invariants, rollback semantics, reader-writer race handling, etc. — all of which is solved-once-and-correctly by the blob-store's native listing.
- **Sprint-4 wire-contract change is intentional + additive.** Existing callers (none today) are unaffected; the Protocol grows.

NEW method on `ObjectStoreAdapter`:

```python
def list_prefix(
    self,
    bucket: str,
    prefix: str,
) -> AsyncIterator[str]:
    """Lazily yield keys under ``bucket/prefix``.

    **Async-generator shape — NOT a coroutine.** Declared as plain
    ``def`` returning ``AsyncIterator[str]`` (NOT ``async def``).
    Implementations are async-generator functions (``async def`` +
    ``yield``); calling the method returns an ``AsyncIterator`` DIRECTLY
    without await. This is the standard Python idiom for lazy async
    iterators per PEP 525 + the typing.AsyncIterator convention.

    The wrong shape — ``async def f() -> AsyncIterator[str]`` — would
    type as "coroutine returning an AsyncIterator" and force callers
    to ``async for x in (await f())`` instead of the natural
    ``async for x in f()``. This Protocol declaration locks the
    right call shape across all driver implementations.

    Caller pattern (the only supported one):

        async for key in object_store.list_prefix("bucket", "prefix/"):
            ...

    Yields full keys (NOT relative-to-prefix) in lexicographic order.
    Empty result is a normal empty iterator (NOT an exception).
    Implementations MUST be lazy — yielding one key per loop iteration
    rather than loading the full key list into memory — so reaper
    sweeps over multi-million-key tenants don't OOM.

    The Sprint-4 ``local_fs`` driver walks the prefix directory via
    ``os.scandir()`` recursively (async-generator function yielding
    one key at a time); the future Sprint-8 ``s3`` driver maps to
    ``ListObjectsV2`` with continuation-token pagination (async-generator
    function yielding keys page-by-page).
    """
    ...
```

**Sprint-4 `local_fs` driver gets the method:** `src/cognic_agentos/db/adapters/local_object_store_adapter.py` adds `async def list_prefix(...)` calling `os.scandir()` recursively under `<root>/<bucket>/<prefix>`. Yields keys (not paths). Defence-in-depth: refuses keys outside the bucket prefix (paranoid pathlib check against `..` traversal attempts).

**Protocol-conformance test:** `tests/unit/db/adapters/test_object_store_adapter_list_prefix.py` (NEW) pins the lazy-iteration contract, lexicographic ordering, empty-prefix behavior, missing-bucket behavior, and the path-traversal defence-in-depth. Drift would surface there before the Sprint-8.5 `CheckpointStore` notices.

**This adds T-task work to Sprint 8.5** — see §12 T-task decomposition (NEW T0.5 for the Sprint-4 Protocol extension before the Sprint-8.5 `CheckpointStore` lands).

### 4.3 Retention-floor enforcement (P1.r3 amended + P1.r4 tombstone redesign + P3.r4 purge-reason discipline)

`CheckpointStore.purge_expired()` walks all checkpoint metadata + tombstone sentinels via `list_prefix()` per §4.1. For each session-prefix:

1. **Tombstoned session path:** if `_tombstoned.json` exists, read its `retained_until` field. If `now >= retained_until`, purge ALL session checkpoints + the tombstone sentinel itself; emit one `sandbox.lifecycle.checkpoint_purged` chain row per checkpoint with `purge_reason="explicit_destroy"` (the original operator action that triggered the deferred purge). Otherwise skip (tombstone retention not yet elapsed; checkpoints retained as evidence).
2. **Non-tombstoned session path:** for each `<checkpoint_id>.metadata.json` blob, read `metadata.created_at` + `metadata.retention_window_s`. If `(now - created_at) >= retention_window_s`, purge the metadata + snapshot pair with `purge_reason="retention_expired"`. Otherwise skip.

**4 `purge_reason` values remain CLOSED + UNCHANGED:** `explicit_destroy` / `max_per_session_cap` / `retention_expired` / `tenant_revocation`. The retention-floor check is an INVARIANT that runs BEFORE any purge attempt; it does NOT introduce a new audit reason. Per the P3.r4 fix, NO `retention_window_active` purge_reason is invented — when retention would block a purge, the right answer is: don't emit a checkpoint_purged row (no purge happened) AND signal the configuration-tension via a typed exception at the persist() call site (`CheckpointMaxPerSessionRetentionLocked` — see below).

**`purge_by_id()` is internal-only** (reaper-callable + persist()-callable for cap eviction) — NOT called by `destroy()` directly per the P1.r4 tombstone redesign. The reaper pre-checks retention before calling, so `purge_by_id()` always succeeds at the storage layer (no WORM lock per the P1.r3 fix). There is no `purge_by_id` failure mode for "blocked by retention" because the reaper never asks it to purge a retained checkpoint.

**Bank-grade WORM compliance (deferred to Sprint 9):** the reaper-enforced retention floor is REVERSIBLE — an AgentOS admin with disk access could delete the metadata blob out-of-band. True unforgeable WORM evidence requires Sprint 9's evidence-pack export to COPY the metadata into a separate compliance bucket with `ObjectStoreAdapter.put(retention_seconds=settings.sandbox_checkpoint_retention_s)` BEFORE the operational checkpoint bucket allows purge. Sprint 8.5 ships the operational-bucket retention discipline; Sprint 9 hardens it to bank-grade WORM via the evidence-pack copy.

**Max-per-session eviction interaction:** when `persist()` would exceed `settings.sandbox_max_checkpoints_per_session`, it evicts the OLDEST checkpoint that is OUTSIDE its retention window (purge_reason='max_per_session_cap'). If every existing checkpoint is INSIDE its retention window (eviction would be blocked by retention), `persist()` raises `CheckpointMaxPerSessionRetentionLocked` WITHOUT writing the new checkpoint AND WITHOUT emitting any `checkpoint_purged` chain row (no purge happened; the typed exception IS the operator-observable signal — per the P3.r4 fix). The exception carries `(session_id, tenant_id, cap, oldest_retention_remaining_s)` for operator response (lower retention OR raise cap). NOT a `SandboxRefusalReason` — this is a configuration-tension surface, not a wake-time refusal. Pinned by `test_checkpoint_max_per_session_retention_tension.py` in §8.

The payload-key NAME `suspend_event_id` is wire-public per ADR-006 for examiner-readability; its VALUE is the UUID returned as the first tuple element from the suspend-time `emit_sandbox_event` call (i.e. the suspended row's primary-key `record_id` column at `decision_history.py:188`). NO new DecisionRecord column added; the chain-verifier walker (§5.2) looks up `decision_history WHERE record_id = payload["suspend_event_id"]` to find the matching suspend row.
- `sandbox_lifecycle_checkpoint_purged(*, tenant_id, session_id, checkpoint_id, purge_reason, actor_subject="reaper") -> DecisionRecord`
  - Payload keys: `tenant_id` / `session_id` / `checkpoint_id` / `purge_reason` (closed-enum: `"explicit_destroy"` / `"max_per_session_cap"` / `"retention_expired"` / `"tenant_revocation"` — unchanged 4-value set per the P3.r4 discipline).

**Existing 8A `sandbox.lifecycle.destroyed` emitter — extended payload (NOT a new emitter; Sprint 8.5 adds 2 optional payload keys per the P1.r4 tombstone redesign):**

- `sandbox_lifecycle_destroyed` (Sprint 8A `sandbox/audit.py` — unchanged function signature; extended payload):
  - Existing payload keys: `session_id` / `tenant_id` (per Sprint 8A).
  - NEW Sprint 8.5 payload keys (conditional — only present when the destroyed session had any persisted checkpoints): `retained_until: <iso-string>` (= `now + settings.sandbox_checkpoint_retention_s` at destroy() time) + `tombstone_object_key: <storage-key>` (= the `<tenant>/<session>/_tombstoned.json` key returned from `CheckpointStore.tombstone_session()`). Presence of these two keys is the wire-public marker that retention is in effect for this session's checkpoints; absence means immediate physical destroy (no tombstone needed because there was nothing to retain). The chain-verifier extension at §5.2 does NOT inspect these new keys (suspend→wake linkage is the verifier's only concern); they exist for examiner correlation across destroy → tombstone → eventual reaper purge.

Each emitter follows the same pattern as the Sprint-8A 8 — pure-functional, tz-aware timestamps per `feedback_evidence_boundary_runtime_validation`, canonical_bytes-safe payload shapes per `feedback_evidence_boundary_runtime_validation` (no tuples in payloads — list shape only).

### 5.2 Chain verifier extension

Existing `core/chain_verifier.py` (a Sprint-2 critical-controls module) walks `decision_history` rows in sequence + verifies hash-chain integrity. Sprint 8.5 adds a NEW assertion that walks via **payload keys** (NOT via a non-existent `parent_trace` column — the existing `DecisionRecord` schema has only `trace_id` / `span_id` / `langfuse_trace_id` per `core/decision_history.py:197-199`, none of which carry per-event parent linkage):

For every `sandbox.lifecycle.woken` row, the verifier runs a **7-check assertion** — the baseline payload-linkage steps 1-5 plus two bank-evidence invariants (6-7) added during the T8 review round (cross-tenant linkage hole + accepted wake-before-suspend ordering; see the T8 amendment note below):

1. Reads `payload["suspend_event_id"]` from the woken row.
2. Looks up the `decision_history` row WHERE `record_id` (the actual primary-key column per `decision_history.py:188`) equals that value.
3. Asserts the looked-up row's `decision_type == "sandbox.lifecycle.suspended"`.
4. Asserts the suspended row's `payload["session_id"] == woken row's payload["session_id"]`.
5. Asserts the suspended row's `payload["final_checkpoint_id"] == woken row's payload["restored_from_checkpoint_id"]`.
6. **Tenant-isolation parity** — asserts the suspended row's `tenant_id` equals the woken row's `tenant_id`. `tenant_id` is a `decision_history` **row column** (NOT a payload key), so steps 4-5 cannot catch a cross-tenant linkage: without check 6, tenant B's forged woken row could point at tenant A's suspended row and verify clean whenever the `session_id` + checkpoint strings collide.
7. **Causal ordering** — asserts the suspended row precedes the woken row in chain `sequence` (`suspend_row.sequence < woken_row.sequence`). `suspend_event_id` is the `record_id` *returned by an earlier suspend emit*; a woken row linking forward to a later suspended row is not a genuine resume.

Check order is steps 1-3 → invariants 6-7 → steps 4-5, so a cross-tenant or out-of-order link surfaces as its own failure class rather than being masked by a `session_id` / checkpoint mismatch.

This is the "suspend/wake chain integrity verifies via Merkle proof" requirement per BUILD_PLAN §738. A forged wake (no matching suspend row at `suspend_event_id`), OR a wake claiming restoration from a checkpoint id that doesn't match the suspend row, OR a wake that points at a non-suspend row, OR a wake whose `tenant_id` differs from the linked suspend row, OR a wake that links forward to a later suspend row → chain verifier fails. The hash chain itself (verified row-by-row by the Sprint-2 base walker) ensures the suspended row's payload can't be tampered with after the fact.

The verifier returns the first linkage break via a closed-enum `SuspendWakeBreakKind` literal — **8 examiner-facing failure classes**: `woken_missing_suspend_event_id` / `woken_missing_restored_checkpoint_id` / `suspend_row_not_found` / `suspend_event_id_wrong_decision_type` / `tenant_id_mismatch` / `suspend_row_not_before_woken_row` / `session_id_mismatch` / `checkpoint_id_mismatch`. This vocabulary is wire-public for evidence-pack export per ADR-006.

> **T8 amendment (review round, 2026-05-20).** §5.2 originally specified a 5-step verifier. The T8 review round found that the 5-step contract left two gaps in a bank evidence verifier: (a) it never checked the row-level `tenant_id`, so a cross-tenant woken→suspended link verified clean whenever the payload strings collided; (b) it indexed the whole chain before validating, so a woken row could link forward to a *later* suspended row and verify clean even though `suspend_event_id` is by definition the id of an *earlier* event. Checks 6-7 + the two failure classes `tenant_id_mismatch` / `suspend_row_not_before_woken_row` close both. The 5-step list was a floor, not a ceiling, for the evidence verifier.

**Critical-controls implication:** `core/chain_verifier.py` is already on the durable coverage gate (Sprint 2). Sprint 8.5 extends its responsibility — the new suspend→wake assertion logic gets coverage on the same module + the existing 95%/90% floor applies to the new code. **NO `decision_history` schema migration** needed — the new linkage is payload-key-only (and `tenant_id` is an already-existing row column).

## 7. Backend-specific snapshot mechanics

Both backends implement the SAME `_create_workspace_tar()` + `_restore_workspace_tar()` shape but with backend-specific I/O paths.

### 7.1 DockerSibling backend

- `checkpoint()`: `aiodocker` exec into the running container — `tar czf - -C /workspace . | base64` (or stream the tar bytes directly through the exec channel). Capture stdout bytes. Construct `CheckpointMetadata` with `vault_lease_refs=()` (always empty in 8.5 per §2.4 amended). Pass bytes + metadata to `CheckpointStore.persist()`.
- `suspend()`: takes final checkpoint, then `container.stop()` + `container.remove()` per the existing Sprint 8A teardown helpers. **Vault-lease revocation deferred to Sprint 10** per §2.4 amended — the Sprint-8A admission gate prevents any vault-bearing session from being created, so there is no lease in flight to revoke here.
- `wake()`: **`CheckpointStore.load_tombstone()` FIRST** (per the P2.r7 fix mirroring the §3.2 canonical pipeline) — non-None `TombstoneRecord` OR `TombstoneCorruptError` raised → `SandboxLifecycleRefused("sandbox_wake_session_tombstoned")` per the P1.r6 fail-closed contract (NOT a fall-through to load_latest()). Only on `None` (genuinely absent tombstone) does the backend proceed: `CheckpointStore.load_latest()` → admit_policy revalidate (which itself refuses fail-loud for vault-bearing policies via the existing 8A `sandbox_credential_adapter_not_configured` step 3) → `backend.create(...)` for a fresh container → `aiodocker` exec `tar xzf - -C /workspace` with the snapshot bytes piped on stdin. **No Vault lease re-issue step** per §2.4 amended; Sprint 10 adds the re-issue helper invocation here when `VaultCredentialAdapter` lands. The tombstone-first ordering is the LOAD-BEARING wake-pipeline invariant — T6 implementers MUST NOT skip it. Per the P3.r8 fix: the unit `test_wake_session_tombstoned.py` (§8) pins the closed-enum + detail-field invariants; **cross-backend parity** (i.e. that BOTH DockerSibling and KubernetesPod call `load_tombstone()` first) is pinned by the NEW `tests/conformance/sandbox/test_wake_session_tombstoned_conformance.py` parametrised-over-both-backends regression alongside `test_checkpoint_round_trip.py`. A backend wake() that started with load_latest() would surface destroyed-but-not-yet-reaped sessions as `sandbox_wake_checkpoint_not_found` — wrong taxonomy + violates the §3.2 step ordering + would fail the conformance regression.

### 7.2 KubernetesPod backend

- `checkpoint()`: `kubernetes_asyncio` exec into the Pod's sandbox container — same `tar czf - -C /workspace .` pattern over the multiplexed websocket channels. STDOUT channel carries the tar bytes; STDERR channel carries any tar errors; ERROR channel surfaces exit code. Constructs `CheckpointMetadata` with `vault_lease_refs=()` (always empty in 8.5).
- `suspend()`: takes final checkpoint, then `Pod.delete()` + `NetworkPolicy.delete()` per the existing Sprint 8B teardown helpers (404-swallow / non-404-propagate idempotency preserved). **Vault-lease revocation deferred to Sprint 10** per §2.4 amended; same fail-closed admission-gate rationale as DockerSibling.
- `wake()`: **`CheckpointStore.load_tombstone()` FIRST** (per the P2.r7 fix mirroring the §3.2 canonical pipeline) — non-None `TombstoneRecord` OR `TombstoneCorruptError` raised → `SandboxLifecycleRefused("sandbox_wake_session_tombstoned")` per the P1.r6 fail-closed contract (NOT a fall-through to load_latest()). Only on `None` (genuinely absent tombstone) does the backend proceed: `CheckpointStore.load_latest()` → admit_policy revalidate → `backend.create(...)` for a fresh Pod + NetworkPolicy → `kubernetes_asyncio` exec `tar xzf - -C /workspace` with the snapshot bytes piped on stdin. **No Vault lease re-issue step** in 8.5; Sprint 10 adds it. Tombstone-first ordering symmetric with §7.1 DockerSibling per the cross-backend wire-public parity invariant; T7 implementers MUST NOT skip it. Per the P3.r8 fix: cross-backend parity for tombstone wake refusal is pinned by the NEW `tests/conformance/sandbox/test_wake_session_tombstoned_conformance.py` parametrised-over-both-backends regression (per §8 / §7.3 — listed alongside `test_checkpoint_round_trip.py`); the unit `test_wake_session_tombstoned.py` at §8 carries the closed-enum + detail-field invariants but cannot detect backend-asymmetric skips of the tombstone-first step, which is what the new conformance test is for.

### 7.3 Cross-backend parity

The workspace-tar mechanism + the tombstone-first wake ordering are the two cross-backend wire-public contracts Sprint 8.5 ships. `tests/conformance/sandbox/` extends with TWO parametrized-over-both-backends regressions (per the P3.r9 fix mirroring the §7.1 / §7.2 / §8 cites):

- **`test_checkpoint_round_trip_preserves_workspace_state`** — pins workspace-tar semantics across backends. Drift in tar-encoding semantics (gzip level, file mode preservation, symlink handling) would surface as conformance-suite failure.
- **`test_wake_session_tombstoned_conformance.py`** (NEW per the P3.r8 fix; cross-referenced from §8) — pins the tombstone-first wake invariant across backends. Three cases × two backends: (a) tombstoned session refuses wake; (b) tombstoned + valid metadata-on-disk refuses wake NOT restore (catches load_latest-first asymmetry); (c) corrupt `_tombstoned.json` + valid metadata refuses wake via `TombstoneCorruptError` fail-closed per P1.r6. A backend wake() that started with `load_latest()` instead of `load_tombstone()` would fail this regression. Env-gated symmetrically on `COGNIC_RUN_DOCKER_SANDBOX=1` AND `COGNIC_RUN_K8S_SANDBOX=1`.

Both conformance tests are the load-bearing cross-backend parity surface for Sprint 8.5; T9 wires them per §12. A future Wave-2 backend (gVisor / Firecracker per ADR-004 §27) MUST pass BOTH regressions to clear the cross-backend wire-public parity gate.

**`_shared_exec.py` extension (consumer-owned per `feedback_consumer_owned_protocol_for_unlanded_dep`):** if the tar-pipe logic ends up shared between backends, the helper lives at `sandbox/backends/_shared_exec.py` (NOT a separate module) consumer-owned by `kubernetes_pod` AND `docker_sibling` (drift detector pinned the same way as `_classify_exec_failure` Sprint 8B T8B-c).
