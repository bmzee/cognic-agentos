**Date:** 2026-05-22

## 1. Context & scope

AgentOS routes every LLM request through *some* model. Today the only identity
the runtime records is the LiteLLM alias (`tier1` / `tier2` → an alias string).
A bank examiner cannot answer "which fine-tuned model version handled this
case, and was it signed and approved?" Sprint 9.5 closes that gap.

**What this sprint builds — a Model Registry primitive (per ADR-013):** a
metadata + audit layer for the lifecycle of every model AgentOS routes a
request through. A new `models/` package (lifecycle state machine + storage +
trust gate), a `portal/api/models/` route package (7 endpoints), 8 RBAC
scopes, ISO-42001-tagged `model.lifecycle.*` hash-chained audit events, and a
gateway linkage that records the registered `model_id` on every LLM call.

**What this sprint does NOT build.** No model training, no weight storage, no
GPU orchestration, no fine-tuning framework. The registry stores *metadata + a
pointer* to the signed artefact (the GB-scale weights sit in an
`ObjectStoreAdapter` per ADR-009); it never stores weights. Fine-tuning
workflows are **Cognic Forge** — a separate Wave-2 repo. ADR-013 is explicit:
AgentOS sees the registered artefact, not the framework that produced it.

**Architectural fit.** The Model Registry is a **platform primitive** —
bundled in AgentOS, not a plugin pack (ADR-013 §Consequences/Neutral: "every
bank deployment needs it", same as audit / decision_history). It deliberately
mirrors the already-shipped, already-on-gate `packs/` lifecycle subsystem
(Sprint 7B.1-7B.3): a pure-functional state machine, a Postgres/Oracle store
that drives every transition through `DecisionHistoryStore.append_with_precondition`,
and a route package with closed-enum RBAC. Reusing that proven shape keeps the
blast radius small and the review surface familiar.

**Sprint shape.** Single sprint, single branch, single PR — built in three
internal blocks (A: `models/` domain; B: portal + RBAC; C: gateway linkage)
with a clean cut line. If execution swells, cut before Block C and ship
**9.5a** (registry fully usable via the portal); Block C (per-call gateway
linkage) ships as **9.5b**. Do not pre-split — only cut if execution proves it.

This is the same start pattern as Sprint 9 / #489: read the ADR + build-plan
section, brainstorm the design, write this spec, then a TDD plan.

## 2. Architecture — the `models/` package

A new top-level package `src/cognic_agentos/models/`, structured as a direct
mirror of `packs/lifecycle.py` + `packs/storage.py` + `protocol/trust_gate.py`.

```
src/cognic_agentos/models/
  __init__.py      — closed-enum + dataclass re-exports (off the CC gate, Doctrine F)
  registry.py      — pure-functional lifecycle state machine          [CRITICAL CONTROL]
  storage.py       — ModelRecordStore: Postgres/Oracle + chain emission [CRITICAL CONTROL]
  trust.py         — cosign signature verification of the model artefact [CRITICAL CONTROL]
```

`models/registry.py` and `models/trust.py` are **already named** as
critical-controls + a `models/` stop-rule boundary in `AGENTS.md` ("Model
registry lifecycle transitions (`models/` + `models/trust.py`)" /
"`models/registry.py` + `models/trust.py` (per ADR-013 …)") — a planted
forward-reference. They are born on the gate. Sprint 9.5 additionally promotes
`models/storage.py` (it owns the atomic chain-transaction surface) and
`portal/api/models/lifecycle_routes.py` (it owns the Human-only-decision
enforcement boundary) — see §7.

### 2.1 Lifecycle state machine (`models/registry.py`)

Pure-functional, no I/O — the exact discipline of `packs/lifecycle.py`. Owns:

- The `ModelLifecycleState` closed-enum: `proposed`, `eval_passed`,
  `tenant_approved`, `serving`, `deprecated`, `retired`.
- A `_VALID_TRANSITIONS` legal-pair table.
- `validate_transition(*, from_state, to_state, kind, transition)` — a
  keyword-only pure validator (positional misuse is a state-machine bug class;
  the keyword-only contract eliminates it, per `packs/lifecycle.py:421`).
- `iso_controls_for(transition)` — returns the canonical ISO-42001 control
  tuple stamped onto every chain row (see §4).
- A closed-enum `ModelLifecycleRefusalReason` Literal — the wire-protocol
  contract carried in every 409 refusal body. **Nine values, pinned** — a
  wire-public closed-enum: a missing reason discovered during implementation
  lands as a reviewed spec amendment first, never an ad-hoc plan-time addition:
  `model_transition_invalid_state_pair`, `model_transition_state_unknown`,
  `model_transition_from_terminal_state`, `model_register_duplicate_id`,
  `model_promote_signature_verification_failed`,
  `model_promote_signature_refs_changed_during_promote` (the TOCTOU guard —
  see §4.1), `model_promote_eval_evidence_missing`,
  `model_promote_eval_evidence_malformed`, `model_retire_already_retired`.
- `ModelLifecycleRefused(reason)` — one thin typed exception carrying only the
  closed-enum reason, raised for register + promote + retire alike (register
  is the genesis transition, so its refusals share the enum — a deliberate
  simplification of the packs two-class `LifecycleTransitionRefused` /
  `PackRecordRefused` split).
- `ModelNotFound` — a separate exception (→ HTTP 404), NOT in the closed-enum,
  for promote/retire against an unknown `model_id` (mirrors `packs`'
  `PackNotFound`).

```
  POST /models    ──promote──▶       ──promote──▶          ──promote──▶     ──promote──▶
  (genesis)       target=eval_passed target=tenant_approved target=serving  target=deprecated
       │          [sig-verify]       [eval-evidence shape]  [human-gated]
       ▼                ▼                   ▼                   ▼                ▼
 ┌──────────┐    ┌─────────────┐   ┌─────────────────┐   ┌──────────┐    ┌────────────┐
 │ proposed │───▶│ eval_passed │──▶│ tenant_approved │──▶│ serving  │───▶│ deprecated │
 └──────────┘    └─────────────┘   └─────────────────┘   └──────────┘    └────────────┘
       │                │                  │                  │                │
       └────────────────┴──────────────────┴──────────────────┴────────────────┘
                                         │
                       POST /{id}/retire │  (from any non-terminal state;
                                         ▼   human-gated when current state = serving)
                                   ┌──────────┐
                                   │ retired  │   ← terminal
                                   └──────────┘
```

**Forward transitions** travel `proposed → eval_passed → tenant_approved →
serving → deprecated` and are driven by `POST /models/{model_id}/promote` with
the **intended target state in the request body** (`{"target_state": "..."}`).
The handler does not infer "the next state" — it validates the exact
`(from_state, target_state)` pair the caller asked for. This makes the
transition explicit, auditable, and resistant to a stale-client race that
would otherwise advance a model further than the operator intended.

**`/retire`** is the only path to `retired`. It accepts any non-`retired`
state → `retired`. `retired` is terminal (no transition out).

**Transition preconditions:**

| Transition | Precondition checked by `models/storage.py` before the chain row is written |
|---|---|
| `proposed → eval_passed` | `models/trust.py` cosign verification of the artefact passes (per-tenant trust root) — **run before the DB transaction opens**; the locked precondition then re-checks the artefact refs + digest are unchanged (§4.1). Failure → `model_promote_signature_verification_failed`; mid-flight ref change → `model_promote_signature_refs_changed_during_promote`. Model stays `proposed`. |
| `eval_passed → tenant_approved` | `eval_results_ref` is present + shape-valid **and** `adversarial_pass_rate` is present + a valid ratio in `[0, 1]`. **Shape/presence only — see §9.** |
| `tenant_approved → serving` | none in the state machine; gated by `RequireHumanActor` at the route (§6). |
| `serving → deprecated` | none. |
| `* → retired` | none in the state machine; `RequireHumanActor` at the route when the from-state is `serving` (§6). |

### 2.2 `ModelRecord` and `kind`

`models/registry.py` defines the `ModelRecord` dataclass — the in-memory
projection of one `models` table row. Fields are the ADR-013 §"What AgentOS
owns" table (§3 below has the persisted column list). `kind` is a closed-enum
Literal: `foundation` / `fine_tune` / `adapter` / `embedding`. All four kinds
flow through the *same* lifecycle; `recipe_hash` and `training_data_fingerprint`
are nullable (a `foundation` model has no fine-tuning recipe).

### 2.3 `models/trust.py`

Mirrors `protocol/trust_gate.py`'s cosign discipline (no shell, list-form
argv, version-pinned binary, asyncio timeout, output ignored for parsing).
`verify_model_signature(*, signed_artifact_ref, sigstore_bundle_ref,
tenant_trust_root)` runs `cosign verify-blob` of the model artefact against the
per-tenant trust root and returns a pass/fail (raising a typed error on
tooling failure, fail-closed).

**It runs *outside* any DB transaction.** cosign is an external subprocess
with its own timeout; holding a `SELECT … FOR UPDATE` row lock across it would
pin the lock for the subprocess's lifetime. The `proposed → eval_passed` flow
is therefore: (1) plain-read the model row's artefact refs + `signature_digest`;
(2) run `verify_model_signature(...)` with **no lock held**; (3) open the
`append_with_precondition` transaction and, inside the locked precondition,
re-check those refs + digest are byte-identical to step (1) before writing —
a TOCTOU guard mirroring `packs/storage.py`'s `expected_manifest_digest` race
fix. A mid-flight change refuses with
`model_promote_signature_refs_changed_during_promote`.

**`signature_digest` is a digest, not a verdict — pinned byte contract.** It is
the **SHA-256 (hex) over the complete raw bytes of the Sigstore bundle file**
(the object at `sigstore_bundle_ref`). One well-defined file; an examiner
reproduces it with `sha256sum` on the bundle. The bundle transitively commits
to the model artefact (its signature is over the artefact digest), so digesting
the bundle is sufficient. *Plan task:* confirm this matches the bundle-digest
convention `protocol/trust_gate.py` / `protocol/supply_chain.py` /
`cli/verify.py` already use for pack Sigstore bundles — the registry must reuse
that convention, not fork it. The `models` table stores `signature_digest`,
`signed_artifact_ref`, and `sigstore_bundle_ref`; there is **no** stored
"is_verified" boolean — verification is enforced as the `proposed → eval_passed`
precondition, and a model past `proposed` has provably passed it via a
hash-chained `model.lifecycle.eval_passed` event.

## 3. Persistence & migrations

### 3.1 The `models` table

A new `models` table, defined as a SQLAlchemy Core `Table` whose types compile
correctly on **both** Postgres and Oracle (the established dual-dialect
discipline — e.g. `sa.TIMESTAMP(timezone=True)` so Oracle preserves offsets,
per `llm/ledger.py:127-132`).

| Column | Type | Notes |
|---|---|---|
| `model_id` | `String(128)` | ADR-013 stable identity (e.g. `cognic-tier1-bank-acme-v3`); unique registry key; the portal path-param |
| `tenant_id` | `String(128)` | Owning tenant; the tenant-isolation boundary (ADR-013: a model is registered under one tenant's scope) |
| `base_model` | `String(256)` | Foundation model the fine-tune started from |
| `version` | `String(64)` | Semver |
| `kind` | `String(32)` | `foundation` / `fine_tune` / `adapter` / `embedding` |
| `recipe_hash` | `String(64)`, nullable | SHA-256 of fine-tuning config + hyperparameters |
| `training_data_fingerprint` | `String(64)`, nullable | SHA-256 of the training-corpus manifest (no raw data) |
| `eval_results_ref` | `String(512)`, nullable | Pointer to the ADR-010 eval-pack run (validated for shape, not resolved — §9) |
| `adversarial_pass_rate` | `Float`, nullable | ADR-011 adversarial pass-rate (validated for ratio shape, not corpus-verified — §9) |
| `signature_digest` | `String(64)`, nullable | SHA-256 (hex) over the raw bytes of the Sigstore bundle file at `sigstore_bundle_ref` — see §2.3 |
| `signed_artifact_ref` | `String(512)`, nullable | Object-store pointer to the signed model artefact |
| `sigstore_bundle_ref` | `String(512)`, nullable | Object-store pointer to the cosign/Sigstore bundle |
| `serving_endpoint` | `String(512)`, nullable | vLLM/SGLang URL or the LiteLLM alias fronting it |
| `lifecycle_state` | `String(32)` | Current state — the state-cache column updated atomically per transition |
| `last_actor` | `String(256)` | Subject of the actor who drove the last transition |
| `created_at` | `TIMESTAMP(tz)` | |
| `updated_at` | `TIMESTAMP(tz)` | |

**No JSONB lifecycle-history column.** Lifecycle history is *not* a blob on
this table. It is reconstructed from the hash-chained `model.lifecycle.*`
events in `decision_history` — exactly as `packs/storage.py:load_lifecycle_history`
walks chain rows. `lifecycle_state` here is only the current-state cache.
(This corrects BUILD_PLAN §784 — see §8.)

## 4. Audit & ISO 42001 control mapping

### 4.1 `model.lifecycle.*` chain events

`models/storage.py` is the `DecisionHistoryStore.append_with_precondition`
consumer — the same primitive `packs/storage.py` uses. Every lifecycle change
is one atomic transaction (chain row + `models`-row write + chain-head update,
single commit). There are **two distinct paths** — register and transition —
and the plan must pin them separately:

**Genesis path — `POST /models`.** A new model has *no* `models` row to lock.
`register(...)` opens `append_with_precondition`; inside the precondition it
checks no row with this `model_id` already exists (`SELECT` → if found, refuse
`model_register_duplicate_id`), then **`INSERT`s the `models` row** (state
`proposed`), and the `model.lifecycle.proposed` chain row is appended in the
*same* transaction. Row creation and the genesis audit event commit together or
not at all — they are never split. (This diverges deliberately from
`packs.save_draft`, which writes no genesis chain row; ADR-013 §53 wants the
whole lifecycle, `proposed` included, in the tamper-evident trail.)

**Transition path — `/promote`, `/retire`.** The model row exists.
`transition(...)` opens `append_with_precondition`; inside the locked
precondition it does `SELECT … FOR UPDATE` on the `models` row (`ModelNotFound`
if absent), runs `validate_transition(...)` + the transition's *in-memory*
precondition (eval-evidence shape; the cosign re-check of artefact refs/digest
— the cosign *subprocess* itself ran earlier, outside the lock, per §2.3),
then `UPDATE models SET lifecycle_state, last_actor, updated_at` and appends the
`model.lifecycle.<target_state>` chain row.

**Where refusals are raised — two locations, distinct rollback shapes (the
plan's tests must assert the right shape per refusal):**

- The `proposed → eval_passed` **cosign-verification failure**
  (`model_promote_signature_verification_failed`) is raised **before the
  transaction opens** (§2.3 step 2). There is no transaction to roll back; the
  handler maps it straight to a 409, and **no chain row is ever attempted**.
- **Every other refusal** — the three state-machine reasons, the two
  eval-evidence shape reasons, the TOCTOU
  `model_promote_signature_refs_changed_during_promote`,
  `model_register_duplicate_id`, and `model_retire_already_retired` — is raised
  as `ModelLifecycleRefused(reason)` from **inside the precondition closure**,
  so the engine's rollback fires: no chain row, no `models`-row write, no
  orphan INSERT/UPDATE.

The chain-row `decision_type` namespace is **`model.lifecycle.<target_state>`**
— keyed by target state (`model.lifecycle.proposed`, `…eval_passed`,
`…tenant_approved`, `…serving`, `…deprecated`, `…retired`), mirroring
`packs/storage.py`'s `pack.lifecycle.<target_state>`. Refusals roll back before
any row is written, so there is no `.refused` event.

The chain-row `payload` carries `model_id` (the event subject), `from_state`,
`to_state`, `kind`, `last_actor`, `actor_type`, and the transition-relevant
evidence (e.g. `signature_digest` on the `eval_passed` row, `eval_results_ref`
+ `adversarial_pass_rate` on the `tenant_approved` row). **`payload["model_id"]`
is the only `decision_history` model-id linkage** — see §5 for why per-call
linkage lives elsewhere. Adding a payload key requires no schema or
canonical-form change: `core/canonical.py` already canonicalises + hashes the
whole `payload` dict.

### 4.2 ISO-42001 control tags

Per ADR-013's control-mapping table, every `model.lifecycle.*` chain row is
stamped (`iso_controls`) with the **5** ADR-006 controls the model registry
genuinely evidences, using the `controls.py` canonical `ISO42001.`-prefixed
form:

`ISO42001.A.6.2.6` (roles & responsibilities — lifecycle transitions are
RBAC-gated) · `ISO42001.A.7.4` (impact assessment — `eval_results_ref`) ·
`ISO42001.A.8.2` (data quality — `training_data_fingerprint`) ·
`ISO42001.A.8.5` (system development — `recipe_hash` / `base_model` in the
record) · `ISO42001.A.10.2` (stakeholder transparency — model-version lineage).

**`A.7.6` (AI system risk evaluation) is deliberately NOT tagged.** Shape-valid
`adversarial_pass_rate` storage is not a machine-verified AI-risk evaluation.
(This drops A.7.6 from BUILD_PLAN §805's 6-control list — see §8.)

### 4.4 Coupling: the flip is only valid if the emission is real

`controls.py`'s `audit_coverage(emitted)` treats an `implemented` control as
correctly covered **iff** its canonical ID appears in the `emitted` set built
by the Sprint-9 `test_control_mapping` suite from real emission-site source.
Therefore the four flips are valid **only** as a coherent unit with:

1. `models/storage.py` actually stamping the 5 canonical IDs on
   `model.lifecycle.*` chain rows (§4.2), and
2. the `test_control_mapping` suite **extended** to scan `models/` emission

The plan must land the emission, the `controls.py` edit, and the
`test_control_mapping` extension together. The test's "3 implemented / 5
deferred" assertion is updated to "7 implemented / 1 deferred".

### 4.5 Evidence-pack export — no new code

`model.lifecycle.*` events are ordinary `decision_history` rows, so the
Sprint-9 evidence-pack exporter (`compliance/iso42001/evidence_pack.py`) and

`test_model_audit_chain` exit-criterion ("full proposed → serving → retired
chain verifies via Merkle proof") reuses Sprint-9 Merkle — no new evidence-pack
code in this sprint.

## 5. LLM gateway linkage (provider honesty)

### 5.1 The locked linkage rule

BUILD_PLAN §9.5 and ADR-013 §48-53 conflate `decision_history` with the
operational `gateway_call_ledger`. Verification of `llm/ledger.py` resolved it
(the per-LLM-call `model_id` belongs in a column Sprint 3 *already created and
reserved*). The locked rule:

- **Per-call operational usage → `gateway_call_ledger.model_id`.** The Sprint-3
  ledger table already has the column (`ledger.py:148`,
  `GatewayCallRow.model_id` at `:108`, docstring `:28-29` — "always `None` on
  write; Sprint 9.5 (ADR-013 model registry) backfills"). `GET /models/{id}/usage`
  reads this table.
- **Governance lifecycle proof → `decision_history.payload["model_id"]`**, only
  on `model.lifecycle.*` transition events emitted by `models/storage.py`
  (§4.1) — the event subject.
- **No `decision_history` schema or canonical-form change.** No expression
  index on `decision_history.payload`.
- Migration `0004` adds the `models` table plus a plain `(model_id, ts)` index
  on `gateway_call_ledger` (§3.2).

This is lower blast radius and honest to the codebase: `decision_history.append`
is **not** "extended to record `model_id` per call" (BUILD_PLAN §806 is wrong —
see §8).

### 5.2 alias → model_id resolution: a static config map

The gateway must turn the dispatched LiteLLM alias into a `model_id`. Sprint
9.5 uses a **static config map**, not a live registry read — a registry-backed
resolution would make `llm/gateway.py` (a cloud-policy stop-rule on the hot
path) depend on model-registry reads + cache invalidation, which is out of
scope here.

- New setting `Settings.llm_model_id_map: dict[str, str]` in `core/config.py`
  (default `{}`; env-overridable via the existing pydantic-settings
  convention). Keys are LiteLLM aliases; values are registered `model_id`s.
- `llm/gateway.py` looks up the dispatched alias in `llm_model_id_map` and
  sets `GatewayCallRow.model_id` when a mapping exists.
- **If no mapping exists:** emit the ledger row with `model_id = None` (its
  Sprint-3 default) and log a clear, structured "unmapped alias" posture. The
  gateway **never invents a `model_id`** — an unmapped alias is recorded
  honestly as unmapped, not silently attributed to a guessed model.

A future sprint may replace the static map with registry-backed resolution;
the static map is a deliberate, swappable seam.

`llm/gateway.py` is a stop-rule (cloud-policy enforcer) and `llm/ledger.py` is
a critical-control (95/90 floor) — both touches are halt-before-commit (§7).

### 5.3 `/effective-routing` and `/usage`

- **`/api/v1/system/effective-routing`** (`portal/api/system_routes.py`, ADR-007
  provider honesty) is extended: each recent-call entry now surfaces
  `row.model_id` next to `row.litellm_alias`. Banks see exactly which
  registered model handled which case.
- **`GET /api/v1/models/{model_id}/usage?from&to`** aggregates
  `gateway_call_ledger` by `model_id` over the time window — call counts,
  index-served by the new `(model_id, ts)` index. This needs a small
  aggregate-read method on `GatewayCallLedger` (`llm/ledger.py` — a
  critical-control; halt-before-commit). The endpoint reads the ledger, **not**
  `decision_history` (corrects BUILD_PLAN §794 / ADR-013 §48 — see §8).

## 6. Portal API, RBAC & enforcement

### 6.1 The `portal/api/models/` route package

A new package mirroring `portal/api/packs/`:

```
src/cognic_agentos/portal/api/models/
  __init__.py
  router.py             — build_models_router(...) composition factory
  lifecycle_routes.py   — POST /models, POST /{id}/promote, POST /{id}/retire  [CRITICAL CONTROL]
```

### 6.3 RBAC — 8 `ModelRBACScope` values

A new closed-enum `ModelRBACScope` Literal in `portal/rbac/scopes.py` — the
same pattern as `PackRBACScope` (13 values) and `UIRBACScope` (8 values). The
8 scopes (BUILD_PLAN §796-802's 7, plus `model.promote.deprecated` — see §8):

1. `model.register`
2. `model.promote.eval_passed`
3. `model.promote.tenant_approved`
4. `model.promote.serving`
5. `model.promote.deprecated`
6. `model.retire`
7. `model.audit.read`
8. `model.usage.read`

Per-target-state promote scopes (one per forward transition) make the
separation-of-duties real: the eval reviewer, the security/compliance
reviewer, the deploying operator, and the operator who deprecates a serving
model are distinct grants. The closed-enum string is the wire-protocol
contract for every 403 `scope_not_held` denial body.

Three `portal/rbac/` files are touched — the whole directory is the RBAC
stop-rule boundary (§7), all three flagged for explicit stop-rule review,
exactly as Sprint 9 flagged its RBAC widening:

- `portal/rbac/scopes.py` — adds the `ModelRBACScope` Literal (and, mirroring
  the pack role-group frozensets, a `MODEL_LIFECYCLE_SCOPES` frozenset).
- `portal/rbac/actor.py` — `Actor.scopes` widens from
  `frozenset[PackRBACScope | UIRBACScope | ComplianceRBACScope]` (the verified
  current annotation at `actor.py:87`) to add `ModelRBACScope`. Additive — every
  pre-9.5 actor still constructs.
- `portal/rbac/enforcement.py` — `RequireScope`'s `scope` parameter is
  explicitly typed `PackRBACScope | UIRBACScope | ComplianceRBACScope`
  (`enforcement.py:240-242`); it widens to add `ModelRBACScope` so the model
  endpoints can call `RequireScope("model.register")` etc. The same one-line
  widening Sprint-7B.4 T5 (`UIRBACScope`) and Sprint-9 T5 (`ComplianceRBACScope`)
  already did; `enforcement.py` is a critical-control already on the gate.

### 6.4 Endpoint authorization

| Endpoint | Required scope | Extra gates |
|---|---|---|
| `POST /models` | `model.register` | — |
| `GET /models` | `model.audit.read` | tenant-scoped list (the storage `WHERE tenant_id` *is* the boundary — no `{model_id}` path-param) |
| `GET /models/{model_id}` | `model.audit.read` | `RequireTenantOwnership` |
| `POST /models/{model_id}/promote` | `model.promote.<target_state>` — resolved from the body | `RequireTenantOwnership`; `RequireHumanActor` when `target_state == "serving"` |
| `POST /models/{model_id}/retire` | `model.retire` | `RequireTenantOwnership`; `RequireHumanActor` when the model's current state is `serving` |
| `GET /models/{model_id}/audit` | `model.audit.read` | `RequireTenantOwnership` |
| `GET /models/{model_id}/usage` | `model.usage.read` | `RequireTenantOwnership` |

`GET /models` (list) and `GET /models/{id}` (detail) gate on `model.audit.read`
— the examiner read grant; there is no separate `model.read` scope.

**`/promote` and `/retire` do body-aware / state-aware authorization in the
handler.** `/promote` is a single endpoint whose required scope depends on the
body's `target_state`, and whose Human-only gate fires only for `→ serving`;
`/retire`'s Human-only gate fires only when the model is currently `serving`
(retiring a serving model is a production-routing change — CLAUDE.md "Model
promotions / rollbacks" is Human-only; the other six static-route endpoints
use plain per-route `RequireScope` dependencies). This non-trivial allow/deny
logic + the Human-only enforcement boundary is exactly why
`lifecycle_routes.py` goes on the critical-controls gate (§7) — the same
rationale that put `portal/api/packs/operator_routes.py` on the gate.

Cross-tenant isolation follows the established 404-not-403 doctrine: a model
owned by tenant A is invisible to tenant B (`RequireTenantOwnership` returns a
genuine 404 so a probe cannot enumerate model-IDs across tenants).
