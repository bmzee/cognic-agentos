## 1. Context

### Phase 3 closure

Phase 3 (sandbox + resumable sessions + compliance + model lifecycle + Vault credential leasing + scheduler) is the bank-grade-isolation + audit-evidence-export + procurement-defensible-model-registry layer. Sprint 8 (sandbox primitive) + Sprint 8.5 (resumable session API) + Sprint 9 (ISO 42001 control mapping) + Sprint 9.5 (model registry) + Sprint 10 (Vault credential leasing) + Sprint 10.1 (TTL hotfix) all merged to `main` by 2026-05-25.

**Sprint 10.5 closes Phase 3** by landing the runtime scheduler primitive (ADR-022) AND the deferred workload credential projection from Sprint 10.1 Finding #1.

### The scheduler gap (ADR-022 motivation)

AgentOS today has all the governance an examiner needs but cannot honestly call itself an **operating system** — an OS without a scheduler is not an OS. The pieces that touch scheduling-adjacent concerns (`core/sla.py`, `core/emergency/quotas.py`, `subagent/policy.py`, `protocol/mcp_host.py`, `protocol/a2a_endpoint.py`, `core/approval/engine.py`) operate post-hoc and cannot:
- Prevent a scheduled overnight eval-run from starving interactive customer-facing calls
- Refuse quota-exhausted work at queue boundary instead of after the LLM is already billed
- Enforce sub-agent budget narrowing at runtime (Sprint 11 dependency)
- Provide bounded backpressure when tools/models/sandboxes saturate

ADR-022 ships `core/scheduler/` as a first-class OS primitive between Vault credential leasing (Sprint 10) and Sub-agent (Sprint 11).

### The credential projection gap (Sprint 10.1 Finding #1)

Sprint 10 landed `VaultCredentialAdapter.mint_lease()` — Vault returns a `CredentialLease` carrying field-keyed credential material. **No wire path projects the credential material into the running container/pod.** The workload cannot read the credentials because there is no projection mechanism.

Sprint 10.1 hotfix deferred Finding #1 explicitly with the rationale: "needs an ADR-022-grounded shape decision (workload credential projection mechanism) that bleeds into the scheduler primitive's task lifecycle — not a hotfix concern."

Sprint 10.5c closes this gap via signed-manifest credential declarations + file-only projection through a pure-functional planner consumed by per-backend executors.

### The link between scheduler and credentials

The scheduler owns "when does this task run"; the natural extension is "and where do its credentials live for the duration of the run". Sprint 10.5 bundles both into one sprint with explicit sub-block separation: scheduler primitive in 10.5a/b; credential projection in 10.5c.

### 3.2 Layered relationship

- **Scheduler is the SUBMISSION layer:** decides WHEN a task runs; applies quota/policy/kill-switch admission
- **Sandbox is the RUNTIME layer:** owns container/pod lifecycle (create/destroy)
- **Projection is a sandbox-layer concern:** credentials are attached to the runtime container

Wiring: `scheduler.submit() → ... → SandboxBackend.create(*, requires_credentials)` extends Sprint 10's pattern. Direct `SandboxBackend.create()` callers continue to work (tests, ad-hoc SDK paths). Production convention: scheduler-routed; not enforced via kernel refusal.

## 4. 10.5a/b — Scheduler primitive

### 4.1 Wave-1 priority model

**Two classes only. FIFO within each class. No weighted fair-share.**

| Class | Use case | Admission-decision SLA | Default queue depth |
|---|---|---|---|
| `interactive` | Live UI session, customer-facing portal call, real-time agent chat | ≤200ms P95 | 32 per (tenant, class) |
| `background` | Eval runs, batch maintenance, scheduled jobs, regulator-erasure sweeper | ≤5s P95 | 256 per (tenant, class) |

Class declared at `scheduler.submit(task, *, class_)` time; immutable after admission.

### 4.2 Closed-enum vocabularies

**`SchedulerAdmissionOutcome`** — 7 values:
- 2 accepted: `accepted_immediate`, `accepted_queued`
- 5 refused: `refused_queue_full`, `refused_quota_exhausted`, `refused_policy_denied`, `refused_kill_switch_active`, `refused_pack_not_installed`

**`SchedulerRefusalReason`** — wire-equal to the 5-value refused subset of `SchedulerAdmissionOutcome` (not an independent vocabulary).

**`SchedulerTaskState`** — 7 values: `pending`, `running`, `completed`, `failed`, `cancelled`, `preempted`, `expired`.

**`task_cancelled.payload.reason`** — 4 values: `actor_cancelled`, `parent_run_cancelled`, `tenant_admin_cancelled`, `sandbox_boundary_killed`.

**`task_preempted.payload.reason`** — Wave-1 1 value: `quota_exhausted_in_flight`. (Wave-2 adds priority-inversion, operator-initiated.)

**`task_failed.payload.reason`** — scheduler closed-enum (e.g. `scheduler_task_failed_sandbox_create_refused`, `scheduler_task_failed_workload_runtime_error`); payload also carries `sandbox_refusal_reason` (upstream sandbox closed-enum value) + `sandbox_event_id` for cross-layer correlation.

### 4.3 Backpressure semantics

1. Queue capacity + concurrency cap headroom → `accepted_immediate` (run now)
2. Queue capacity + concurrency cap saturated → `accepted_queued` (FIFO wait; do NOT refuse)
3. Queue full for (tenant, class) → `refused_queue_full` with `retry_after_s` derived from oldest queued task's age + class SLA
4. Quota / policy / kill-switch / pack-state denies → refuse immediately with matching closed-enum

### 4.4 Task state machine (ADR-022 with amendments)

```
                  ┌────────────────┐
       submit()──▶│    pending     │
                  └─┬──────┬───────┘
                    │      │ (capacity opens + sandbox create + credential projection + workload start succeeds)
                    │      ▼
                    │  ┌────────────────┐  cancel() → ┌────────────┐
                    │  │    running     │────────────▶│ cancelled  │
                    │  └─┬────────────┬─┘             └────────────┘
        completion  │    │            │ (preempted: in-flight token budget exhaustion — Wave-1 only trigger)
                    │    ▼            ▼
                    │  ┌──────────┐  ┌────────────┐
                    │  │completed │  │ preempted  │
                    │  └──────────┘  └────────────┘
                    │
                    │  NEW (10.5a): create/projection/workload-start failure during pending
                    ├──▶┌──────────┐
                    │   │  failed  │
                    │   └──────────┘
                    │
                    │  NEW (10.5a): cancellation during pending/create
                    ├──▶┌──────────┐
                    │   │cancelled │
                    │   └──────────┘
                    │
                    │  (queue TTL exceeded)
                    └──▶┌──────────┐
                        │ expired  │
                        └──────────┘
```

**ADR-022 amendments triggered:**
- NEW transition `pending → failed` (sandbox create / credential projection / workload-start failure)
- NEW transition `pending → cancelled` (cancellation during create/projection)
- Semantic clarification: `running` means workload actually started (not "task dispatched"); `task_started` fires only when workload observed running

### 4.5 Concurrency caps

| Cap | Default (Settings) | Configured at |
|---|---|---|
| Per-tenant total concurrent tasks | 32 (interactive) + 64 (background) | `core/config.py` |
| Per-pack concurrent invocations within a tenant | 8 | Pack manifest (Wave-2 extension; Wave-1 uses Settings default) |
| Per-actor concurrent submissions | 4 | RBAC binding (Wave-2 extension; Wave-1 uses Settings default) |

A cap exceeded → enqueue (NOT refuse). FULL queue → `refused_queue_full`. Caps and queue capacity are orthogonal.

Numbers are Settings defaults shipped for a sane bootable kernel, NOT wire-protocol contract. Pinning regressions enforce the *bounded* invariant (caps cannot be unbounded / negative / unset), NOT that specific defaults survive forever.

### 4.6 Cooperative cancellation

`SchedulerEngine.cancel(task_id, *, actor)` is the public cancel seam. The running task receives `asyncio.CancelledError` at the next cooperative await point. The scheduler **does not** kill blocking-IO sections mid-syscall — no Python-thread force-kill, no SIGKILL injection, no thread-pool eviction. Pack authors must await at scheduler-aware points (LLM gateway, MCP tool calls, A2A inter-agent calls — all already await-points today).

**Wave-1 boundary-level kill (sandbox-holding tasks only):** if the task holds a `SandboxSession`, scheduler MAY request `sandbox.destroy(session_id)`. The Python coroutine still terminates via `asyncio.CancelledError` at next await; the boundary call releases the underlying compute without waiting for cooperative cooperation. **`session.suspend()` is NOT used for cancellation in Wave-1** — Sprint 10 T10 blocks suspend with active credentials; using `suspend()` in the cancellation language would confuse the contract.

`scheduler.task_cancelled` carries `payload.actor_subject` + `payload.reason` (closed-enum 4-value).

### 4.7 Quota integration (ADR-018 amendment)

Pre-10.5: `core/emergency/quotas.py` checks during gateway call — LLM already billed by the time quota check fires.

Post-10.5: `SchedulerEngine.submit()` consults `quotas.would_admit(tenant, pack, estimated_tokens)` BEFORE enqueueing. Refusal at queue boundary, not at the LLM. Quotas become a first-class scheduling input.

Estimate-vs-actual gap closed by existing post-execution reconciliation in `gateway.py`: scheduler reserves estimate at submit; gateway records actual at completion; next submit decision uses reconciled total. Over-reservation never wedges the queue — reservations released on every terminal state.

### 4.8 Policy integration — `scheduler.rego` (10.5b)

New default-ship bundle at `policies/_default/scheduler.rego`. Decision point: `data.cognic.scheduler.admit.allow`.

Inputs: `tenant_id`, `pack_id`, `actor_subject`, `class` (interactive/background), `pack_kind`, `pack_risk_tier`, `current_tenant_concurrent_count`, `requested_estimated_tokens`.

**Default `allow := false`** — admission requires explicit allow.

**Wave-1 default allow set:** `read_only` + `internal_write` ONLY. **Refuse all 6 high-risk tiers pre-13.5** (`customer_data_read`, `customer_data_write`, `payment_action`, `regulator_communication`, `cross_tenant`, `high_risk_custom`) — matches Sprint 8A `sandbox.rego` policy. `customer_data_read`-interactive-only refinement is post-13.5 work once `core/approval/engine.py` is wired.

Bank overlays may tighten; **loosening kernel defaults requires kernel + ADR amendment** (wire-protocol-public stop rule; joins `elicitation.rego` / `sampling.rego` / `supply_chain.rego` / `sandbox.rego` in AGENTS.md stop-rule list).

### 4.9 Audit event taxonomy

All hash-chain into `decision_history`. ISO 42001 `A.6.2.5` for all.

| Event | When emitted | Payload discriminator |
|---|---|---|
| `scheduler.admission_accepted` | Task admitted | `payload.outcome ∈ {immediate, queued}` |
| `scheduler.admission_refused` | Submit denied | `payload.reason` = SchedulerRefusalReason |
| `scheduler.task_started` | `pending → running` (workload observed running) | — |
| `scheduler.task_completed` | `running → completed` | — |
| `scheduler.task_failed` | `running → failed` OR `pending → failed` | `payload.reason` + `payload.sandbox_refusal_reason` + `payload.sandbox_event_id` |
| `scheduler.task_cancelled` | `running → cancelled` OR `pending → cancelled` | `payload.reason` = 4-value closed-enum |
| `scheduler.task_preempted` | `running → preempted` | `payload.reason` = 1-value Wave-1 |
| `scheduler.task_expired` | `pending → expired` | — |

8 events. The `task_id` field is chain-derived; lets Sprint 9 trace explorer walk the full lifecycle of a single agent invocation across queue → run → completion.

### 4.10 Sub-agent budget inheritance (Sprint 11 hook)

`SchedulerEngine.submit(task, *, parent_task_id=...)` accepts a parent task identifier. When a sub-agent invokes `tool_call`, the parent's remaining token budget is snapshotted at child-submit time and the child's quota reservation is narrowed accordingly. The child cannot exceed `min(child_pack_quota, parent_remaining_budget)`. Parent completion releases the child's residual budget back to the parent's pool.

This is the substantive ADR-005 amendment — sub-agent "budget narrowing" becomes a scheduler operation instead of a static config field.

## 5. 10.5c — Workload credential projection

### 5.1 Signed manifest grammar

**NEW `[credentials.<logical_name>]` block.** Required fields:

| Field | Type | Constraints |
|---|---|---|
| `vault_path` | str | Case-sensitive; chars `[A-Za-z0-9/_-]`; max 512; no leading/trailing `/`; no `//`; no `.`; contains at least one `/` |
| `expected_fields` | list[str] | 1-16 items; each matches `^[a-z][a-z0-9_]{0,31}$`; no `_*` prefix (reserved); no duplicates |
| `ttl_s` | int | Positive; per-tenant `Settings.max_credential_ttl_s` upper cap still enforced |
| `purpose_category` | Literal | Wave-1 vocabulary of 8: `application_database_read \| application_database_write \| audit_log_write \| external_api_authentication \| cryptographic_signing \| cryptographic_decryption \| service_account_token \| monitoring_endpoint_access` |
| `purpose_description` | str | Non-empty; max 256 chars |

Optional fields: **none in Wave-1**. Unknown fields refused with closed-enum `credentials_unknown_field`.

**Pack-wide NEW `[runtime].expected_workload_gid: int`** — required when any `[credentials]` block exists; range `1..65535` (0 = root, refused).

**Pack-wide caps:**
- Max 16 `[credentials.<logical_name>]` blocks per pack
- Max 16 `expected_fields` per credential
- Max 64 KiB per field value
- Worst-case theoretical projected payload per pack: 16 × 16 × 64 KiB = 16 MiB (realistic <100 KiB)

**Pack-wide constraints:**
- `logical_name` unique within pack
- `vault_path` unique within pack
- `purpose_category` + `purpose_description` may repeat across credentials
- Pack with `risk_tier ∈ 6 high-risk tiers` AND `[credentials]` refused pre-13.5 with `credentials_risk_tier_not_permitted_pre_13_5`

**Cross-validator pattern:** when `[risk_tier]` is missing AND `[credentials]` is declared, risk-tier validator emits its existing missing-risk-tier finding; credentials validator skips its risk-tier-restriction check (one-validator-owns-each-refusal invariant per `[[feedback_dual_path_doctrine]]`).

### 5.3 Workload-visible projection contract

**Canonical fixed path layout:** `/run/credentials/<logical_name>/<field>`

No `mount_path` override in Wave-1. Path-collision impossible by construction (logical_name uniqueness validated).

**NO `_metadata.json`.** Workload reads only field files. The `_*` prefix is reserved for AgentOS future use (Wave-2 may add `_rotation.json`, `_provenance.json`, etc.); drift guard asserts no `_*` files appear in any credential directory at projection time.

**File contents:**
- Byte-exact Vault value — no normalization, no `.strip()`, no trailing newline
- UTF-8 encoded if Vault returns Python `str`
- Permissions `0440` (group-readable)
- Read-only mount

**Resource grain:** per-credential. One K8s Secret per logical_name; one Docker staging subdir per logical_name. NOT aggregated per session.

### 5.4 Planner/Executor split (Round-7 Gap O wire-public-artifact-owner doctrine)

**NEW `sandbox/projection.py`** — pure-functional planner; ON the durable CC gate:

```python
def compute_projection_plan(
    *,
    lease: CredentialLease,
    manifest_decl: CredentialDecl,
) -> ProjectionPlan | ProjectionRefused:
    """Pure-functional projection planner. No I/O. No backend coupling.

    Per-credential shape — called once per credential by the lifecycle
    integration in §5.8. Validates field-set match, computes path layout,
    redacts content appropriately (no _metadata.json in Wave-1), returns
    a frozen plan of (relative_path, content_bytes, mode) tuples on
    success OR a frozen ProjectionRefused with structured diagnostics
    on failure. Each ProjectionRefused relates to exactly one credential.

    Ownership (UID/GID) is intentionally NOT a planner input — backend
    executors own chgrp (Docker) and fsGroup pinning (K8s) via their
    native API. Mode `0440` is the planner's output; ownership is
    applied by the executor at write time. Substrate preflight (§5.8
    step 2) validates the effective workload GID before this planner
    is ever called.
    """
```

**`ProjectionPlan`** frozen dataclass:
- Ordered tuple of `(relative_path: str, content_bytes: bytes, mode: int)` (per field of the credential)
- Audit metadata: `logical_name`, `lease_id`, `projected_field_count`, `vault_path`, `purpose_category`, `purpose_description`, `tenant_id`

**`ProjectionRefused`** frozen dataclass:
- `reason: ProjectionRefusalReason` Literal (wire-equal subset of `SandboxRefusalReason`)
- `logical_name: str`
- Optional diagnostic fields populated by reason:
  - `field_set_mismatch`: `expected_fields` + `actual_fields` (alphabetized) + `extras` + `missing`
  - `field_value_non_string`: `field_name` + `actual_type`
  - `field_value_empty_string`: `field_name` + `actual_length` (0 = exact empty; N>0 = whitespace-only)
  - `field_value_size_exceeded`: `field_name` + `actual_size` + `cap`

**Per-backend executors execute the plan via native API.**

#### Docker executor

- Verify `/dev/shm` is tmpfs via `/proc/mounts` parse; refuse with `sandbox_credential_staging_path_not_tmpfs` if not
- Stage path layout: `/dev/shm/cognic/<session-opaque>/<credential-opaque>/<field>` — **opaque at every level** (logical_name leaks NOWHERE on host filesystem)
- Per-session opaque: 16-hex random
- Per-credential opaque: 16-hex random per credential
- chgrp to resolved workload GID; chmod 0440 on files, 0750 on dirs
- Bind-mount read-only: `<credential-opaque-dir>` → `/run/credentials/<logical_name>/` (semantic only at the workload-facing mount target)
- One bind-mount per credential

#### K8s executor

- `create_namespaced_secret(type="Opaque", data={...base64...})`
- Name: `cognic-cred-<16-hex>` (opaque random; reveals nothing through `kubectl get secrets`)
- Encoding: `data` field (base64-encoded) — NOT `stringData` (byte-exact for binary credential values)
- **Required labels** (visible through K8s RBAC list scope): `cognic/component=credential-projection`, `cognic/session-id`, `cognic/logical-name` (kept consciously for label-selector cleanup/query despite leak tradeoff)
- **Annotations** (less visible than labels): `cognic/lease-id` (Vault lease IDs contain `/` chars; label-hostile), `cognic/tenant-id` (moved to annotation to avoid label-listing leak class)
- Pod spec mounts the Secret at `/run/credentials/<logical_name>/` with `defaultMode: 0440` + pod `fsGroup` pinning workload GID
- ownerReference as cleanup defense-in-depth; explicit AgentOS-driven `delete_namespaced_secret` remains authoritative
- Secret lifetime = sandbox session lifetime (NOT immediate-delete-after-mount; preserves kubelet projection semantics + Wave-2 rotation path)

### 5.5 Workload GID resolution

| Backend | Primary source | Manifest cross-check |
|---|---|---|
| Docker | Image USER directive (numeric `UID:GID` or `UID` only; name forms refused with `sandbox_credential_projection_image_user_directive_non_numeric`) | `[runtime].expected_workload_gid` must match resolved GID |
| K8s | Pod security context / `fsGroup` | `[runtime].expected_workload_gid` must match resolved group |

**Refusal cases:**
- Root workload: `expected_workload_gid` cannot be 0; resolved GID cannot be 0; → `sandbox_credential_projection_root_workload_refused`
- Unresolvable GID: → `sandbox_credential_projection_workload_gid_unknown`
- Manifest vs resolved mismatch: → `sandbox_credential_projection_image_gid_manifest_mismatch`

**Docker-only dev escape:** `COGNIC_DEV_ALLOW_PERMISSIVE_CREDENTIAL_PROJECTION=1` env flag downgrades to 0644 + structured warning log. Covers GID-unknown AND root-workload bypass in dev. **Never available in K8s; never available in production profile** (tested by explicit fail-loud regression).

| Event | Payload fields |
|---|---|
| `credentials_projected` | `logical_name`, `vault_path`, `tenant_id`, `lease_id`, `projected_field_count`, `purpose_category`, `purpose_description`, backend resource name (Docker opaque path / K8s Secret name), `sandbox_session_id` |
| `credentials_projection_failed` | `logical_name`, `vault_path`, `tenant_id`, `lease_id`, `reason`, `expected_fields` (alphabetized), `actual_fields` (alphabetized), `extras` (alphabetized), `missing` (alphabetized), optional diagnostic fields per reason, `revoke_outcome` ∈ {`revoked`, `revoke_failed`} |
| `credentials_projection_cleaned_up` | `logical_name`, `lease_id`, `cleanup_target` ∈ {`staging_dir`, `secret_resource`}, backend resource name, `sandbox_session_id` |
| `credentials_projection_cleanup_failed` | Same shape + `partial_state`, `error_class`, sanitized `error` |

### 5.8 Lifecycle integration with `SandboxBackend.create()`

1. Stage-2 admission completes (per Sprint 8A `admit_policy()` pipeline)
2. **Backend substrate preflight** — knowable BEFORE any lease is minted; fail-loud here avoids the mint-then-immediately-revoke waste:
   - Verify `/dev/shm` is tmpfs (Docker) — refuse with `sandbox_credential_staging_path_not_tmpfs`
   - Parse image USER directive (Docker) — refuse name forms with `sandbox_credential_projection_image_user_directive_non_numeric`
   - Resolve effective workload GID (Docker: from image USER; K8s: from pod security context / fsGroup) — refuse `0` with `sandbox_credential_projection_root_workload_refused`; refuse unresolvable with `sandbox_credential_projection_workload_gid_unknown`
   - Cross-check `[runtime].expected_workload_gid` against resolved GID — refuse mismatch with `sandbox_credential_projection_image_gid_manifest_mismatch`
   - Dev escape (`COGNIC_DEV_ALLOW_PERMISSIVE_CREDENTIAL_PROJECTION=1`; Docker only; never K8s; never production) downgrades root/GID-unknown refusals to 0644 + warning log
   - If preflight fails: no leases minted; raise `SandboxLifecycleRefused` directly to scheduler caller
3. **Mint-then-project loop** — for each credential in **manifest declaration order**:
   - Mint via Vault adapter → emit `lease_minted`
   - Compute projection plan (per-credential per §5.4 signature)
   - On `ProjectionPlan`: execute (write files / create K8s Secret) → **emit `credentials_projected` immediately**; continue to next credential
   - On `ProjectionRefused` for the current credential N:
     - **Best-effort revoke the just-minted lease for credential N** (revoke-only; no projection cleanup since this credential never projected) — emit `lease_revoked` or `lease_revoke_failed`
     - Emit `credentials_projection_failed` for credential N carrying `revoke_outcome` ∈ {`revoked`, `revoke_failed`}
     - Then unwind already-projected credentials 1..N-1 LIFO (step 5 below)
4. If all credentials projected successfully: start workload; transition `pending → running`; emit `scheduler.task_started`
5. **LIFO unwind of already-projected credentials** (triggered by step 3's failure path OR by post-projection failure such as workload-start failure):
   - For each already-projected credential, in reverse mint/projection order:
     - **Projection cleanup FIRST** (remove staging dir / delete K8s Secret) — minimizes active-risk-surface window during cleanup
     - Emit `credentials_projection_cleaned_up` or `credentials_projection_cleanup_failed`
     - **Then Vault revoke** (best-effort; runs even if projection cleanup fails)
     - Emit `lease_revoked` or `lease_revoke_failed`
6. Raise `SandboxLifecycleRefused(reason=<sandbox-side closed-enum>)` to scheduler caller
7. Scheduler caller catches refusal, transitions `pending → failed`, emits `scheduler.task_failed` with `payload.reason = "scheduler_task_failed_sandbox_create_refused"` + `payload.sandbox_refusal_reason = <upstream closed-enum>` + `payload.sandbox_event_id` for cross-layer correlation

**Three distinct cleanup paths**, each with deterministic event ordering:

| Failure stage | Failed credential lease | Already-projected credentials | Ordering |
|---|---|---|---|
| Substrate preflight (step 2) | N/A (no mint occurred) | N/A (no projection occurred) | Refuse directly; no events emitted beyond the refusal |
| Projection refusal for credential N (step 3) | Revoke-only (just minted; never projected) — emit `lease_revoked`/`lease_revoke_failed` + `credentials_projection_failed` with `revoke_outcome` | Unwind 1..N-1 LIFO via step 5 | Failed credential cleaned up first, then already-projected stack |
| Post-projection failure (workload start, etc.) | N/A | Unwind ALL projected LIFO via step 5 | Already-projected stack only |

### 5.9 Wiring point

Credential projection lives in `SandboxBackend.create()` as natural extension of Sprint 10's `requires_credentials` parameter. Scheduler.submit() routes through this seam; direct callers (tests, ad-hoc SDK) also trigger projection. No scheduler-only restriction in Wave-1; production convention is scheduler-routed but not enforced by kernel refusal.

**Both Z3 and Z4 follow the Sprint 10.1 Finding #3 fail-loud doctrine, with the env gate (not extras availability) controlling skip vs run:**
- **Opt-out** (`COGNIC_RUN_*_CREDENTIAL_PROJECTION_INTEGRATION` unset): module-level `pytest.skip(<message>, allow_module_level=True)` fires BEFORE any optional imports — skip reason is "env gate unset", independent of whether extras are installed
- **Opt-in** (env gate set): plain `import hvac` / `import aiodocker` / `import kubernetes_asyncio` etc. (NOT `pytest.importorskip`) — missing extras → ImportError → pytest collection error (fail-loud)
- Why this matters: `pytest.importorskip` skips when extras are missing regardless of intent — that's wrong for opt-out (extras may be installed; env still unset = should skip) AND wrong for opt-in (missing extras must fail loud, not skip silently)
- Subprocess contract regression at `tests/unit/test_z3_z4_import_fail_loud_contract.py` pins both branches via `sys.path` shim that makes optional extras raise ImportError; asserts opt-out path emits `CASUAL_SKIP_OK` regardless and opt-in path emits `FAIL_LOUD_OK`
