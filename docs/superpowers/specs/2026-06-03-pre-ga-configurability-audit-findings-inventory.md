## Structural findings

1. **No general per-tenant configuration-overlay mechanism (§9 — CONFIRMED).** ≈10 independent `tenant-overlay-candidate` findings across config / memory / DLP / scheduler / sandbox / emergency all need the *same* missing seam. Today the only per-tenant surfaces are Vault path templates + `plugin_allowlist.json`. **The single highest-leverage structural gap; it gates the entire Wave-2.** Per charter §9 D1, designing the overlay mechanism is its own downstream spec — now strongly justified by the candidate set.
2. **Developer-personal defaults in shipped artifacts — and a broken override path.** Canonical sandbox images point at `ghcr.io/bmzee/...` in config defaults *and* as hardcoded backend fallbacks (`docker_sibling.py:202` / `kubernetes_pod.py:260`). Worse, `backend_factory.py` wires the Settings override into the **catalog only, not the backend launch selector** (CFG-sandbox-launch), so the documented re-home produces a catalog/launch mismatch. A GA pass must neutralize all developer-account references **and** thread the image Settings into the backend constructor.
3. **Secret-material has no uniform Vault-resolution path.** 5 secret fields are plain-env; reserved `*_vault_path` fields exist (embedding/dynatrace) but are unconsumed; `vault_token`/`litellm`/`langfuse` have no vault-path field at all. A single "resolve-from-Vault-in-prod + fail-loud-on-plain-env" pattern closes all five.
4. **Harness-injection-deferred state is load-bearing.** Kill-switch (emergency) + scheduler engine are DI-only, not yet app-wired; the SLA / cache-ttl / quota seams only "go live" once **workstream #2 (Harness Injection)** lands — directly validating the chosen sequence (#1 audit → #2 harness → #3 recall).
5. **`core/emergency/quotas.py` is unbuilt** (ADR-018 quota engine → Sprint 13.5); per-tenant quotas are a Wave-2 candidate that also depends on that build.
6. **`.env.example` documents only ~Sprint-7A;** ~45 later Settings fields are undocumented — reconcile (and ideally drift-gate) before GA.
