# Sprint 13.5b1 — Portal Approval API + store read methods (ADR-014) — Design

**Status:** DRAFT (acceptance Human-only per ADR-014).
**Date:** 2026-06-11.
**Predecessor:** Sprint 13.5a (`core/approval/` engine core, merged `cf91887`, CC 128).
**Successor:** Sprint 13.5b2 (MCP-host seam cutover — recorded in §11, NOT in this spec).

---

## 1. Goal & scope

Expose the Sprint-13.5a `ApprovalEngine` to human reviewers over HTTP: a thin
portal router (`GET` queue + `GET` detail + `POST` grant / grant-second / deny)
plus the two read methods the API needs on `ApprovalRequestStore`. The engine
remains the single authoritative enforcement boundary; the portal is a
delegate-to-engine surface that adds only the HTTP shape, the `Actor →
ApprovalActor` projection, the human-actor gate (defence-in-depth), and the
refusal→status mapping.

**In scope (13.5b1):**
- `ApprovalRequestStore.list_pending` + `load_detail` (tenant-scoped reads).
- `portal/api/approvals/routes.py` — `build_approval_routes(*, store, engine)` factory + 5 endpoints.
- `portal/api/approvals/dto.py` — frozen `extra="forbid"` request/response DTOs.
- A narrow read-only engine helper `grant_scope_for_risk_tier(risk_tier) -> str | None`.
- Composition-root wiring: construct the store + engine in `build_runtime`; thread into `create_app`; mount gated on both present.

**Explicitly OUT of scope (deferred):**
- **No public `POST /approvals` create endpoint** — the first producer is the
  MCP host (in-process, 13.5b2). A public mint surface before a real OS caller
  needs it is rejected (YAGNI + attack-surface).
- **No MCP-host changes, no `ToolInvocationRefusalReason` enum changes** — all
  13.5b2.
- No quota / kill-switch (ADR-018 → 13.6).

## 2. Architecture

```
HTTP reviewer ─▶ portal/api/approvals/routes.py (build_approval_routes)
                   │  bind Actor (RBAC) ─▶ map Actor → ApprovalActor
                   │  GET: store.list_pending / store.load_detail  (tenant-scoped)
                   ▼
                core/approval/engine.py  ◀── AUTHORITATIVE
                   grant / grant_second / deny  (human-only · tenant-binding ·
                   scope-per-tier · 4-eyes · expiry · reason-policy)
                   │
                   ▼
                core/approval/storage.py (DecisionHistoryStore, value-free chain)
```

`portal/api/approvals/*` imports `core/approval` (the allowed direction). The
13.5a architecture fence already forbids the reverse (`core/approval` imports no
`portal`/`cli`), pinned by `tests/unit/architecture/test_approval_fences.py`.

## 3. Store read methods (`core/approval/storage.py` — CC, on the gate)

Two new pure-read methods. Both are tenant-scoped — the `WHERE tenant_id`
clause IS the cross-tenant boundary (mirrors `packs.storage.list_for_tenant`
and `models.storage.list_for_tenant`). Neither emits a chain row.

### 3.1 `list_pending`

```python
async def list_pending(
    self, tenant_id: str, *, limit: int = 50, cursor: uuid.UUID | None = None
) -> list[ApprovalRequestSummary]:
    """The reviewer queue: requests in an ACTIONABLE state
    (``pending`` + ``awaiting_second``) for ``tenant_id``, newest-first,
    bounded by ``limit``, paginated by ``cursor`` (request_id keyset).
    The ``ix_approval_requests_tenant_state`` composite index (storage.py:102)
    backs the WHERE clause."""
```

- `tenant_id` is a **required positional** arg (NOT optional) — making it
  optional would re-open the cross-tenant-leak class (the
  `packs.list_for_tenant` doctrine). The only `state` values returned are
  `pending` + `awaiting_second` (terminal states are not actionable; a separate
  history/audit surface is out of scope here).
- Returns a lighter `ApprovalRequestSummary` projection (no `redacted_context`
  body — that is the detail view): `request_id`, `flow`, `risk_tier`,
  `tool_identity`, `originator_subject`, `state`, `first_approver`,
  `created_at`, `expires_at`.
- A module-private `_build_list_pending_stmt(...)` is the SOLE query-construction
  path; a SQL-shape regression imports the SAME builder and asserts the compiled
  SQL contains `approval_requests.tenant_id = ` (always) + the state filter —
  the shared-builder pattern that pins the production path uses the indexed
  columns (mirrors `packs.storage._build_list_for_tenant_stmt`).

### 3.2 `load_detail`

```python
async def load_detail(
    self, *, request_id: uuid.UUID, tenant_id: str
) -> ApprovalRequestDetail | None:
    """The reviewer detail projection. Cross-tenant / unknown → ``None``
    (the route maps None → 404 per the invisibility doctrine). Richer than
    the engine-facing ``load`` (storage.py:366): adds ``data_classes`` +
    ``redacted_context`` + ``created_at`` from the persisted row so the
    reviewer panel can render the full redacted context. ``check()`` returns
    only the lifecycle ``ApprovalCheckResult`` — insufficient for the panel."""
```

- New frozen `ApprovalRequestDetail` projection: every `ApprovalRequestRow`
  field (storage.py:107-125) **plus** `data_classes: tuple[str, ...]`,
  `redacted_context: str`, `created_at: datetime`. `args_digest` /
  `envelope_digest` surface as hex strings at the DTO layer (never raw bytes on
  the wire; never raw args anywhere).
- The existing engine-facing `load` (storage.py:366) is UNCHANGED — its minimal
  shape is what `_lazy_expire_and_project` reads; `load_detail` is the separate
  reviewer projection (keeps the engine's read path lean).

## 4. Endpoints (`portal/api/approvals/routes.py`)

`build_approval_routes(*, store: ApprovalRequestStore, engine: ApprovalEngine)
-> APIRouter`, mounted under `/api/v1/approvals`. Module invariants (every one
a known loss-case in prior sprints):

- **OMIT `from __future__ import annotations`** — PEP 563 stringified annotations
  break FastAPI's `get_type_hints` on `Annotated[..., Depends(<closure-local>)]`
  (`feedback_pep563_breaks_closure_local_depends`; same as `operator_routes.py`).
- Closure-local shared dependency instances (one `RequireScope("tool.approve.observe")`,
  one `RequireHumanActor()`); FastAPI's per-request sub-dep cache dedupes the
  actor bind.
- Per-verb request-id minters (`_PACK_*`-style prefixes), module-foot
  `assert len(prefix) + 32 <= 64` against the `decision_history.request_id`
  String(64) cap.
- Mutually-exclusive structured-log refusal contract (exactly one
  `portal.approvals.<verb>` OR `portal.approvals.<verb>_refused` per request;
  sibling-gate RBAC refusals emit zero route-level logs).

| Endpoint | Method | Dep chain | Body | Delegates to | 200 body |
|---|---|---|---|---|---|
| `/` | GET | `RequireScope("tool.approve.observe")` + bare-list tenant preflight | — | `store.list_pending(actor.tenant_id, …)` | `list[ApprovalSummaryResponse]` |
| `/{request_id}` | GET | `RequireScope("tool.approve.observe")` | — | `store.load_detail(request_id, actor.tenant_id)` → None ⇒ 404 | `ApprovalDetailResponse` |
| `/{request_id}/grant` | POST | bind actor + `RequireHumanActor()` | `GrantRequest{reason: str \| None}` | `engine.grant(request_id, actor.tenant_id, approver, reason)` | `ApprovalActionResponse{state}` |
| `/{request_id}/grant-second` | POST | bind actor + `RequireHumanActor()` | `GrantRequest{reason: str \| None}` | `engine.grant_second(...)` | `ApprovalActionResponse{state}` |
| `/{request_id}/deny` | POST | bind actor + `RequireHumanActor()` | `DenyRequest{reason: str}` | `engine.deny(request_id, actor.tenant_id, approver, reason)` | `ApprovalActionResponse{state}` |

- **Bare-list preflight:** `GET /` has no `{request_id}` path-param, so it cannot
  use a tenant-ownership dependency. The handler runs an explicit preflight: if
  `actor.tenant_id` is falsy → 500 with the closed reason (mirrors
  `inspection_routes.py` `actor_tenant_id_missing`). Under live Pydantic
  validation `Actor.tenant_id: str` the only reachable falsy value is `""`.
- **Tenant isolation via the engine, not a new dependency:** the action handlers
  pass `actor.tenant_id` to the engine, which loads by `(request_id, tenant_id)`
  and raises `ApprovalRequestNotFound` on cross-tenant / unknown → 404. The GET
  detail handler gets the same invisibility from `load_detail` returning `None`.
  No new `RequireTenantOwnership` variant is introduced. Cross-tenant and
  genuine-unknown both render 404 — a probe cannot distinguish them
  (`feedback_wire_body_collapse_cross_tenant_invisibility`).

## 5. `Actor → ApprovalActor` + the authority boundary (USER PIN)

Each action handler maps the bound portal `Actor` to the core-owned
`ApprovalActor` (subject / tenant_id / scopes / actor_type) at the boundary
(`ApprovalActor` is core-owned by design — `_types.py:107-116`; the portal must
not pass its `Actor` into `core`).

**The engine is authoritative.** The portal performs an optional pre-engine
read ONLY for response/preflight clarity (a clean wire 403/404 with a readable
reason). The engine independently re-enforces — and WINS on any disagreement —
for **tenant-binding, RBAC scope-per-tier, human-only, state, expiry, 4-eyes
distinctness, and reason-policy**. Concretely:

- **Human-only:** the portal's `RequireHumanActor()` dep is **defence-in-depth**.
  The engine's own `approver_not_human` guard (`engine.py` grant pipeline) is the
  authoritative gate — a service actor is refused even if the portal dep were
  ever misconfigured.
- **Scope-per-tier (defence-in-depth, optional pre-check):** the action handlers
  MAY compute the required scope via the §6 helper and surface an early 403
  `scope_not_held`, but the engine's `approver_scope_not_held`
  (`engine.py:252`, reading `_TIER_GRANT_SCOPE`) is authoritative. The handler
  pre-check is a UX/wire affordance, NOT a second decision. If the handler does
  the pre-check, it MUST still call the engine and surface the engine's verdict —
  the pre-check can only produce a 403 that the engine would also produce.
- **4-eyes / expiry / state / reason:** the portal does NOT replicate these. They
  surface only as mapped engine refusals (§5 table below in §5.1). There is no
  portal-side expiry clock, distinctness check, or state machine.

This keeps scope-per-tier (and every other rule) single-sourced in the CC engine
while still giving the wire a clean, readable refusal.

### 5.1 Refusal → HTTP status mapping

| Source | Reason | Status | Body |
|---|---|---|---|
| `ApprovalRequestNotFound` | — | 404 | `{"reason": "approval_request_not_found"}` |
| `ApprovalTransitionRefused` | `approver_not_human` | 403 | `{"reason": <reason>}` |
| | `approver_scope_not_held` | 403 | |
| | `grant_reason_required` | 400 | |
| | `approval_expired` | 409 | |
| | `approval_already_finalized` | 409 | |
| | `four_eyes_approver_not_distinct` | 409 | |
| | `grant_second_requires_awaiting_second` | 409 | |
| | `deny_requires_non_terminal` | 409 | |
| | `auto_tier_no_approval_required` | 409 | (defensive — auto tiers never create a request) |
| | `approval_binding_mismatch` | 409 | (not reachable from these endpoints; mapped for completeness) |

The 10-value `ApprovalTransitionRefusedReason` (`_types.py:35-46`) is consumed
verbatim as the 403/400/409 body `reason`; drift is a wire-protocol change. A
route-level mapping test pins every enum value to a status (so adding an 11th
reason without a mapping fails the test).

## 6. Engine helper — `grant_scope_for_risk_tier` (USER PIN: narrow, read-only)

```python
def grant_scope_for_risk_tier(risk_tier: str) -> str | None:
    """Return the single RBAC scope that may grant ``risk_tier`` (the
    ``tool.approve.*`` value), or ``None`` for tiers that need no grant scope
    (``read_only`` / ``internal_write`` — auto-run) and for unknown tiers.
    Read-only projection of the engine's ``_TIER_GRANT_SCOPE`` single source
    of truth (engine.py:67-74). NOT a policy decision — it makes NO approval
    decision, performs NO I/O, and is NOT a second approval engine; it is a
    pure lookup the route uses for body-aware defence-in-depth."""
    return _TIER_GRANT_SCOPE.get(risk_tier)
```

- Exposed as a module-level function in `core/approval/engine.py` (or a thin
  `_types`-adjacent helper) reading the existing `_TIER_GRANT_SCOPE` — the
  mapping stays single-sourced; the helper only surfaces it. `None` for the 2
  auto tiers + any unknown tier (the dict has 6 entries; `.get` yields `None`
  otherwise).
- Pinned by a drift test asserting the helper's non-None domain == the 6
  `_TIER_GRANT_SCOPE` keys and the returned values == the 6 `tool.approve.*`
  grant scopes (the observe scope is never returned).

## 7. Composition-root wiring

Mirror the ADR-023 config-overlay precedent exactly (`harness/runtime.py` +
`create_app`):

- **`harness/runtime.py` `build_runtime`:** construct `ApprovalRequestStore(decision_history_store)`,
  an `ApprovalPolicy` (an `OPAEngine` pointed at `policies/_default/tools.rego`,
  mirroring the memory-branch `OPAEngine.create` calls already in
  `build_runtime`), and `ApprovalEngine(policy=…, store=…, settings=settings,
  clock=<UTC clock>)`. Add `approval_store` + `approval_engine` fields to the
  frozen `Runtime` dataclass (runtime.py:38-52).
- **`portal/api/app.py` `create_app`:** add two optional None-default kwargs
  `approval_store: ApprovalRequestStore | None = None` + `approval_engine:
  ApprovalEngine | None = None` (the established opt-in pattern, after
  `config_overlay_store`/`config_overlay_resolver`). When BOTH are non-None,
  mount `build_approval_routes(store=…, engine=…)` and set
  `app.state.approval_router_mounted = True`. Attach eagerly before the lifespan
  (the config-overlay attach precedent) so non-lifespan test transports see them.
- The lifespan threads `runtime.approval_store` + `runtime.approval_engine` into
  `create_app`. A single shared engine instance is constructed here — 13.5b2
  injects the SAME engine into `MCPHost`.

## 8. DTOs (`portal/api/approvals/dto.py`)

Frozen, `extra="forbid"`, `from_attributes=True` on response models (the
`packs/dto.py` `PackBaseModel` convention). All digests rendered as **hex
strings**, never bytes:

- `ApprovalSummaryResponse` — the list item (no `redacted_context`).
- `ApprovalDetailResponse` — full reviewer projection (incl. `data_classes`,
  `redacted_context`, `args_digest` hex, `first_approver`, `expires_at`).
- `GrantRequest{reason: str | None = None}` / `DenyRequest{reason: str}`.
- `ApprovalActionResponse{request_id, state}`.

## 11. Carried-forward 13.5b2 design (recorded; NOT built here)

The MCP-host seam cutover, locked at the 13.5b reconciliation — its own spec:

1. **Thread `originator_subject` explicitly into `MCPHost.call_tool`** (a CC
   signature change; honest over ambient inference).
2. **Carry `data_classes` onto `MCPServerEntry`** at registration/discovery time
   (the registry already validates data-classes) — no seam-time manifest lookup;
   keeps `call_tool` deterministic.
3. **Compute `args_digest = sha256(canonical_bytes(arguments))` at the seam** —
   the binding key 13.5a's replay gate was built for.
4. **Pending + re-call flow:** first `call_tool` → `engine.create_request` →
   refuse `tool_approval_pending` (carrying `request_id`); after a portal grant,
   the caller re-calls `call_tool(..., approval_request_id=…)`; the host
   recomputes the digest and calls `engine.verify_grant_for_action` — only a
   matching binding dispatches; changed args → `approval_binding_mismatch`.
5. **Extend `ToolInvocationRefusalReason`** (currently the single
   `tool_approval_engine_not_available`, `mcp_host.py:552`) with
   `tool_approval_pending` / `tool_approval_denied` / `tool_approval_expired` /
   `tool_approval_binding_mismatch` / `tool_approval_request_not_found`. Keep
   `tool_approval_engine_not_available` as the engine-missing fallback. This is
   wire-protocol-public + drift-pinned.
6. Inject the shared `ApprovalEngine` (built in `build_runtime` per §7) into
   `MCPHost.__init__`.

Then **13.5c** does the remaining seams + the coordinated `sandbox.rego` /
`scheduler.rego` CONVERT amendment; **13.6** does ADR-018 quota/kill-switch.
