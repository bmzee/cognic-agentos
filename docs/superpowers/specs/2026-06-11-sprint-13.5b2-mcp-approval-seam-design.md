# Sprint 13.5b2 — MCP-host approval seam cutover (ADR-014) — Design

**Date:** 2026-06-11
**Predecessor:** Sprint 13.5b1 (portal approval API — merged `main @ db33202`, PR #60).
This spec executes the carried-forward design recorded at 13.5b1 spec §11, with
the four reconciliation forks resolved (F1-F4 below).
**Successor:** Sprint 13.5c (remaining seams + coordinated `sandbox.rego` /
`scheduler.rego` CONVERT amendment); Sprint 13.6 (ADR-018 quota / kill-switch).

## 1. Scope

Make `MCPHost.call_tool` an **approval-engine consumer**: high-risk tool
invocations create an approval request, refuse `tool_approval_pending`, and
dispatch only after a portal grant verified through the 13.5a replay-binding
gate. The Sprint-5 T10 transitional gate becomes the **engine-absent fallback**,
byte-for-byte.

**In scope:**
- `protocol/mcp_host.py` (CC + stop-rule): optional `approval_engine` ctor dep;
  engine-wired `call_tool` path; `ToolInvocationRefusalReason` 1 → 6 values;
  `MCPServerEntry.data_classes`; the canonical `tool_identity` digest helper.
- ADR-014 amendment recording the wire contract.
- Tests: engine-absent byte-compat pins; engine-wired behavioural suite;
  migrated-DB e2e (create → portal grant → re-call dispatch).

**NOT in scope (explicit):**
- **NO production MCPHost construction path** (F1 = A). `create_prod_app` keeps
  its availability-log-only branch; the registry-walk →
  `MCPServerEntry` mapping → `app.state.mcp_host` wiring is a separate
  composition-root sprint. 13.5b2 makes `MCPHost` approval-capable and
  test-proven; it does NOT claim a production host construction path exists.
- NO sandbox / scheduler / memory / credentials seam changes (→ 13.5c).
- NO `sandbox.rego` / `scheduler.rego` CONVERT amendment (→ 13.5c).
- NO ADR-018 quota / kill-switch (→ 13.6).
- NO public `POST /approvals` create endpoint (engine-internal create only —
  standing b-series rule).
- NO `tools.rego` changes (landed at 13.5a; bank overlays may tighten).
- NO portal route changes (13.5b1 surface is complete for this consumer).

## 2. Constructor seam (F1 = A)

`MCPHost.__init__` gains one optional keyword-only dep after `settings`:

```python
approval_engine: ApprovalEngine | None = None,
```

- `None` (default) → **engine-absent fallback**: today's T10 transitional gate
  byte-for-byte — static `_ADR_014_LOW_RISK_TIERS = {read_only, internal_write}`
  allow-list; everything else refuses `tool_approval_engine_not_available` with
  the same audit/decision payloads incl. `sprint_13_5_followup=True`. The
  existing `test_mcp_high_risk_tier_refused.py` suite continues to pin this path
  unchanged (fixture hosts pass no engine).
- Wired → the engine-authoritative path (§3). The shared instance is
  `runtime.approval_engine` (built unconditionally by `build_runtime` since
  13.5b1); the future composition-root sprint threads it at host construction.

Import note: `mcp_host.py` imports `ApprovalEngine` + the typed exceptions from
`core.approval` at module top (the module already imports `core` stores; no
architectural-arrow violation — protocol → core is the allowed direction).

## 3. Engine-wired `call_tool` path (F2 = B: engine-authoritative)

`call_tool` gains two keyword-only params (additive; CC signature change):

```python
originator_subject: str,                      # required WHEN engine is wired (§3.4)
approval_request_id: uuid.UUID | None = None, # the re-call correlator
```

When `approval_engine is not None`, the static tier set is NOT consulted.
Sequencing is unchanged in spirit: the approval gate fires BEFORE token-acquire
and BEFORE session-open (same security + audit-cleanliness rationale as T10).

### 3.1 First call (no `approval_request_id`)

1. Compute `args_digest = hashlib.sha256(canonical_bytes(dict(arguments))).digest()`.
2. Build the value-free `ApprovalEnvelope` (§3.3) and call
   `engine.create_request(envelope=...)`.
3. `ApprovalTransitionRefused("auto_tier_no_approval_required")` → the tier
   classified `auto_run` via `tools.rego` → **proceed to dispatch** (the
   existing step-2+ orchestration, unchanged). ONE OPA eval per auto-run call;
   no static set on the wired path — a bank overlay that tightens `tools.rego`
   (e.g. `internal_write → require_single_approval`) is honoured.
4. Otherwise a `pending` request now exists → refuse
   `tool_approval_pending`, carrying `approval_request_id=str(request.request_id)`
   in the `MCPToolInvocationRefused` payload AND in both evidence rows, so the
   caller can correlate the portal grant and re-call.

### 3.2 Re-call (`approval_request_id` supplied)

1. Recompute `args_digest` + `tool_identity` (same helpers — §3.3, F4).
2. `engine.verify_grant_for_action(request_id=approval_request_id,
   tenant_id=tenant_id, expected_args_digest=..., expected_tool_identity=...)`.
3. Outcome mapping (closed-enum; §4):
   - result.state == `granted` → **dispatch** (steps 2+ unchanged).
   - `pending` / `awaiting_second` → refuse `tool_approval_pending` (still
     carrying the request_id).
   - `denied` → refuse `tool_approval_denied`.
   - `expired` (incl. lazy-expired by this very call) → refuse
     `tool_approval_expired`.
   - `ApprovalTransitionRefused("approval_binding_mismatch")` (changed args or
     tool identity) → refuse `tool_approval_binding_mismatch`.
   - `ApprovalRequestNotFound` (unknown OR cross-tenant — the engine load is
     tenant-scoped, so cross-tenant is indistinguishable from unknown by
     construction) → refuse `tool_approval_request_not_found`.

A grant authorises exactly ONE action shape (ADR-014 §replay-binding); a second
re-call after a successful dispatch is NOT consumed/single-use in Wave-1 — the
grant remains `granted` until expiry. (Single-use consumption is a deliberate
non-goal here; it requires an engine-side `consume` transition and is recorded
as a 13.5c/14 follow-up, NOT silently approximated at the seam.)

### 3.3 Envelope sourcing (F3, F4)

| Envelope field | Source at the seam |
|---|---|
| `risk_tier` | `_normalize_risk_tier_for_gate(entry.risk_tier)` (bounded; malformed manifests still fail closed — an out-of-vocab tier refuses at engine envelope validation `risk_tier_unknown` → routed per §3.5) |
| `tool_identity` | **F4 collision-proof canonical digest** (§3.3.1) |
| `originator_subject` | the new `call_tool` param (§3.4) |
| `tenant_id` | the existing `call_tool` param |
| `data_classes` | `entry.data_classes` (NEW `MCPServerEntry` field — §5) |
| `args_digest` | `sha256(canonical_bytes(dict(arguments)))` — 32 bytes, satisfies `_DIGEST_LEN` |
| `redacted_context` | bounded human-readable descriptor: `f"mcp_tool server_id={server_id} tool_name={tool_name}"`, truncated to ≤ `APPROVAL_REDACTED_CONTEXT_MAX_LEN` (4096) — values pass through `_sanitize_string_for_operator_surface` first (same R4 P2 rationale as the exception message) |
| `required_refs` | `{}` for all tiers EXCEPT `regulator_communication` → `{"audit_record_ref": request_id}` (**F3**: the invocation `request_id` IS the correlator every `call_tool` audit row is keyed by) |

#### 3.3.1 Canonical tool identity (F4)

One module-level pure helper, used by BOTH the create path and the verify path
(same-function-or-bust — drift between the two would make every grant
unverifiable):

```python
def _canonical_tool_identity(*, server_id: str, tool_name: str) -> str:
    return "mcp:" + hashlib.sha256(
        canonical_bytes({"server_id": server_id, "tool_name": tool_name})
    ).hexdigest()
```

- Collision-proof regardless of separator characters in either field (raw
  `f"{server_id}:{tool_name}"` rejected — neither field is proven colon-free).
- 68 chars — fits the `approval_requests.tool_identity` `String(256)` column.
- The human-readable pair lives in `redacted_context` (above), so reviewers in
  the 13.5b1 portal detail panel can still see WHICH tool, while the replay
  gate compares the digest form.
- Pinned by a vector test (fixed inputs → fixed digest) + a
  same-function regression (create + verify both route through the helper —
  AST or call-graph pin).

### 3.4 `originator_subject` contract

Required when the engine is wired: the engine's 4-eyes distinctness check
(second approver ≠ first ≠ originator) and the envelope validation
(`originator_subject_missing`) are only meaningful with a real subject. Shape:
keyword-only `originator_subject: str = ""` — the default keeps the
engine-absent fixtures signature-compatible (that path never reads it). On the
WIRED path the seam adds NO empty-string pre-check (delegate-first doctrine;
the engine is authoritative): an empty originator is a caller-contract
violation, not an approval state, so it flows into `create_request`, which
refuses `ApprovalEnvelopeInvalid("originator_subject_missing")` and routes per
§3.5 (errored, not a policy refusal).

### 3.5 `ApprovalEnvelopeInvalid` routing

Envelope-invalid at the seam means a misconfigured entry or caller contract
violation (the manifest pipeline already validates `risk_tier` + `data_classes`
vocab at build/admission, so these are impossible-by-construction in a
correctly-admitted pack). It is a SYSTEM error, not a policy refusal: it
reaches `call_tool`'s existing generic-`Exception` arm → ERRORED evidence rows
(`audit.tool_invocation_error` + `error_taxonomy =
_GENERIC_INVOCATION_ERROR_TAXONOMY`), NOT a new refusal-reason value. This
ONLY works because of the §3.6 placement requirement — the approval consult
MUST sit inside the evidence-emitting `try`; placed before it (where the T10
gate sits today), `ApprovalEnvelopeInvalid` would escape `call_tool` with ZERO
evidence rows, violating the T11 every-request_id-emits invariant. Pinned by
one test per representative case (`risk_tier_unknown` via a hand-built
malformed entry) asserting the errored row EXISTS.

### 3.6 Evidence-envelope placement (structural requirement)

The wired approval consult runs INSIDE the existing evidence-emitting `try`
(the one wrapping `_call_tool_inner` at `mcp_host.py:1104`), BEFORE the
`_call_tool_inner` call — preserving the single "every request_id emits
evidence" envelope rather than manually duplicating errored-evidence emission
outside it. Required shape:

- The consult is a NESTED try/except inside the outer try body, because the
  `auto_tier_no_approval_required` catch must fall through to dispatch
  (`_call_tool_inner`) within the same envelope — an outer except arm cannot
  resume the try body.
- The nested arms catch the approval outcomes EXPLICITLY and in
  domain-specific-before-generic order (`ApprovalRequestNotFound`,
  `ApprovalTransitionRefused`): emit the refused evidence row
  (`mcp_session_id=None`, `token=None` — truthfully pre-token-acquire), then
  raise `MCPToolInvocationRefused`. `ApprovalEnvelopeInvalid` is deliberately
  NOT caught by the nested arms — it falls to the outer generic-`Exception`
  arm per §3.5.
- The OUTER try gains a bare re-raise guard arm — `except
  MCPToolInvocationRefused: raise` — ordered FIRST, before `MCPAuthzError` /
  `MCPTransportError` / generic `Exception`. Without it the generic arm
  catches the just-raised refusal (`MCPToolInvocationRefused` inherits
  `RuntimeError`, test-pinned) and DOUBLE-EMITS an errored row on top of the
  refused row. Pinned by a no-double-emission regression (exactly one
  audit row + one decision row per refused request_id) + the
  symmetric-exception-ordering doctrine.
- The engine-absent T10 fallback gate stays where it is today (before the
  try, manual emission) — byte-for-byte per §2.

## 4. Wire vocabulary (`ToolInvocationRefusalReason` 1 → 6)

```python
ToolInvocationRefusalReason = Literal[
    "tool_approval_engine_not_available",  # engine-absent fallback (kept)
    "tool_approval_pending",
    "tool_approval_denied",
    "tool_approval_expired",
    "tool_approval_binding_mismatch",
    "tool_approval_request_not_found",
]
```

- Wire-protocol-public (AGENTS.md MCP-host stop rule). Drift-pinned by a
  `typing.get_args` count + value test, mirroring the 13.5b1
  `_REFUSAL_STATUS` pin style.
- Every refusal flows through `_emit_call_evidence` with
  `event_type="audit.tool_invocation_refused"`, `decision="refused"`,
  `decision_reason=<the value>`; the approval-specific extras
  (`approval_request_id`, `declared_risk_tier`, and `flow` WHEN KNOWN) ride
  `extra_audit_payload` + `extra_decision_payload`. Value-free: arguments
  NEVER appear — only `args_digest.hex()`.
- **`flow` is included only where the seam already holds it without an extra
  read**: first-call pending (`create_request` returns `ApprovalRequest.flow`)
  and the re-call states where `verify_grant_for_action` RETURNS an
  `ApprovalCheckResult` (`pending` / `awaiting_second` / `denied` / `expired`).
  It is OMITTED for `tool_approval_request_not_found` (no flow exists by
  definition) and `tool_approval_binding_mismatch`
  (`verify_grant_for_action` raises `ApprovalTransitionRefused` without
  returning the result; the seam does NOT issue an extra store read to
  recover it).
- `sprint_13_5_followup=True` is EMITTED ONLY on the engine-absent fallback
  (it marks the transitional state, which the wired path has left).

## 5. `MCPServerEntry.data_classes`

```python
data_classes: tuple[str, ...] = ()
```

- Carried at registration/discovery time (the locked 13.5b call: metadata into
  the entry, no call-time manifest lookup). Sourced from the manifest
  `[data_governance].data_classes` (already closed-enum-validated by
  `cli/validators/data_governance.py` at build time + the registry at
  admission) whenever the future registration walk lands; the 5 existing
  test-entry builders set it explicitly.
- Default `()` keeps every existing constructor call green (additive).
- `DiscoveredMCPServer` is NOT extended in 13.5b2 (it is an explicit versioning
  boundary; the discovery surface doesn't need data classes yet) — recorded so
  the omission reads as deliberate.
- An out-of-vocab value in a hand-built entry refuses at engine envelope
  validation (`data_class_unknown`) → §3.5 errored routing.

## 6. Evidence + audit contract

- Step order preserved: approval gate (engine OR fallback) → token-acquire →
  session-open → send. A refused call MUST NOT touch the AS or the MCP server
  (T10 rationale, unchanged).
- Exactly ONE `audit.tool_invocation_*` row + ONE `mcp_call` decision row per
  `call_tool` request_id (T11 invariant, unchanged) — the approval refusals are
  new `decision_reason` values through the SAME helper, not new row shapes.
- The engine separately emits its own value-free `approval.requested` /
  `approval.granted*` / `approval.denied` / `approval.expired` chain rows
  (13.5a) — the seam does NOT duplicate them. Correlation: the seam's
  refusal/invocation rows carry `approval_request_id`; the approval rows carry
  the engine-minted `appr-*` request_id. An examiner joins on
  `approval_request_id` ↔ `approval_requests.request_id`.

## 7. Exception surface

`MCPToolInvocationRefused` is unchanged structurally (reason + message +
`**payload`); the new reasons ride the existing class. The pending refusal's
payload carries `approval_request_id` (str) + `flow` + `declared_risk_tier`;
its operator-facing message uses sanitized identifiers (R4 P2 doctrine,
unchanged). No new exception classes.

## 8. Testing

- **Byte-compat fallback suite:** the existing `test_mcp_high_risk_tier_refused.py`
  runs UNCHANGED (engine-absent fixtures) — the load-bearing pin that F2's
  fallback claim is real.
- **Wired-path suite** (new `test_mcp_approval_seam.py`): real
  `ApprovalRequestStore` + `ApprovalEngine` on a migrated SQLite DB (the
  13.5b1 `_mk_store` pattern) + `_StubPolicy` for flow control where OPA-free
  determinism is needed, + the REAL `ApprovalPolicy`/`tools.rego` for at least
  one auto-run classification test (OPA-binary-gated like the 13.5a policy
  tests if needed — follow the existing test_policy.py gating convention):
  - auto tier (read_only) → dispatches, no approval row.
  - high tier first call → `tool_approval_pending` + request_id in payload +
    refused evidence row carries it; NO token acquire (spy authz).
  - portal-grant → re-call dispatches (the e2e: engine.grant via core API or
    the 13.5b1 HTTP route — prefer the HTTP route for a true cross-surface e2e).
  - changed args on re-call → `tool_approval_binding_mismatch`.
  - denied / expired / unknown request_id / cross-tenant request_id →
    `tool_approval_denied` / `tool_approval_expired` /
    `tool_approval_request_not_found` (cross-tenant byte-identical to unknown).
  - empty originator → errored (NOT refused) per §3.5; malformed entry
    (`risk_tier_unknown`) → errored row EXISTS (the §3.5/§3.6 placement pin).
  - no-double-emission regression: a refused request_id produces EXACTLY one
    audit row + one decision row (pins the §3.6 outer
    `except MCPToolInvocationRefused: raise` guard arm).
  - vocabulary drift pin (get_args, 6 values).
  - `_canonical_tool_identity` vector + same-function pin.
- **CC discipline:** `mcp_host.py` is on the durable gate — verify-at-promotion
  against fresh `--cov-branch` at the batch boundary; gate count stays **128**
  (no new module promoted; `_canonical_tool_identity` lives in `mcp_host.py`).
