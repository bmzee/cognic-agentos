## 2. Seam shape (mirrors 13.5b2 ¬ß2)

`admit_policy` gains two keyword-only params (after `requires_credentials`):

```python
approval_engine: ApprovalEngine | None = None,
approval_request_id: uuid.UUID | None = None,
```

- `None` engine (default) ‚Üí **engine-absent fallback byte-for-byte**: Step 4
  refuses the 6-tier `_HIGH_RISK_TIERS_PRE_13_5` set with
  `sandbox_high_risk_tier_refused_pre_13_5` exactly as today; the Step-9 Rego
  input gains `approval_verified: False` (always threaded ‚Äî input-contract
  completeness per the precomputed-bool precedent), which changes nothing for
  safe tiers and keeps high tiers refused at the bundle.
- Wired ‚Üí ¬ß3. The engine import is runtime (`sandbox ‚Üí core` arrow is
  allowed); `Actor` stays TYPE_CHECKING-only as today.

## 3. Wired path (F2=B: engine-authoritative, every admission consults)

Step 4 is REPLACED (wired branch) with the approval consult. Sequencing
unchanged: the consult runs where Step 4 runs today ‚Äî after the
credential-precondition step, BEFORE caps/catalog/cosign/Rego ‚Äî so a refused
admission does no further I/O.

### 3.1 First admission (no `approval_request_id`)

1. Compute the binding digests (¬ß3.3).
2. Build the value-free `ApprovalEnvelope` (¬ß3.4) and call
   `engine.create_request(envelope=...)`.
3. `ApprovalTransitionRefused("auto_tier_no_approval_required")` ‚Üí
   tools.rego classified `auto_run` ‚Üí **proceed to Step 5** with
   `approval_verified = False`. (Safe tiers pass the bundle via the safe-tier
   arm; a hypothetically LOOSENED tools.rego classifying a high tier as auto
   is caught by the bundle's high-tier arm requiring `approval_verified` ‚Äî
   pure-Rego defence-in-depth against bundle loosening, test-pinned.)
4. Otherwise a `pending` request exists ‚Üí raise
   `SandboxLifecycleRefused("sandbox_approval_pending", detail=...,
   approval_request_id=str(request.request_id))` ‚Äî the correlator rides the
   NEW additive exception attr (¬ß4), and the detail string names the flow +
   the re-submit contract.

A tools.rego-TIGHTENED safe tier (e.g. overlay maps `internal_write` ‚Üí
`require_single_approval`) therefore refuses pending ‚Äî overlay tightening
honoured, test-pinned (the second F2=B direction).

### 3.2 Re-admission (`approval_request_id` supplied)

1. Recompute both digests (same helpers ‚Äî same-function-or-bust).
2. `engine.verify_grant_for_action(request_id=..., tenant_id=tenant_id,
   expected_args_digest=..., expected_tool_identity=...)`.
3. Mapping (all refusals carry the correlator attr; `flow` semantics follow
   13.5b2 ¬ß4 ‚Äî known for returned `ApprovalCheckResult` states, absent
   otherwise ‚Äî surfaced in `detail`, since `SandboxLifecycleRefused` has no
   payload dict):
   - `granted` ‚Üí set `approval_verified = True`, **continue Steps 5-9** (the
     bundle re-checks every other condition; approval does NOT bypass caps /
     catalog / cosign / egress).
   - `pending` / `awaiting_second` ‚Üí `sandbox_approval_pending`.
   - `denied` ‚Üí `sandbox_approval_denied`.
   - `expired` (incl. lazy-expired by this call) ‚Üí `sandbox_approval_expired`.
   - `ApprovalTransitionRefused("approval_binding_mismatch")` ‚Üí
     `sandbox_approval_binding_mismatch` (changed policy/pack identity ‚Äî a
     grant authorises exactly one admission shape).
   - `ApprovalRequestNotFound` (unknown OR cross-tenant, indistinguishable by
     construction) ‚Üí `sandbox_approval_request_not_found`.

### 3.3 Binding digests (the F4 pattern, sandbox-shaped)

One module-level pure helper pair in `sandbox/admission.py`, used by BOTH the
create and verify paths:

```python
def _canonical_sandbox_identity(*, pack_id: str, pack_artifact_digest: str) -> str:
    return "sandbox:" + hashlib.sha256(
        canonical_bytes({"pack_id": pack_id, "pack_artifact_digest": pack_artifact_digest})
    ).hexdigest()
```

- Identity binds the **immutable** pack identity (`pack_artifact_digest` is
  the cosign-verified sha256 per the `PackAdmissionContext` docstring ‚Äî
  `pack_version` is human-mutable and deliberately excluded). 72 chars
  (`sandbox:` + 64) fits `approval_requests.tool_identity` `String(256)`.
- `args_digest = sha256(canonical_bytes({"policy":
  _policy_binding_projection(policy), "pack_context": <6-key pack_context
  projection>}))`. The policy projection is a NEW dedicated helper covering
  **every security-load-bearing `SandboxPolicy` field** ‚Äî `cpu_cores`,
  `cpu_time_budget_s`, `memory_mb`, `walltime_s`, **`runtime_image`**,
  `egress_allow_list`, `vault_path`, **`read_only_root`**,
  **`writable_mounts`** (projected as dicts; tuples ‚Üí lists per the
  canonical-form no-tuples doctrine) ‚Äî and EXCLUDES only `warm_pool_key`
  (an internal pooling hint, not approved behaviour). The Step-9 Rego
  `policy` projection (5-key) is deliberately NARROWER and stays its own
  shape ‚Äî it omits `runtime_image`/`read_only_root`/`writable_mounts`
  because the bundle gates those via other inputs, but the GRANT must bind
  them: an image swap, a root-fs writability flip, or a mount change between
  grant and re-admit MUST `approval_binding_mismatch`.
- The pack_context projection is the same 6-key shape Step 9 sends. The NEW
  `data_classes` field is in NEITHER the digest NOR the Step-9 Rego input:
  it rides the envelope's first-class `data_classes` field, and it is
  immutable-by-identity anyway (same `pack_artifact_digest` ‚áí same manifest
  ‚áí same data classes).
- `tenant_max` + the catalog bools are DELIBERATELY excluded from the
  digest: caps are enforcement state, not the approved action ‚Äî a tenant-cap
  overlay change between grant and re-admit must not break the binding (the
  bundle still enforces the current caps at Step 9).

### 3.4 Envelope sourcing

| Envelope field | Source |
|---|---|
| `risk_tier` | `pack_context.risk_tier` (validated `RiskTier` at dataclass construction) |
| `tool_identity` | `_canonical_sandbox_identity(...)` (¬ß3.3) |
| `originator_subject` | `actor.subject` (the requesting Actor `admit_policy` already takes) |
| `tenant_id` | the existing `tenant_id` param |
| `data_classes` | `pack_context.data_classes` (NEW additive field, default `()` ‚Äî the harness populates from the manifest `[data_governance].data_classes`, exactly the `MCPServerEntry.data_classes` pattern) |
| `args_digest` | ¬ß3.3 |
| `redacted_context` | bounded sanitized descriptor: `f"sandbox_admission pack_id={...} pack_version={...} runtime_image={policy.runtime_image}"` truncated ‚â§ `APPROVAL_REDACTED_CONTEXT_MAX_LEN` (reviewers see WHICH pack/image in the 13.5b1 portal panel) |
| `required_refs` | `{}` except `regulator_communication` ‚Üí `{"audit_record_ref": <admission correlation id>}` (¬ß3.5) |

### 3.5 Admission correlation id (LOCKED ‚Äî seam-minted)

`admit_policy` has no request-id param (unlike `call_tool`), and there is NO
reusable pre-admission id: verified at reconciliation, `admit_policy` runs
BEFORE `session_id` creation in both the docker and k8s backend paths, and
`emit_sandbox_event` mints its own `sandbox-evt-*` only at emission time. The
seam therefore mints
`admission_correlation_id = f"sandbox-admit-{uuid4().hex}"` at consult time,
uses it as the regulator-tier `required_refs["audit_record_ref"]`, AND
includes it in the pending refusal's `detail` so the correlator chain is
examiner-followable.

### 3.6 `ApprovalEnvelopeInvalid` routing

Propagates RAW (fail-loud). Unlike `call_tool`, `admit_policy` has no
evidence-emitting envelope ‚Äî it raises typed refusals and the backend owns
evidence emission. An envelope-invalid here is a system misconfig
(manifest-validated fields), not a policy refusal; wrapping it in a new
closed-enum value would launder a bug into policy vocabulary. Pinned by one
test (malformed hand-built context ‚Üí `ApprovalEnvelopeInvalid` escapes).

## 5. `sandbox.rego` CONVERT (the c1 design lock)

The allow rule's tier conjuncts are extracted into a `_tier_admissible`
helper with two arms; everything else is untouched:

```rego
allow if {
    _tier_admissible
    _within_tenant_max
    _credential_precondition_satisfied
    _runtime_image_authorised
    _egress_http_only
    _credential_ttl_within_tenant_max
}

# Safe tiers admissible as before (disjointness guard preserved).
_tier_admissible if {
    input.pack_context.risk_tier in safe_tiers
    not input.pack_context.risk_tier in high_risk_tiers
}

# Sprint 13.5c1 CONVERT: a high-risk tier is admissible ONLY when the
# Python seam attests a verified grant (strict bool ‚Äî absence is falsy,
# fail-closed; mirrors the precomputed-bool precedent).
_tier_admissible if {
    input.pack_context.risk_tier in high_risk_tiers
    input.approval_verified == true
}
```

- Python ALWAYS threads `approval_verified` (False except the ¬ß3.2 granted
  path) ‚Äî input-contract completeness.
- The bundle stays pure + fail-closed: engine-absent or bypassed-Python
  callers can never admit a high tier (`approval_verified` absent/false).
- Comment-block reality fixes in the same commit: rule-2 prose rewritten from
  "refuse unconditionally ‚Ä¶ pre-13.5" to the attestation contract, and the
  stale "Cutover at Sprint 13.5 module-load ‚Ä¶ audit event
  (sandbox_approval.engine_enabled)" paragraph replaced by the landed ¬ß6
  evidence shape (comment drift is doctrine drift on a stop-rule bundle).
- Default `allow := false` + bank-overlays-may-tighten posture unchanged;
  THIS amendment is the coordinated kernel + ADR change the stop rule
  requires for loosening.

## 6. Cutover evidence (the `engine_enabled` promise superseded; sandbox-event deferred)

The Rego comments promised a one-shot `sandbox_approval.engine_enabled` audit
event. That is NOT implementable in a stateless seam-only sprint:
`admit_policy` is a pure function (process-once flags are untestable smell),
it holds no `DecisionHistoryStore` (it cannot emit chain rows itself), and ‚Äî
verified at reconciliation ‚Äî admission refusal happens BEFORE session
creation and the backends do NOT catch `SandboxLifecycleRefused` around
`admit_policy` to emit a lifecycle row. A sandbox-side
`approval_gated_admission` event would therefore require docker + k8s backend
catch-and-emit work, a pre-session `session_id` substitute, and
audit-failure-semantics tests ‚Äî a real scope expansion.

**c1 records cutover evidence as (DEFERRING the sandbox-specific event to the
future backend/composition wiring sprint):**

1. The engine's own value-free `approval.*` chain rows (13.5a) ‚Äî every
   created / granted / denied / expired decision is already hash-chained;
   the ¬ß3.5 `admission_correlation_id` + the request's `tool_identity`
   (`sandbox:` prefix) make sandbox-originated requests examiner-isolable.
2. The raised sandbox approval refusal carrying `approval_request_id` (the
   new additive exception attr) ‚Äî the caller-visible correlator.
3. Rego unit tests proving `approval_verified` is REQUIRED for high tiers ‚Äî
   the bundle-level proof that the cutover regime is in force.
4. ADR-014 c1 amendment + the `sandbox.rego` comment fix superseding the
   one-shot `engine_enabled` promise explicitly (not silently dropping it).

The `SandboxLifecycleEvent` taxonomy is NOT extended in c1; the
`approval_gated_admission` per-decision event lands with the backend
wiring sprint that gives it an emission seam.
