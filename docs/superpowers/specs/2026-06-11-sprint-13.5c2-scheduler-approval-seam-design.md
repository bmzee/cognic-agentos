## 3. Wired path (engine-authoritative, every submit consults)

### 3.1 Step order

```text
1. parent-budget narrowing (unchanged)        → effective tokens
2. pack_state.is_installed                    → refused_pack_not_installed
3. kill_switch.is_active                      → refused_kill_switch_active
3.5 APPROVAL CONSULT (new; wired only)        → refused_approval_* | verified bool
4. policy_evaluator (rego input now carries
   the engine-computed approval_verified)     → refused_policy_denied
5. quota.would_admit                          → refused_quota_exhausted
6-8. caps / queue                             → accepted_* | refused_queue_full
```

Pack-not-installed and kill-switch BEAT approval (a killed or uninstalled
pack never reaches `create_request` — zero approval rows; ordering pinned).
Approval BEATS policy and quota: no Rego eval before the attestation is
computed (the input needs the bool), and no quota reservation is ever made
for an unapproved high-tier task. A verified grant does NOT bypass policy /
quota / caps / queue — Steps 4–8 still run.

### 3.2 First admission (`approval_request_id is None`)

The consult computes the binding digests (§3.3), builds the value-free
envelope (§3.4), and calls `engine.create_request(envelope=...)`:

- `ApprovalTransitionRefused("auto_tier_no_approval_required")` →
  `approval_verified = False`, proceed to Step 4 (classification is
  engine-authoritative via `tools.rego`; an overlay-TIGHTENED bundle on a
  safe tier is honoured — pending fires even though `scheduler.rego`'s safe
  arm would admit).
- Any other flow → return
  `AdmissionDecision(outcome="refused_approval_pending", task_id=None,
  approval_request_id=str(request.request_id))` + ONE
  `scheduler.admission_refused` chain row whose payload carries
  `approval_request_id` + `approval_flow=request.flow` (§6).

### 3.3 Binding digests (F4 — actor-bound, USER-LOCKED)

One module-level pure helper pair in `core/scheduler/engine.py`, used by
BOTH the create and verify paths:

```python
tool_identity = "scheduler:" + hashlib.sha256(
    canonical_bytes({"pack_id": pack_id, "pack_kind": pack_kind})
).hexdigest()                       # 74 chars; fits String(256)

args_digest = hashlib.sha256(canonical_bytes({
    "class": submit_input.class_,
    "pack_risk_tier": submit_input.pack_risk_tier,
    "requested_estimated_tokens": submit_input.requested_estimated_tokens,
    "parent_task_id": submit_input.parent_task_id,
    "actor_subject": submit_input.actor.subject,
    "actor_type": submit_input.actor.actor_type,
})).digest()                        # bytes — ApprovalEnvelope.args_digest: bytes
```

- **Actor identity is IN the binding** (the c2 refinement): the engine's
  replay gate compares only `args_digest` + `tool_identity`, so without
  `actor_subject`/`actor_type` a granted scheduler approval could be
  re-submitted by a DIFFERENT actor in the same tenant with the same
  pack/class/token shape. An actor swap between grant and re-submit MUST
  `approval_binding_mismatch`.
- **Requested tokens, NOT parent-narrowed effective tokens**: narrowing
  shifts as the parent spends between grant and re-submit; binding the
  effective value would spuriously mismatch. The digest binds the caller's
  declared intent — and because the consult (Step 3.5) runs AFTER
  parent-budget narrowing (Step 1), which REPLACES
  `requested_estimated_tokens` on the effective input
  (`engine.py:337-341`), the digest helper MUST read the ORIGINAL
  `submit_input` (the pre-replace object), never `effective_submit_input`.
  Quota and storage continue to see the narrowed value; only the binding
  digest reads the original. Pinned by a test where a parent-budget-
  narrowed re-submit still verifies (no spurious mismatch).
- **Name-based identity (honest weakness):** `SubmitInput` carries no
  artifact digest (the scheduler never sees the wheel; `PackStateInterrogator`
  answers installed-or-not). `tool_identity` is therefore name-based
  (`pack_id` + `pack_kind`). Defence-in-depth: the artifact-bound identity
  (`sandbox:` + cosign-verified `pack_artifact_digest`, 13.5c1) fires
  downstream at the SANDBOX seam when the admitted task actually starts.
- `tenant_id` is envelope-first-class (engine tenant-binding), in neither
  digest. `data_classes` in neither digest. `approval_request_id` /
  `approval_verified` are carrier/attestation — never digested.
- **Every-field-dispositioned drift pin (c1 doctrine extended):** a test
  walks `dataclasses.fields(SubmitInput)` against an explicit disposition
  map (digest / identity / envelope-first-class / excluded-by-design) so a
  future `SubmitInput` field FAILS the pin until its binding decision is
  made explicitly.

### 3.5 Required refs (NO minted correlator — the b2 pattern)

For `risk_tier == "regulator_communication"`:
`required_refs = {"audit_record_ref": request_id}` — the `submit(*,
request_id)` parameter every `scheduler.admission_accepted` /
`scheduler.admission_refused` chain row is keyed by. Unlike c1 (sandbox
admission runs pre-session with no usable id), the scheduler HAS a natural
pre-existing correlator; no `scheduler-admit-<uuid>` is minted. All other
tiers: `required_refs = {}`.

### 3.6 Re-submission (`approval_request_id` supplied, engine wired)

`verify_grant_for_action(request_id=<parsed UUID>, tenant_id=...,
expected_args_digest=<§3.3>, expected_tool_identity=<§3.3>)`:

- `ApprovalRequestNotFound` → `refused_approval_request_not_found`
  (cross-tenant == unknown BY CONSTRUCTION — the engine's load is
  tenant-scoped; no probe can distinguish).
- `ApprovalTransitionRefused("approval_binding_mismatch")` →
  `refused_approval_binding_mismatch` ("a grant authorises exactly one
  submission shape"). Any OTHER `ApprovalTransitionRefused` reason
  re-raises RAW (none is reachable from the verify path today; fail-loud
  beats silent mapping).
- Returned `res.state == "granted"` → `approval_verified = True`, proceed
  to Step 4. Steps 4–8 still run (no bypass).
- `res.state in ("pending", "awaiting_second")` → `refused_approval_pending`;
  `"denied"` → `refused_approval_denied`; `"expired"` →
  `refused_approval_expired` (lazy authoritative expiry — the engine
  flipped the row + emitted `approval.expired` on this read).
- Every refused arm returns `AdmissionDecision(..., approval_request_id=
  str(request_id))` on the pending arm ONLY (the caller needs the id to
  keep polling; denied/expired/not-found/mismatch carry `task_id=None`
  without it) + emits ONE `scheduler.admission_refused` row with the §6
  payload keys.

### 3.7 `ApprovalEnvelopeInvalid` routing

Propagates RAW (fail-loud), no chain row — a malformed envelope is a
seam-construction bug, not an admission decision. Mirrors both the c1
choice AND the scheduler's existing typed-raise style
(`SchedulerSubmitInputInvalid` raises before any emission).

## 6. Cutover evidence (per-decision, DELIVERED — the c1 contrast)

Unlike c1 (stateless seam → evidence deferred to backend wiring), the
scheduler owns a chain-row path for every admission decision, so c2
delivers real per-decision evidence NOW:

- `scheduler.admission_refused` payload gains CONDITIONAL keys
  `approval_request_id` (str) + `approval_flow` (the engine's flow value)
  — included ONLY when known: first-call pending (both, from the returned
  `ApprovalRequest`), verify-returned states pending/denied/expired (both,
  from `ApprovalCheckResult.flow`), binding-mismatch + not-found
  (`approval_request_id` only — no result object is returned; no extra
  store read, the b2 doctrine). Conditional inclusion keeps every
  NON-approval refusal row byte-identical to pre-c2 (additive-only
  schema; the 5 existing reasons' payload shape is pinned unchanged).
- `scheduler.admission_accepted` payload gains `approval_verified` (bool,
  ALWAYS present post-c2: `False` on safe/auto-tier admissions, `True` on
  granted re-submissions) + CONDITIONALLY `approval_request_id` (str) when
  a granted re-submit is accepted. Without the correlator on the accepted
  row, the examiner join `scheduler.admission_accepted` → the engine's
  `approval.*` rows is weak or impossible for non-regulator tiers (the
  `audit_record_ref=request_id` back-link exists ONLY for
  `regulator_communication`; other tiers' approval rows carry the
  `scheduler:` identity but no scheduler submit request id). The accepted
  row joins through `approval_request_id` to recover `approval_flow` from
  the approval record — flow itself stays refusal-evidence-only.
  Non-approval accepted rows carry `approval_verified=False` and NO
  correlator key. Historical rows lack both keys — readers handle all
  shapes (the `conformance`-key precedent on pack submit rows).
- The engine's own value-free `approval.*` chain rows (13.5a) remain the
  authoritative decision evidence; scheduler-originated requests are
  examiner-isolable via the `scheduler:` tool-identity prefix + the
  `audit_record_ref=request_id` back-link.
- The one-shot "cutover is an audit event" promise (`scheduler.rego:35-36`)
  is EXPLICITLY superseded (same process-once rationale as c1's
  `engine_enabled`), with the difference recorded: per-decision evidence is
  COMPLETE at this seam — nothing is deferred to backend wiring.
