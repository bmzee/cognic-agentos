## 2. Seam shape

`MemoryAPI.__init__` (`api.py:128`) gains:

```python
approval_engine: ApprovalEngine | None = None,
```

threaded into the internally-constructed `MemoryGate` (`api.py:151`), which
gains the same kwarg (direct `core.approval` import — the dependency landed
at 13.5a; no `_seams.py` Protocol). The harness factory
(`harness/runtime.py:210`) passes the unconditionally-built engine:
`approval_engine=approval_engine` — **the first live consult**: deployments
with high-tier contexts writing `long_term` go from the hard
`memory_approval_engine_not_available` refusal to the pending → grant flow
(strictly more functional; exactly ADR-014's intent). `MemoryAPI`'s direct
constructor keeps the `None` default, so every existing test/caller is
byte-compatible.

**Engine-absent fallback (byte-for-byte):** Step 7's predicate and reason
are UNCHANGED on the unwired arm — `tier == "long_term" and ctx.risk_tier
in _APPROVAL_REQUIRED_RISK_TIERS` → `memory_approval_engine_not_available`
(`gate.py:272-274`). The existing suite passing unchanged is the pin.

**The consult trigger is the SAME predicate on BOTH arms (F2 LOCK).** The
5-value `_APPROVAL_REQUIRED_RISK_TIERS` (`gate.py:103-111`; ADR-019:120
"risk_tier >= customer_data_write" — `customer_data_read` DELIBERATELY
excluded) gates the consult when wired, exactly as it gates the refusal
when unwired. This seam deliberately does NOT honour `tools.rego` overlay
tightening below the 5-set: a bank wanting to gate lower tiers' long_term
writes already has the tenant-overridable `long_term.allow` Rego point at
Step 2. Within the 5-set, classification IS engine-authoritative via
`tools.rego` through `create_request`; the catch-`auto_tier_no_approval_
required` → proceed arm is kept for template fidelity + defence (under the
default bundle it is unreachable for the 5 tiers — all map to approval
flows).

## 3. Wired path

### 3.1 Consult site + the ordering LOCK

The consult replaces Step 7's raise — and Step 7 stays LAST. **Approval
never bypasses Steps 0–6** (the user-locked ordering rule): kill-switch,
sub-agent durable-access, subject scope, long_term Rego admission, DLP,
restricted-class Rego, purpose, consent, and the retention clamp all run
first; any Steps-0–6 refusal raises its own reason with ZERO approval rows
(no `create_request` ever fires) — pinned per-step (at minimum: DLP-failure
+ wired engine, consent-failure + wired engine, kill-switch + wired
engine). Because the gate's "ordered precedence IS the contract" doctrine
already places Step 7 last, no step-reordering is needed — the lock is
enforced by construction and pinned by tests.

A consequence worth recording: Steps 0–6 RE-RUN on every attempt
(including the granted re-write), so consent, DLP, purpose, and tenant
policy are re-validated at re-submit time — the binding digest (§3.3) only
needs to cover what those steps do NOT re-check.

### 3.2 Request-id hoist (F3 LOCK)

The Step-8 mint `request_id = f"memory-write-{uuid.uuid4().hex}"`
(`gate.py:287`) HOISTS above Step 7. The same id then serves as:
- the approval envelope's `required_refs["audit_record_ref"]` for
  `risk_tier == "regulator_communication"` (all other tiers: `{}`),
- the success-path `MemoryWriteRecord.request_id` (unchanged semantics),
  and therefore the `memory.write` chain row's request id.

Nothing seam-minted; no new prefix. **Forward-edge contract (USER-LOCKED
at spec review):** each attempt mints fresh and memory emits no refusal
chain row, so without more, the regulator-tier `audit_record_ref` bound at
create time (the FIRST attempt's id) would be an orphan correlator with no
forward edge. c3 closes it bidirectionally, the leanest-robust way:

- **The 13.5a engine read surface gains `required_refs`** (additive
  CC-ADJ): `ApprovalCheckResult` gains `required_refs: dict[str, str]`
  (it carries state/flow/identity/digests today, NOT refs — verified) and
  the engine-facing storage load projects the persisted column so
  `verify_grant_for_action` echoes the refs trustworthily. A caller-passed
  ref was rejected — caller-supplied evidence is forgeable; the read-back
  comes from the engine's own persisted row.
- **On a granted re-write the gate persists the forward edge:**
  `MemoryWriteRecord.approval_audit_record_ref =
  res.required_refs.get("audit_record_ref")` (None → key absent). The
  granted regulator write's `memory.write` row therefore carries BOTH
  `approval_request_id` (the join to the engine's `approval.*` rows) AND
  `approval_audit_record_ref` (the original attempt's `memory-write-*`
  id) — examiners traverse both directions from the final row, and the
  approval row's ref is no longer orphaned.
- The pending exception stays LEAN: `approval_request_id` only (the
  user-locked leanest variant; the attempt id is recoverable from the
  approval row's `required_refs` and, post-grant, from the final row).

Non-regulator tiers bind `required_refs = {}` as before, so their granted
rows carry no `approval_audit_record_ref` key.

### 3.3 Binding digests (F4 LOCK — value digest, never raw value)

Module-level pure helpers in `core/memory/gate.py`, used by BOTH create
and verify:

```python
tool_identity = "memory:" + hashlib.sha256(
    canonical_bytes({"agent_id": ctx.agent_id, "operation": "memory_write"})
).hexdigest()              # 71 chars ("memory:" = 7 + 64); fits String(256)

args_digest = hashlib.sha256(canonical_bytes({
    "tier": tier,
    "purpose": purpose,
    "data_classes": sorted(data_classes),
    "key": key,                                   # None for block writes
    "block_kind": block_kind,                     # None for keyed writes
    "subject": resolved_subject.canonical,
    "actor_id": ctx.actor_id,
    "risk_tier": ctx.risk_tier,
    "value_digest": _value_digest(value),
})).digest()
```

- **The engine NEVER sees the raw value** (F4): the binding carries
  `_value_digest(value)` — the SAME `sha256(canonical_bytes(value))
  .hexdigest()` helper the `memory.write` chain row already uses
  (`storage.py:193-202`, payload key `redacted_value_digest`) — single
  digest definition, no drift; imported by the gate from `storage`.
  A grant authorises exactly one write **shape including content**: a
  value change between grant and re-write MUST `binding_mismatch`.
- **Caller identity is IN the binding** (the c2 refinement carried
  forward): `actor_id` in the digest; `agent_id` in the identity. A
  different actor in the same tenant cannot ride a grant.
- **`risk_tier` is IN the digest** (user F4 list) — a context re-bound at
  a different tier cannot reuse a grant.
- Exclusions (each with rationale, dispositioned by a drift pin over the
  `check_write` parameter list + the context fields it reads):
  `tenant_id` (envelope-first-class; engine tenant-binding),
  `consent_token` + `retention_window_s` / `tenant_retention_max_s`
  (Steps 5/6 RE-RUN on every attempt per §3.1 — the gate re-validates;
  retention is a clamp, not approved behaviour), `approval_request_id`
  (carrier), and the remaining `MemoryCallerContext` fields
  (`declared_purposes` / `declared_data_classes` / capability flags — gate
  inputs re-checked at Steps 2/4, not approved shape).

### 3.4 Envelope sourcing

```python
ApprovalEnvelope(
    risk_tier=ctx.risk_tier,
    tool_identity=<§3.3>,
    originator_subject=ctx.actor_id,      # the acting subject; 4-eyes
                                          # distinctness compares approvers
                                          # against THIS
    tenant_id=ctx.tenant_id,
    data_classes=tuple(data_classes),     # the CALL's declared classes
    args_digest=<§3.3>,
    redacted_context=(
        f"memory_write agent_id={ctx.agent_id} tier={tier} "
        f"risk_tier={ctx.risk_tier} "
        f"target={block_kind or key}"
    )[:APPROVAL_REDACTED_CONTEXT_MAX_LEN],
    required_refs=<§3.2>,
)
```

`redacted_context` carries metadata only (agent, tier, target key/block
name) — key names are caller-chosen identifiers, included so the 13.5b1
reviewer panel shows WHAT is being approved; the VALUE never appears
anywhere (F4).

### 3.5 First write (no `approval_request_id`)

Consult fires only on the §2 predicate. `create_request(envelope=...)`:
- catch `ApprovalTransitionRefused("auto_tier_no_approval_required")` →
  proceed to Step 8 with `approval_verified=False` (defensive; unreachable
  under the default bundle for the 5-set).
- otherwise → raise
  `MemoryOperationRefused("memory_approval_pending")` carrying the NEW
  additive `approval_request_id=str(request.request_id)` attr (the c1
  exception-carrier pattern; `MemoryOperationRefused.__init__` gains the
  keyword-only defaulted param — existing raise sites unchanged).

### 3.6 Re-write (`approval_request_id` supplied)

New keyword-only param on `remember()` / `upsert_block()` →
`check_write(approval_request_id=...)`, parsed via `uuid.UUID(...)` inside
the consult. **Malformed-id LOCK:** a malformed string raises `ValueError`
from the gate — the same fail-loud caller-contract-bug class as the
blocks-are-long_term-only precondition (`gate.py:201-202`). Memory has no
`SchedulerSubmitInputInvalid` equivalent, and minting a closed-enum
refusal reason for a malformed string would blur caller-bug vs
governance-refusal (the `MemoryRefusalReason` vocabulary is reserved for
governance outcomes). A VALID id supplied while the engine is unwired, or
on a non-consulted write (safe tier / scratch / task), is INERT — pinned.

`verify_grant_for_action(request_id=..., tenant_id=ctx.tenant_id,
expected_args_digest=..., expected_tool_identity=...)`:
- `ApprovalRequestNotFound` → `memory_approval_request_not_found`
  (cross-tenant == unknown by construction).
- `ApprovalTransitionRefused("approval_binding_mismatch")` →
  `memory_approval_binding_mismatch` ("a grant authorises exactly one
  write shape"); any OTHER reason re-raises RAW (fail-loud; the c1/c2
  arm).
- `res.state == "granted"` → proceed to Step 8 with
  `approval_verified=True`.
- `pending`/`awaiting_second` → `memory_approval_pending` (attr carries
  the id); `denied` → `memory_approval_denied`; `expired` →
  `memory_approval_expired` (lazy authoritative expiry).

### 3.7 `ApprovalEnvelopeInvalid` routing

Propagates RAW (no evidence envelope at this seam; the c1 precedent).

## 6. Evidence (F5 + F7 LOCKS — c2's delivered model, exception-style refusals)

- `MemoryWriteRecord` gains 3 defaulted fields (`_context.py:56-71`):
  `approval_verified: bool = False` + `approval_request_id: str | None =
  None` + `approval_audit_record_ref: str | None = None`. The gate sets
  them at Step 8 (`approval_verified` from the consult outcome; the other
  two ONLY when a verified grant admitted the write — the ref read back
  from `ApprovalCheckResult.required_refs` per §3.2). The record is
  gate-built — callers never construct it, so there is no forgery surface
  (no c2-style overwrite dance needed; recorded as a doctrine note).
- `memory.write` chain payload (`storage.py:253-276`) gains
  `approval_verified` (ALWAYS present post-c3; `False` on safe-tier /
  unconsulted writes) + CONDITIONAL `approval_request_id` when
  `record.approval_verified and record.approval_request_id is not None` —
  the examiner join `memory.write` → the engine's `approval.*` rows (the
  c2 user-P1 reasoning applied proactively) — + CONDITIONAL
  `approval_audit_record_ref` when present on the record (regulator-tier
  granted writes only; the §3.2 forward edge). Historical rows lack the
  keys — readers handle both (the established additive-schema precedent).
- Refusals stay exception-only: NO new chain event types; pending /
  denied / expired / mismatch / not-found surface as §4 reasons + the
  attr. The engine's own value-free `approval.*` rows are the decision
  evidence; memory-originated requests are examiner-isolable via the
  `memory:` tool-identity prefix.
- The one-shot `memory_approval.engine_enabled` promise (ADR-019:122) is
  EXPLICITLY superseded (F7) — third instance of the promise, same
  process-once rationale as c1/c2, and here per-decision evidence is
  delivered on the row that matters (`memory.write`).
