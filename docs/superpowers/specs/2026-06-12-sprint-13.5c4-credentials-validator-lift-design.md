## 1. Scope + the safety case (USER-REQUIRED framing)

**This sprint is build-time de-blocking ONLY. There is NO runtime behavior
change.** The safety case: the four runtime approval seams (sandbox
admission, scheduler submit, MCP host, memory write gate — 13.5b2/c1/c2/c3)
now OWN high-tier approval enforcement, each with its engine-absent
fail-closed fallback. The credentials validator runs at BUILD time and
cannot know deployment wiring, so its pre-13.5 high-risk-tier refusal is
now pure double-blocking: it prevents legitimately-approvable high-tier
credential packs from even being built, while the runtime would gate them
correctly either way (wired → pending/grant flow; unwired → the seams'
own refusals). The refusal's OWN message names the lift condition —
"until Sprint 13.5's approval engine lands and the ADR-014 coordinated
amendment lifts this restriction" — and both conditions are satisfied.

IN scope: the REMOVE (validator check + vocabulary value + ownership-map
entry + test pins) and the ADR-014 Sprint-13.5c4 closing section. OUT:
everything else — no runtime module, no Rego, no other validators, no new
vocabulary. The capability-map pillar-6 gap line updates at closeout.

`ValidatorReason` Literal and the ownership map. Rationale: this is

never-emittable value preserves confusion, not compatibility. Every
