## 3. Kill-switch engine (`core/emergency/kill_switches.py` — on-gate CC, stop-rule)

New `KillSwitchEngine` class ALONGSIDE the untouched `RedisMemoryWriteFreezeKillSwitch` seed.

- **Classes (closed enum `KillSwitchClass`):** `memory_write_freeze` / `pack` / `tool` / `model`
  / `tenant_packs` / `tenant_full` / `cloud_routing` / `feature`.
- **Key scheme (review patch 2):** `cognic:killswitch:<class>:<scope_key>` — no class-name
  duplication inside the scope_key. Per-class `<scope_key>`: `pack` → `<pack_id>`; `tool` →
  `<tool_id>`; `model` → `<litellm_alias>` (alias-only doctrine — see §11 item 5; global
  switches); `tenant_packs` / `tenant_full` →
  `<tenant_id>`; `cloud_routing` → `global`; `feature` → `<name>` (global) or
  `<tenant_id>:<name>` (per-tenant — ADR-018 §42 "per-tenant or globally"). The seed key
  (`cognic:killswitch:memory_write_freeze:<tenant_id>`) already conforms.
- **Redis payload (new classes):** `{"active": bool, "updated_at", "actor_id", "reason"}` —
  mirrors the seed's shape with `active` in place of `frozen`. The flip CATEGORY lives on the
  chain row only (uniform for all 8 classes; the seed's Redis payload shape stays untouched).
- **Fail-closed read:** identical semantics to the seed per ADR-018 §48 — last-known-good cache
  ≤ `emergency_kill_switch_cache_ttl_s` (new Setting, default 60, capped at 60 like the memory
  TTL), then ACTIVE (refuse) on Redis unavailability or malformed payload. Redis stays
  `/readyz`-critical.
- **Read API (consumer-facing):** `is_active_for_scheduler(*, tenant_id, pack_id)` (the F3
  aggregation; exposed via a thin conformer object satisfying `KillSwitchInterrogator` so the
  engine class itself is not signature-bound to one consumer), and
  `check_gateway(*, tenant_id, model_alias, external) -> KillSwitchClass | None` (evaluates
  `model` keyed by the LiteLLM ALIAS per §11 item 5, `cloud_routing` when `external`,
  `tenant_full`; returns the FIRST tripped class for the refusal payload, deterministic
  precedence `tenant_full → model → cloud_routing`).
- **Write API + the brake-before-evidence doctrine (review patch 3):** `flip(*, class_,
  scope_key, actor_id, reason, category)` + `revert(...)` — Redis write FIRST (the brake takes
  effect even if evidence fails), THEN the chain row + UI emit. On evidence-append failure after
  a successful Redis write: the switch STAYS live (evidence failure must never resurrect a
  killed path); the API returns an explicit evidence-degraded error (closed-enum
  `kill_switch_live_evidence_degraded` — the operator KNOWS the brake is on but the row is
  missing) and a loud structured log fires. Retrying the same flip is IDEMPOTENT: re-flipping an
  already-active switch performs no Redis state change but appends the evidence row, so the
  chain converges on retry. `set_write_freeze` on the seed remains; the portal memory-freeze
  route delegates to it under the same doctrine.
- **Enforcement-status honesty (review patch 6 + flag 4):** `tool` and `feature` flips are
  "armed, evidence-supported, NO live consumer" until the composition-root / MCP / sandbox /
  subagent wiring lands. The flip chain payload AND the portal list response carry
  `enforcement_status: "live" | "armed_no_live_consumer"` (engine-owned static per-class map,
  updated when consumers wire) so an operator cannot mistake an armed switch for an enforced
  one.

## 5. Quota engine (Half 2 — `core/emergency/quotas.py`, NEW on-gate CC, stop-rule)

- **Wave-1 classes:** `tokens_per_pack_per_day` + `tokens_per_tenant_per_day` ONLY. Tool-rate /
  subagent classes defer to the sprints where their consumer surfaces get production wiring;
  spend classes defer per §11 flag 1.
- **Meter:** Redis windowed counters, UTC calendar-day windows, ~48h TTL:
  `cognic:quota:tokens:<tenant_id>:<yyyymmdd>` + `cognic:quota:tokens:<tenant_id>:<pack_id>:<yyyymmdd>`.
  Two counter families — RESERVATIONS (scheduler plane) and ACTUALS (gateway plane) — so
  reservation + actual usage never double-count; `would_admit` checks
  (actuals + outstanding reservations + requested) ≤ limit atomically (single Lua script, one
  round trip) and writes the reservation hash `cognic:quota:reservation:<task_id>`.
  `release_reservation` is idempotent (guarded DEL + DECRBY; unknown task_id is a no-op) per the
  10.5b T9 contract.
- **Gateway increments + enforcement:** post-completion, increment ACTUALS by the provider-usage
  token counts where present (value-free counts only). At the F4 gate slot the gateway also
  refuses when the tenant/pack hard limit is already breached — outcome `quota_exhausted`
  (13th enum value) + `quota.exhausted` chain row (hard breach only; 80% soft-threshold is a
  structured-log warn in Wave-1, no chain row — see §11 flag 3).
- **Fail-posture (review patch 1):** NO cache-then-allow for reservation. `would_admit` on ANY
  Redis error returns `False` (refuse immediately) — a reservation that cannot be made
  atomically must never be granted from cache. `release_reservation` on Redis error logs loud
  and is retry-safe (idempotent). The last-known-good cache is permitted ONLY for the read-only
  portal usage display (`GET /quotas`), never for admission.
- **Limits source:** kernel defaults in Settings; per-tenant limits via the shipped ADR-023
  tenant config overlay (tighten-only fits quotas exactly) — see §11 flag 2. The ADR's
  `PUT /api/v1/quotas` write surface defers with that flag.
