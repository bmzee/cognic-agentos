## 1. Scope + the production-hardening framing (precision locks)

Sprint 13.6b ships the ADR-018 **token quota meter** — the second half of ADR-018 (kill switches landed at 13.6a). It is a coherent, self-contained slice: the `QuotaEngine` (decomposed atomic Redis counters), `gateway_call_ledger` token/cost evidence columns + migration, gateway + scheduler enforcement, and a read-only quota portal surface. **It does NOT close any remaining ADR-018 work beyond quotas** (spend-class enforcement, the operator override / `PUT /quotas`, the Sprint-14 dashboard all stay deferred).

Two **precision locks** the spec MUST honor verbatim (the user's anti-over-claim guard):

1. **"Atomic counter reservation," NOT "single atomic transaction."** `INCRBY`/`DECRBY` give concurrency-safe admission + rollback (each caller checks its own post-increment cumulative), but there remains a **bounded fail-closed crash window** between the counter mutation and the reservation-record persistence/release. This is ACCEPTABLE for 13.6b precisely because every leak is **quota-DENYING, never quota-OPENING**, and self-heals at the 48h key TTL. The spec documents the window at each site; a test pins the deny-safe direction. We never claim transactional all-or-nothing.
2. **Gateway quotas are tenant-aggregate + actual-usage-based.** The gateway has NO `pack_id` and NO pre-dispatch completion-token count (`gateway.py:330` — `completion(*, tier, messages, request_id, tenant_id, agent_workforce_id)`), so it CANNOT reserve a known amount before dispatch. The gateway gate is a coarse **"is this tenant already exhausted?"** check (existing `actuals + outstanding reservations ≥ tenant_limit` → refuse) + a post-completion **actuals INCRBY**. True per-call reservation semantics belong to the **scheduler**, which has `task_id` + `pack_id` + `requested_estimated_tokens`. The spec never implies exact pre-dispatch hard caps for gateway calls with unknown completion tokens.

## 3. `QuotaEngine` (NEW `core/emergency/quotas.py` — on the CC durable gate; `core/` stop-rule)

### 3.1 Consumer-owned Redis seam

```python
@runtime_checkable
class _AsyncRedisQuotaLike(Protocol):
    async def get(self, key: str) -> Any: ...
    async def set(self, key: str, value: Any, **kwargs: Any) -> Any: ...
    async def incrby(self, key: str, amount: int) -> int: ...
    async def decrby(self, key: str, amount: int) -> int: ...
    async def getdel(self, key: str) -> Any: ...
    async def expire(self, key: str, seconds: int) -> Any: ...
```

The real `redis.asyncio.Redis` satisfies this structurally (all six are native). `getdel` is Redis 6.2+ (2021) — within the bundled-Redis floor. NO shared-adapter change; `QuotaEngine` is constructed over `adapters.cache.client` at the composition root.

### 3.2 Key scheme (UTC calendar-day window via injectable clock; ~48h `expire`)

- `cognic:quota:reserved:tokens:<tenant_id>:<yyyymmdd>` — tenant reserved counter (scheduler plane).
- `cognic:quota:reserved:tokens:<tenant_id>:<pack_id>:<yyyymmdd>` — per-pack reserved counter.
- `cognic:quota:actual:tokens:<tenant_id>:<yyyymmdd>` — tenant actuals counter (gateway plane).
- `cognic:quota:reservation:<task_id>` — the per-task reservation record (JSON: `tenant_id` / `pack_id` / `tokens` / `yyyymmdd`), so `release_reservation` knows exactly what to `DECRBY`. TTL ~48h.

The reserved (scheduler) and actuals (gateway) families are SEPARATE so reservation + real usage never double-count; admission checks `actuals + outstanding_reservations + requested ≤ limit`.

### 3.3 `would_admit` — the atomic-counter reservation (satisfies the `QuotaInterrogator` seam byte-for-byte)

`async def would_admit(*, task_id: uuid.UUID, tenant_id: str, pack_id: str, estimated_tokens: int) -> bool`

1. **Duplicate-`task_id` contract (production-hardening lock).** `GET cognic:quota:reservation:<task_id>`:
   - **exists + same shape** (same `tenant_id`/`pack_id`/`tokens`) → **idempotent: return `True` WITHOUT a second increment** (the reservation already holds; a retried submit must not double-charge).
   - **exists + mismatched shape** (same `task_id`, different tenant/pack/tokens) → **fail loud**: raise the typed `QuotaReservationConflict` (a reused `task_id` with different params is a caller-contract violation, never a quota outcome — it must surface, not silently refuse; the scheduler mints a fresh `task_id` per submit so this never fires in practice, but a real bug must be visible). The scheduler's `_do_admission_work` `except BaseException` (`engine.py:565`) releases + propagates.
   - **absent** → proceed to step 2.
2. `INCRBY` the tenant reserved counter by `estimated_tokens` → `reserved_after` (atomic; the cumulative incl. this request). `INCRBY` the pack reserved counter likewise. `expire` both (~48h, refreshed each write).
3. `GET` the tenant actuals counter (default 0).
4. Resolve `tenant_limit` (Settings + overlay) + `pack_limit`. Check `actuals + tenant_reserved_after ≤ tenant_limit` AND `pack_reserved_after ≤ pack_limit` (pack actuals are gateway-unavailable — pack is reservation-only; documented). Soft-warn if either ≥ 80%.
   - **over** → `DECRBY` both reserved counters by `estimated_tokens` (rollback), do NOT write the reservation record, return `False`.
   - **under** → `SET cognic:quota:reservation:<task_id>` (the JSON record) + `expire` (~48h), return `True`.
5. **Crash window (precision lock 1):** between the step-2 `INCRBY` and the step-4 record `SET`, a crash leaks the increment with no release record → the reserved counter is inflated → **quota-DENYING** → self-heals at the 48h TTL. Documented inline + pinned by a test asserting the leak direction is deny-safe.

### 3.4 `release_reservation` — idempotent, GETDEL-first (deny-safe crash window)

`async def release_reservation(task_id: uuid.UUID) -> None`

1. `GETDEL cognic:quota:reservation:<task_id>` (atomic get-and-delete, exactly-once):
   - **nil** → no-op (idempotent — terminal-state paths may fire multiple times per the seam contract).
   - **value** → `DECRBY` the tenant + pack reserved counters by the stored `tokens`.
2. **Crash window:** GETDEL-FIRST means a crash between the GETDEL and the DECRBYs leaves the counter inflated (the record is already gone) → **quota-DENYING** → 48h TTL self-heal. (DECRBY-first would risk a double-release double-decrement; GETDEL-first is the deny-safe + idempotent ordering.) Never raises on Redis-present paths.

### 3.5 Fail-posture (precision lock 1 + review-patch-1 from the 13.6 spec)

ANY Redis error in `would_admit` → return `False` immediately (refuse). NO cache-then-allow — a reservation that cannot be made atomically must never be granted from cache. **Partial-increment posture:** if a Redis error occurs AFTER one counter increment has already succeeded (e.g. tenant `INCRBY` ok, pack `INCRBY` errors), it is treated exactly like the documented crash window — `would_admit` returns `False`, the already-leaked counter is **quota-DENYING** (never quota-opening), and the 48h TTL self-heals it; no best-effort rollback is attempted (a rollback `DECRBY` against an unreachable Redis would itself fail, and the deny-safe leak is the acceptable outcome). `release_reservation` on a Redis error logs loud + is retry-safe. The last-known-good cache is permitted ONLY for the read-only portal usage display, never for admission.
