## 1. Scope + the locked posture

Sprint 13.7 turns the **scheduler** from a seam-only type (constructed only in tests) into a **production-constructed `SchedulerEngine` in `build_runtime`**, DI-binding its real consumer conformers (approval, quota, kill-switch, pack-state, storage, policy). This closes the largest "built but not live" gap for the scheduler's three emergency/approval seam consumers — they become *constructed-and-bound*.

**Scope locks (Forks A–E, resolved 2026-06-13):**
- **Fork A — scheduler-only.** 13.7 does the scheduler composition-root wiring ONLY.
- **Fork B — MCP is its own sprint (13.8).** The MCP-host production construction (the manifest→`MCPServerEntry` mapper + `registry.mcp_servers()` + a long-lived `MCPAuthzClient` + transports + `app.state.mcp_host` mount — ~500-800 LOC the reconnaissance found un-built) gets its own design space. NOT in 13.7.
- **Fork C — sandbox approval moves to 14A.** Threading `approval_engine` into `admit_policy` is BLOCKED on there being no Runtime-owned sandbox backend; constructing one is the sandbox **execution** path, intrinsically 14A's "execute via sandbox/sub-agent primitive." 13.7 does NOT touch sandbox.
- **Fork D — construct + expose only.** 13.7 constructs `SchedulerEngine` + exposes it on `Runtime.scheduler` + `app.state.scheduler`. NO submit→execute caller, NO portal route — 14A (managed-runtime vertical slice) owns invocation. The e2e drives `submit()` **directly** through the constructed engine.
- **Fork E — `_NullParentBudgetResolver` sentinel.** Parent-budget resolution stays the fail-loud sentinel until 14A wires the real `LocalParentBudgetResolver` with the live sub-agent budget snapshot. A top-level submit (`parent_task_id=None`) never consults it; a sub-agent submit pre-14A fails loud (correct).

**The cache-conditional posture (user lock — the load-bearing nuance):** the scheduler is production-constructed ONLY when its required runtime dependencies are available. Its quota + kill-switch conformers wrap the `QuotaEngine` + `KillSwitchEngine`, which exist ONLY when the cache (Redis) adapter is present. So the scheduler is constructed **inside the `if adapters.cache is not None:` block** (symmetric with `quota_engine` / `kill_switch_engine`); `Runtime.scheduler` is `None` on the gateway-only path. We do NOT silently create a scheduler with `_Null` quota/kill-switch sentinels that would fail-loud on the first submit's quota/kill-switch consult. Pinned both ways.

This is NOT a feature cut — MCP and sandbox are relocated to the sprints where their prerequisites exist (13.8 / 14A), not dropped.

## 2. What `build_runtime` constructs (the exact wiring)

Inside the existing cache block (after `quota_engine` is built, where `cache_client` + `kill_switch_engine` + `overlay_resolver` are in scope), construct — function-local imports per the cache-block convention:

```python
scheduler_opa = await OPAEngine.create(
    bundle_path=settings.scheduler_policy_bundle,          # NEW Setting (§4)
    audit_store=audit_store,
    decision_history_store=decision_history_store,
    opa_path=settings.opa_path,
    eval_timeout_s=settings.opa_eval_timeout_s,
)
scheduler = SchedulerEngine(
    storage=SchedulerStorage(engine),                      # the relational engine
    caps=ConcurrencyCaps(
        per_tenant_interactive=settings.scheduler_per_tenant_interactive,
        per_tenant_background=settings.scheduler_per_tenant_background,
        per_pack=settings.scheduler_per_pack,
        per_actor=settings.scheduler_per_actor,
    ),
    class_settings={
        "interactive": (settings.scheduler_queue_depth_interactive,
                        settings.scheduler_class_sla_interactive_s),   # NEW Setting (§4)
        "background": (settings.scheduler_queue_depth_background,
                       settings.scheduler_class_sla_background_s),      # NEW Setting (§4)
    },
    policy_evaluator=SchedulerPolicy(opa_engine=scheduler_opa).evaluate,   # the bound method
    quota_interrogator=quota_engine,
    kill_switch_interrogator=SchedulerKillSwitchConformer(engine=kill_switch_engine),
    pack_state_interrogator=PackStoreStateInterrogator(store=PackRecordStore(engine)),
    approval_engine=approval_engine,
    # parent_budget_resolver OMITTED → _NullParentBudgetResolver (Fork E).
)
```

## 6. E2e proof shape — the composition proof

**Prerequisite (test-only — the cache-client quota-ops gap).** The build_runtime e2e drives `cache_driver="memory"`, whose test cache client is `tests/support/adapter_fixtures.py::_InMemoryKVClient` — it exposes ONLY `get`/`set`, while the build_runtime-constructed `QuotaEngine.would_admit` needs `incrby`/`decrby`/`getdel`/`expire`. Without them the submit-cycle would fail CLOSED at quota (the `except Exception → return False` arm) instead of admitting. Fix (the locked lean): **extend the test-only `_InMemoryKVClient` with `incrby`/`decrby`/`getdel`/`expire`** (deterministic dict-backed, mirroring the T1 quota-engine fake) + a pin asserting it structurally satisfies the QuotaEngine's needs (e.g. `isinstance(adapter.client, _AsyncRedisQuotaLike)`). This touches ONLY the test-support fixture — NOT the shared `_AsyncKVClient` adapter Protocol (which stays get/set; the real `redis.asyncio.Redis` already satisfies the QuotaEngine's consumer-owned `_AsyncRedisQuotaLike` for production). This extension is a T3 prerequisite step.

`tests/integration/scheduler/test_scheduler_composition_e2e.py`: drive `build_runtime` (settings with `cache_driver="memory"` + the extended in-memory cache client, a migrated DB) → `runtime.scheduler.submit(...)` **directly** (Fork D — no route). Two cases compose the whole engine:
1. **Installed pack, safe tier → admit cycle.** Seed an `installed` pack row; `submit` (internal_write, top-level) → `accepted_immediate`; assert the quota counter reserved; `mark_running` → `complete` → quota released; assert the `scheduler.admission_accepted` + lifecycle chain rows. This exercises quota + kill-switch + approval (auto-tier) + pack-state + policy + storage in ONE flow through the build_runtime-constructed engine.
2. **Non-installed pack → `refused_pack_not_installed`** (proves the `PackStoreStateInterrogator` is genuinely wired, not bypassed).

Plus harness pins (`tests/unit/harness/test_runtime.py`):
- **cache-present:** `runtime.scheduler` non-None; identity pins — its `_quota is runtime.quota_engine`, its `_kill_switch._engine is runtime.kill_switch_engine`, its `_approval_engine is runtime.approval_engine`, its `_pack_state` is a `PackStoreStateInterrogator`.
- **cache-absent (`cache_driver="none"`):** `runtime.scheduler is None` (no silent `_Null`-quota construction) + `app.state.scheduler is None` after lifespan.
- **the `_Null` parent-budget posture (honesty pin):** a sub-agent submit (`parent_task_id` set) raises `NotImplementedError` (sub-agent budget inheritance is 14A) — proves the sentinel is correctly the parent-budget seam.
